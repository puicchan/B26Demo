Data Storage & External Services

The application stores operational data in SQL Server (compliance reports, transactions, fraud alerts, users, and session tokens). It also integrates with an external authentication gateway (WhoAmI.ashx) to resolve SSO cookie sessions into application session tokens.

Key Architectural Decisions

  • Uses a classic servlet + JSP monolith with route protection enforced via a servlet filter.
  • Uses direct JDBC and handwritten SQL instead of an ORM/repository framework.
  • Delegates SSO cookie decryption to an external auth gateway rather than handling forms-auth token parsing locally.

Primary Workflows

Workflow 1: Authenticate Compliance Officer

  1. Officer opens /login and optionally submits a session token.
  2. SsoSessionService resolves token from form/header/bearer/cookie-gateway flow.
  3. Service validates token and active user against database.
  4. On success, servlet stores SessionUser in session and redirects to /reports; otherwise returns login error.

Workflow 2: Generate CTR or SAR Compliance Report

  1. Authenticated officer submits report generation form on /reports.
  2. Application validates reportType and parses threshold for CTR.
  3. For CTR: query high-value transactions joined with account numbers.
  4. For SAR: query high/critical or unresolved fraud alerts.
  5. Application formats report text and inserts a new row in ComplianceReports.
  6. Officer is redirected to /reports and can download generated output.

Workflow 3: Download Filed Report

  1. Officer selects a report from list and requests /reports/download?reportId=....
  2. Application validates numeric report id and fetches report payload.
  3. If found, response returns text attachment with report-specific filename.

Cross-Service Data Flows

Cross-system flow appears in authentication: the app may call the external auth gateway (WhoAmI) to translate the SSO cookie into a session token, then validates that token in SQL Server. Report generation flows are internal to the service, aggregating data from transactions/accounts or fraud alerts into a single compliance report record. If auth gateway lookup fails, login falls back to error/redirect behavior.

Business Rules & Decision Logic

  • Access to report operations requires an authenticated session (SessionUser present).
  • Only two report types are permitted: CTR and SAR; invalid values redirect back to reports page.
  • CTR threshold defaults to 10000.00 when parsing fails.
  • SAR includes records where severity is High/Critical or status is New/InReview.
  • Missing or invalid report identifiers result in redirect rather than file output.
