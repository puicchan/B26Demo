Data Storage & External Services

The application uses a single Microsoft SQL Server database (ZavaBankDB) accessed directly via JDBC with the Microsoft SQL Server JDBC driver. The database contains tables for ComplianceReports, Transactions, Accounts, FraudAlerts, SessionTokens, and Users. An external SSO Auth Gateway (a .NET endpoint at http://zava-auth-gateway:8080/WhoAmI.ashx) is called over HTTP to resolve .ZAVAAUTH FormsAuthentication cookies into session tokens when a user cannot be authenticated from the database directly.

Key Architectural Decisions

  • Servlet/JSP MVC without a framework: The application uses raw Java EE Servlets and JSP views rather than Spring MVC or another framework, requiring explicit URL mapping in web.xml.
  • Direct JDBC data access: All database interactions use raw PreparedStatement calls through a static ComplianceConnectionFactory, without any ORM or repository abstraction layer.
  • Hybrid SSO authentication: Session tokens are first looked up in the local SQL Server SessionTokens table; as a fallback, the .ZAVAAUTH cookie is forwarded to an external .NET auth gateway via HTTP to obtain the session token.

DTOs & Contracts

Two POJO classes serve as data contracts within the application:

  • SessionUser — Represents an authenticated user (userId, username). Created by SsoSessionService and stored in the HTTP session under the key zavaComplianceReporter.user. Effectively acts as the authentication principal passed between the filter, servlets, and session service. Immutable (final fields, no setters).
  • ReportRecord — Represents one row from the ComplianceReports table returned by ReportsServlet for display. Mutable POJO with getters/setters for reportId, reportType, filingDate, status, and filedBy. Full field details are in data-architecture.md.

No OpenAPI/Swagger specifications, protobuf schemas, or GraphQL schemas are present. Serialization is not configured — all responses are HTML (JSP-rendered) or plain text; no JSON API layer exists.

Communication Patterns

Synchronous (inbound): All client communication is synchronous HTTP/1.1 via the Servlet API. There is no REST/JSON API; all responses are HTML (JSP) or plain text file downloads. Form submissions use application/x-www-form-urlencoded.

Synchronous (outbound): SsoSessionService makes a single synchronous GET call to http://zava-auth-gateway:8080/WhoAmI.ashx when a .ZAVAAUTH cookie is present. The URL is hardcoded from ComplianceConfig. No service discovery (Eureka, Consul, Kubernetes DNS) is used — the auth gateway is referenced by a fixed hostname (zava-auth-gateway).

Resilience policies: The outbound auth gateway call has a 3-second connect timeout and 3-second read timeout (conn.setConnectTimeout(3000); conn.setReadTimeout(3000)). There is no retry policy, circuit breaker, or fallback beyond silently returning an empty string on any exception.

Asynchronous messaging: None. No message queues, event brokers, or pub/sub patterns are used.

Security posture: Authentication is enforced by AuthFilter on all /reports and /reports/* URLs, redirecting unauthenticated requests to /login. The auth mechanism relies on session tokens validated against the SessionTokens table in SQL Server. No TLS/HTTPS is configured — the application serves plain HTTP on port 8080. There is no authorization (RBAC) layer beyond the session check; any authenticated user can generate any report type. No CSRF protection is implemented.

Startup dependencies: The application assumes SQL Server is reachable at startup (static JDBC driver loading in ComplianceConnectionFactory); no readiness probes or startup-order mechanisms are configured. See configuration-inventory.md for full configuration details.

Primary Workflows

Workflow 1: User Authentication (SSO Login)

A compliance officer accesses the application and must authenticate before viewing or generating reports. The AuthFilter guards all /reports URLs and redirects unauthenticated requests to /login.

Steps:

  1. User visits /login (GET) — the login page is shown.
  2. User submits a session token (POST /login) or arrives with an active session already set.
  3. SsoSessionService.resolveSessionUser() attempts to identify the user:
    • From form field: reads sessionToken parameter directly.
    • From HTTP header: checks X-Session-Token header.
    • From checks `Authorization: ** header.
    • From .ZAVAAUTH cookie: if none of the above, calls the external SSO Auth Gateway (GET /WhoAmI.ashx) to exchange the FormsAuthentication cookie for a session token (3-second timeout).
  1. The resolved token is validated against the SessionTokens table in SQL Server: must be active, not expired, and belong to an active user.
  2. On success, a SessionUser (userId, username) is placed in the HTTP session.
  3. On failure, the user is redirected to /login with an error message or continues to be blocked.

Workflow 2: View Compliance Reports

An authenticated compliance officer views the list of previously generated reports.

Steps:

  1. Officer navigates to GET /reports (guarded by AuthFilter).
  2. ReportsServlet fetches the latest 100 compliance reports from ComplianceReports table, ordered by FilingDate DESC.
  3. The report list (type, date, status, filer) is displayed in the reports.jsp view.

Workflow 3: Generate CTR (Currency Transaction Report)

A compliance officer generates a CTR, which aggregates all bank transactions above a configured monetary threshold.

Steps:

  1. Officer submits POST /reports/generate with reportType=CTR and an optional thresholdAmount.
  2. Authentication check: session must contain a valid SessionUser; otherwise redirect to /login.
  3. Report type validation: only CTR and SAR are accepted (case-insensitive); any other value redirects to /reports.
  4. Threshold parsing: thresholdAmount is parsed as a BigDecimal; on invalid or missing input, defaults to $10,000.00.
  5. Data retrieval: fetches the 200 most recent transactions (ordered by date DESC) where ABS(Amount) >= threshold, joined with account numbers.
  6. Report composition: generates a plain-text CTR document with header, threshold, and transaction lines.
  7. Persistence: inserts a new row into ComplianceReports with ReportType='CTR', Status='Generated', RegulatoryBody='FinCEN', and the full report text.
  8. Redirects to /reports.

Workflow 4: Generate SAR (Suspicious Activity Report)

A compliance officer generates a SAR from outstanding fraud alerts.

Steps:

  1. Officer submits POST /reports/generate with reportType=SAR.
  2. Authentication check: same as CTR (steps 2–3 above).
  3. Data retrieval: fetches the 200 most recent fraud alerts (ordered by CreatedDate DESC) where Severity IN ('High', 'Critical') OR Status IN ('New', 'InReview').
  4. Report composition: generates a plain-text SAR document with header, severity description, and alert lines.
  5. Persistence: inserts a new row into ComplianceReports with ReportType='SAR', Status='Generated', RegulatoryBody='FinCEN'.
  6. Redirects to /reports.

Workflow 5: Download Report

An authenticated compliance officer downloads a previously generated report as a text file.

Steps:

  1. Officer requests GET /reports/download?reportId=<id> (guarded by AuthFilter).
  2. DownloadReportServlet parses reportId as an integer; invalid values redirect to /reports.
  3. Fetches ReportType and ReportData from ComplianceReports by ReportID.
  4. If found, streams the plain-text content as a text/plain attachment with filename <type>-<id>.txt.
  5. If not found, redirects to /reports.

Cross-Service Data Flows

There is no microservice architecture; all data flows are internal to the single WAR application reading a shared SQL Server database. The one external cross-service call is to the SSO Auth Gateway:

  • SSO Cookie Resolution: When a user presents a .ZAVAAUTH FormsAuthentication cookie (issued by the organization's .NET auth system), the application forwards it to GET /WhoAmI.ashx on the auth gateway. The gateway decrypts the cookie (which Java cannot do natively) and returns a JSON response containing a sessionToken field. This token is then validated against the local SessionTokens table. If the gateway is unavailable or returns a non-200 response, authentication silently fails and the user is redirected to /login — there is no retry, no graceful degradation, and no user-visible error distinguishing a gateway timeout from an invalid credential.

Business Rules & Decision Logic

Validation Rules

  • Report type constraint: Only CTR and SAR are valid report types (case-insensitive). Any other value silently redirects to /reports without an error message.
  • CTR threshold default: If thresholdAmount is absent, empty, or not parseable as a decimal, the threshold defaults to $10,000.00 (the US federal CTR reporting threshold).
  • Download ID validation: reportId must be a valid integer; otherwise the request is silently redirected.

Decision Logic

  • CTR vs SAR branching: Report type determines which database query is executed and which template header is used. CTR queries Transactions filtered by amount threshold; SAR queries FraudAlerts filtered by severity (High, Critical) or status (New, InReview).
  • Authentication precedence: Token resolution is tried in priority order: (1) explicit form parameter, (2) X-Session-Token header, (3) Authorization: Bearer header, (4) .ZAVAAUTH cookie + auth gateway call. The first non-empty result is used.
  • Auth gateway fallback: If the SSO Auth Gateway is unreachable (timeout, error, non-200 response), authentication silently returns null, effectively blocking all cookie-based logins without any user-visible explanation.

State Transitions

  • ComplianceReport status: Reports are created with Status='Generated'. There is no further lifecycle management (no Submitted, Approved, Filed states) implemented in this application.
  • Session lifecycle: Sessions are validated on every request by checking IsActive=1 and ExpiresAt > GETDATE() in the SessionTokens table. There is no logout mechanism implemented.

Business Constraints

  • Top-N limits: CTR retrieves at most 200 transactions; SAR retrieves at most 200 fraud alerts; the reports list shows the latest 100 reports. These are hard-coded caps with no pagination.
  • RegulatoryBody is always FinCEN: Hard-coded in GenerateReportServlet.saveReport(); no multi-regulator support.
  • Report content is plain text: Report data is stored as a pre-formatted text string, not structured data; there is no machine-readable output format (e.g., XML FinCEN BSA e-filing format).

Cross-Cutting Concerns

  • Transactions: No @Transactional annotations or explicit transaction management. Each SQL operation is a single statement; if the INSERT INTO ComplianceReports fails after the data-gathering SELECT, no rollback or compensation occurs. Exceptions are swallowed silently (catch (Exception ignored)).
  • Error handling: All database exceptions and auth gateway failures are caught and silently discarded. Business failures (e.g., unable to retrieve transactions) result in empty reports rather than error responses.
  • Audit trail: The FiledBy column records the username of the compliance officer who generated each report. No other audit logging is implemented.
  • Authorization: Any authenticated user (valid SessionUser in session) can generate any report type or download any report by ID. There is no role-based authorization distinguishing read-only staff from report generators.

Version & Compatibility Risks

Java 8 (sourceCompatibility = JavaVersion.VERSION_1_8) reached Oracle's end of free public updates in March 2019 and is the primary migration risk. The declared javax.servlet-api:3.1.0 and javax.servlet:jstl:1.2 use the legacy javax. namespace, which was replaced by jakarta. in Jakarta EE 9+; migrating to a modern servlet container (e.g., Tomcat 10+) would require recompiling with the new namespace. The mssql-jdbc:12.8.1.jre8 driver is recent and does not pose a compatibility risk, though migrating the runtime to JDK 17 or 21 would require switching to the jre11 or jre11 variant. No BOM or dependency-version management plugin is configured, so all versions are pinned manually.

Notable Observations

  • Minimal dependency footprint: The application has only 3 declared external dependencies, reflecting a raw Servlet/JSP approach with no web framework, ORM, logging framework, or security library.
  • No dedicated logging framework: There is no SLF4J, Log4j, or Logback declared; the application relies on System.err/exception swallowing (catch (Exception ignored)), making observability a modernization concern.
  • No security library: Authentication relies entirely on custom session-token logic and an external SSO gateway with no standard security framework (e.g., Spring Security, Shiro) protecting endpoints.
  • Legacy javax namespace: Both javax.servlet-api and jstl use the pre-Jakarta javax namespace; upgrading to Jakarta EE 9+ or Spring Boot 3.x would require changing all javax.servlet. imports to jakarta.servlet..

Caching Strategy

No caching layer is configured at any level. Each HTTP request that requires data opens a new JDBC connection, executes queries, and closes the connection immediately. There is no in-process cache (Caffeine, EhCache), distributed cache (Redis, Memcached), HTTP-level response caching, or Hibernate second-level cache. Report data for CTR and SAR generation is fetched fresh on every POST request.

Data Ownership Boundaries

The application uses a shared monolithic database (ZavaBankDB) that contains all application tables in a single schema. There is no database-per-service boundary, logical schema separation, or multi-tenancy. All read/write operations are performed directly by the single WAR application; no other service accesses these tables (based on available source). All operations are reads or single-row inserts; there is no CQRS separation, event sourcing, or outbox pattern.

The ComplianceReports.ReportData column stores the full text of generated reports (potentially large string payloads) inline in the database rather than in object storage, which may create performance and storage concerns at scale.
