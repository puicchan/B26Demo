Data Storage & External Services

ZavaAuditWorker depends on two external services. RabbitMQ acts as the message broker, with the worker continuously polling the audit.events queue via the AMQP BasicGet method. Failed messages are negatively acknowledged and routed to a dead letter exchange (zava.dlx). SQL Server (ZavaBankDB) is the persistence target; the worker supports two AuditLog schemas — a legacy schema with an ActionID foreign key referencing an AuditActions lookup table, and a modern schema with direct event columns. Both the RabbitMQ connection details and the SQL Server connection string are configurable via environment variables or app.config app settings.

Key Architectural Decisions

  • Pull-based (polling) message consumption: The worker uses BasicGet with manual acknowledgment rather than a push-based consumer, giving it simple, predictable resource usage at the cost of latency.
  • Dual schema support: A runtime check (HasLegacyAuditColumns) detects whether the target database uses the old AuditLog schema and branches accordingly, enabling zero-downtime migration between schema versions.
  • Configuration layering: All settings are resolved in priority order — environment variables first, then app.config app settings, then hard-coded defaults — supporting both containerized and traditional deployment.

DTOs & Contracts

Inbound Message Contract

The worker accepts two message formats on the audit.events queue:

  • JSON format — a flat JSON object with the following well-known field names (case-insensitive key lookup): eventType/EventType/action, serviceName/ServiceName/service, userId/UserId/userName, timestamp/Timestamp, correlationId/CorrelationId/requestId. The full raw body is stored in the Details/DetailsJson column.
  • XML format — an XML document whose root element contains child elements EventType/Action/Type, ServiceName/Service, UserId/UserName, Timestamp, CorrelationId/RequestId. The full raw body is stored as details.

Internal Data Model

AuditEvent is the internal data transfer object produced by the parsers. See data-architecture.md for field-level details. The class is mutable (public fields), not a C# record or immutable type.

No OpenAPI/Swagger specification, protobuf schema, or GraphQL schema exists. Serialization uses System.Web.Script.Serialization.JavaScriptSerializer (JSON) and System.Xml.Linq.XDocument (XML) — both are .NET Framework built-ins with no modern equivalents in .NET 6+.

Communication Patterns

Asynchronous — AMQP (Inbound)

The worker uses a pull-based (polling) consumption pattern over AMQP 0-9-1 via RabbitMQ.Client 5.2.0. On each poll cycle, the worker:

  1. Opens a new TCP connection and channel.
  2. Calls channel.BasicGet(queue, autoAck: false) — a synchronous, blocking pull.
  3. On success, processes the message and calls channel.BasicAck.
  4. On processing failure, calls channel.BasicNack(requeue: false), routing the message to the dead letter exchange (zava.dlx).

The poll interval is configurable (WORKER_POLL_INTERVAL_MS / Worker.PollIntervalMs, default 5000 ms). A new TCP connection is opened and closed every poll cycle (no persistent channel or consumer is reused).

Synchronous — ADO.NET (Outbound)

Audit events are persisted via direct ADO.NET (SqlConnection / SqlCommand). There is no ORM, no connection pool configuration, and no retry logic at the data access layer.

Resilience Patterns

  • Retry: None. Failed messages are dead-lettered immediately without retry.
  • Circuit breaker: None.
  • Timeout: No explicit timeouts are configured on the RabbitMQ connection or SQL Server commands.
  • Dead letter handling: RabbitMQ BasicNack with requeue: false sends failed messages to zava.dlx. No consumer for the dead letter exchange is present in this repository.

Service Discovery

No service discovery mechanism is used. RabbitMQ and SQL Server hostnames are resolved via environment variables / app.config configuration values — i.e., hardcoded logical names (rabbitmq, sqlserver) typically resolved by container DNS or infrastructure configuration outside this codebase.

Security Posture

No transport-level security (TLS) is configured. The RabbitMQ ConnectionFactory does not set Ssl properties; the SQL Server connection string contains Encrypt=False;TrustServerCertificate=True. Credentials (RabbitMQ username/password, SQL Server sa password) are supplied via environment variables or app.config. There is no mutual TLS, no token-based authentication between services, and no authorization layer within the worker itself. All security relies entirely on network-level isolation.

Core Business Workflows

ZavaAuditWorker provides the audit trail collection function for ZavaBank: it continuously harvests audit events emitted by other services, normalises them, and appends them to a durable audit log for compliance and operational review.

Primary Workflows

Workflow 1: Audit Event Collection and Persistence

Trigger: The worker's main loop wakes every PollIntervalMs (default 5 seconds).

Steps:

  1. Poll — The worker connects to RabbitMQ and issues a BasicGet on the audit.events queue.
  2. Empty queue — If no message is waiting, the worker logs "No audit events available" and sleeps until the next poll cycle. No state change occurs.
  3. Message received — The message body (a UTF-8 string) and its delivery tag are passed to the parser.
  4. Format detection — The first non-whitespace character determines the parser: < → XML path, anything else → JSON path.
  5. Parse JSON — JavaScriptSerializer deserialises the body into a dictionary; field values are extracted by trying multiple key aliases (eventType / EventType / action, etc.). The full raw body is preserved as DetailsJson.
  6. Parse XML — XDocument parses the body; element values are extracted by trying multiple element-name aliases. The full raw body is preserved as DetailsJson.
  7. AuditEvent produced — Both parse paths yield a normalised AuditEvent with EventType, ServiceName, UserId, Timestamp, DetailsJson, and CorrelationId.
  8. Schema detection — The worker queries INFORMATION_SCHEMA.COLUMNS to determine which AuditLog schema is active. This runs on every message.
  9. Legacy schema path — If ActionID column is present, EnsureAction looks up or inserts a row in AuditActions for the event type, returning its ActionID. The audit row is then inserted into AuditLog with the legacy column set.
  10. Modern schema path — If ActionID column is absent, the audit row is inserted directly into AuditLog using the modern column set.
  11. Acknowledge — On successful persistence, BasicAck is sent to RabbitMQ — the message is removed from the queue permanently.
  12. Dead-letter on failure — If any step from 5 through 10 throws an exception, BasicNack(requeue: false) is sent and the message is routed to the dead letter exchange (zava.dlx). The exception is logged to stdout. The worker continues to the next poll cycle.

Workflow 2: Worker Lifecycle Management

Trigger: Process start (Program.Main).

Steps:

  1. Load configuration — All settings are resolved from environment variables → app.config → code defaults.
  2. Register shutdown hooks — Console.CancelKeyPress (Ctrl+C) and AppDomain.ProcessExit set a volatile _keepRunning = false flag.
  3. Poll loop — The worker enters an infinite loop calling PollQueue() followed by Thread.Sleep(_pollIntervalMs), repeating until _keepRunning is false.
  4. Graceful shutdown — When _keepRunning becomes false (signal or ProcessExit), the loop exits and a shutdown message is written to stdout. Any in-progress RabbitMQ connection is closed by the using block in PollQueue.

Cross-Service Data Flows

ZavaAuditWorker participates in a unidirectional data flow: upstream ZavaBank services produce audit events and publish them to RabbitMQ; ZavaAuditWorker is the sole consumer. There is no request/response interaction, no gateway aggregation, and no data returned upstream.

The worker does not read from any upstream service's API or database. It receives self-contained event payloads over AMQP and persists them as-is. The correlation ID embedded in each event is the only link back to upstream business transactions; the worker stores it but does not use it for cross-service lookups.

If RabbitMQ is unavailable, messages accumulate in the broker (managed externally). If the worker crashes mid-processing (after BasicGet but before BasicAck), the un-acknowledged message will be re-queued by RabbitMQ according to its consumer-timeout policy (not managed by this application).

Business Rules & Decision Logic

Validation Rules

  • Non-empty message body: If the raw message body is null or whitespace, ParseAuditEvent throws InvalidOperationException("Audit message body is empty.") — the message is dead-lettered.
  • Timestamp leniency: If the timestamp / Timestamp field cannot be parsed as a valid DateTime, the current system time is used as a fallback. No validation error is raised.
  • Missing fields: All event fields (EventType, ServiceName, UserId, CorrelationId) are optional from a parsing standpoint; unresolved fields default to empty string. No mandatory-field validation is applied before persistence.

Decision Logic

  • Format routing: Message format is determined by the first non-whitespace character: < → XML parser, otherwise → JSON parser. There is no Content-Type header inspection.
  • Schema detection: Runtime detection of the AuditLog schema via INFORMATION_SCHEMA.COLUMNS drives which INSERT branch executes. This is evaluated on every message — there is no cached result.
  • Action lookup/upsert: In the legacy path, if an AuditActions row for the event type does not exist, one is created automatically with Severity = 'Info' and Description = 'Auto-created by ZavaAuditWorker'. Unknown event types are stored as 'UNKNOWN'.

State Transitions

The AuditLog record is append-only — there are no state transitions on existing records. The entity lifecycle is: Message received → Parsed → Persisted (terminal state). Dead-lettered messages do not create a database record.

Error Handling

  • Processing exceptions (parse failure, DB failure): Caught in PollQueue, logged to stdout, message is negatively acknowledged without requeue. The worker continues its loop.
  • Poll-cycle exceptions (RabbitMQ connection failure): Caught in Main, logged to stdout. The worker sleeps and retries.
  • Dead letter exchange: Failed messages are routed to zava.dlx. No dead-letter consumer is present in this repository; dead-lettered messages must be monitored and replayed externally.

Audit / Logging

All logging is performed via Console.WriteLine to stdout. Log entries include: startup confirmation, queue/host parameters, per-message delivery tags, saved event metadata (EventType, ServiceName, UserId, CorrelationId), empty-queue status, and exception messages. There is no structured logging, no log levels, no correlation ID propagation to log statements, and no audit trail of the worker's own actions.

Authorization

No authentication or authorization is implemented within the worker. Access to RabbitMQ and SQL Server is controlled exclusively by the credentials supplied via configuration. There is no role-based or attribute-based access control within the application logic.

Version & Compatibility Risks

.NET Framework 4.8 is the final version of the .NET Framework line and is in long-term maintenance mode (security fixes only, no new features). It is not cross-platform. RabbitMQ.Client 5.2.0 is a significantly outdated release — the current stable series is 6.x/7.x, which introduced async-first APIs, breaking changes in the consumer model, and dropped support for older AMQPs. System.Web.Script.Serialization (JavaScriptSerializer) was deprecated in .NET Core / .NET 5+ and does not exist in the modern System.Text.Json-based ecosystem; migrating to modern .NET would require replacing it with System.Text.Json or Newtonsoft.Json. System.Data.SqlClient (the .NET Framework built-in namespace) has been superseded by the cross-platform Microsoft.Data.SqlClient NuGet package — a migration target when moving to modern .NET.

Notable Observations

  • Single external dependency: The project has only one NuGet package (RabbitMQ.Client), keeping the dependency footprint minimal. The trade-off is heavy reliance on aging .NET Framework built-ins that have no direct equivalents in modern .NET.
  • RabbitMQ.Client 5.2.0 is end-of-branch: RabbitMQ.Client 5.x targeted .NET Framework / .NET Standard 2.0 and uses a synchronous pull model (BasicGet). The 6.x+ series moved to async consumers and renamed/removed several types — upgrading will require code changes beyond a version bump.
  • No logging framework: The application uses only Console.WriteLine for all diagnostic output. There is no structured logging (Serilog, NLog, Microsoft.Extensions.Logging), which limits observability in production environments and cloud-native deployments.
  • No dependency injection or DI container: All dependencies are instantiated directly as static methods. While appropriate for the current small scope, this makes unit testing and future extension harder without refactoring.

Data Architecture & Persistence Layer

ZavaAuditWorker uses raw ADO.NET (no ORM) against a SQL Server database, persisting audit events into a single AuditLog table that supports two schema variants detected at runtime.

Caching Strategy

No caching layer is present. All reads (the HasLegacyAuditColumns schema detection query and the EnsureAction lookup) execute a live database query on every single poll cycle. There are no @Cacheable, IMemoryCache, IDistributedCache, Redis, or any other cache abstractions in use.

This is a notable performance concern: the schema detection query (INFORMATION_SCHEMA.COLUMNS) and the AuditActions lookup run once per message, not once per application lifetime. For high-throughput scenarios, caching the schema detection result and the ActionID lookup in memory would significantly reduce database round-trips.

Data Ownership Boundaries

ZavaAuditWorker is the sole service in this repository and operates as the exclusive writer to the ZavaBankDB database. It does not share a database with other observable services in this codebase.

The schema detection pattern (querying INFORMATION_SCHEMA) implies the database schema is managed externally (by another migration process, DBA scripts, or another application), and this service adapts passively. There is no CQRS split — the same service reads (schema detection, action lookup) and writes (audit inserts) against the same database with the same connection.

Cross-service data access is not applicable: no other service in this codebase reads from ZavaBankDB, and this service does not call external service APIs to fetch data.
