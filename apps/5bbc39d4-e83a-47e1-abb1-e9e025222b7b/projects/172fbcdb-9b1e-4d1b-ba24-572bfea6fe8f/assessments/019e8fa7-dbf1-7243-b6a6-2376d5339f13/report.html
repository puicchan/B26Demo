Data Storage & External Services

The worker reads events from RabbitMQ (audit.events) and writes them to SQL Server (AuditLog, with optional AuditActions). Failed messages are negatively acknowledged and routed through a dead-letter exchange.

Key Architectural Decisions

  • Uses polling (BasicGet) with a configurable sleep interval instead of push consumers.
  • Supports dual persistence paths for legacy (ActionID schema) and modern audit table schema.
  • Prefers environment variables over app.config values for runtime configurability.

DTOs & Contracts

The primary contract model is AuditEvent, used as the normalized message contract after JSON/XML parsing. It acts as both input-normalization and persistence transfer object with fields for event type, service name, user, timestamp, details payload, and correlation ID. No OpenAPI, protobuf, or GraphQL contracts are defined.

Communication Patterns

Communication is asynchronous from upstream publishers to RabbitMQ and then to the worker via pull-based BasicGet. Persistence is synchronous SQL execution inside the worker process. Retry/circuit-breaker libraries are not configured; failure handling is queue-level (BasicNack with dead-letter routing). Service discovery is not used, and no API-level authentication, authorization, or TLS configuration is defined in this repository.

This worker supports the audit and compliance domain by reliably ingesting system audit events and storing them in the platform audit database for downstream review.

Primary Workflows

Workflow 1: Process audit queue message

  1. Worker polls audit.events and receives one message.
  2. Payload is parsed from JSON or XML into AuditEvent.
  3. Business decision checks database schema mode (legacy vs modern).
  4. Event is persisted into the appropriate audit table(s).
  5. Message is acknowledged on success; dead-lettered on failure.

Cross-Service Data Flows

Upstream platform services publish audit events to RabbitMQ. ZavaAuditWorker is the consumer and source of truth for persistence into audit tables. If persistence fails, the message is dead-lettered so upstream services are not synchronously blocked while failures remain traceable.

Business Workflow Sequence

sequenceDiagram participant Producer as "Platform Service" participant Broker as "RabbitMQ" participant Worker as "Audit Worker" participant DB as "Audit Database" Producer-->>Broker: Emit audit event Worker->>Broker: Poll next audit message Broker-->>Worker: Return payload Worker->>Worker: Validate non-empty payload and parse format alt Legacy audit schema detected Worker->>DB: Resolve or create action Worker->>DB: Insert legacy audit row else Modern audit schema detected Worker->>DB: Insert modern audit row end alt Persistence success Worker->>Broker: Acknowledge message else Persistence failure Note over Worker,Broker: Message is negatively acknowledged and dead-lettered Worker->>Broker: Reject without requeue end

Business Rules & Decision Logic

  • Empty payloads are invalid and fail processing.
  • Message format routing is content-based: XML when payload starts with <, otherwise JSON parser path.
  • Legacy schema writes require an action lookup/create step before audit insertion.
  • On any processing exception, the workflow rejects the message without requeue to avoid infinite retry loops in this worker.

No caching layer is implemented. All reads/writes in this worker are immediate queue and database operations.
