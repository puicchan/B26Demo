Data Storage & External Services

The worker depends on two external services. RabbitMQ (default host: rabbitmq, port: 5672, vhost: /zavabank) acts as the message broker: the worker polls the audit.events queue using BasicGet and routes unprocessable messages to the zava.dlx dead-letter exchange. SQL Server (ZavaBankDB) stores the persisted audit records; the worker supports two schema variants — a legacy AuditLog schema that uses an AuditActions lookup table with an ActionID foreign key, and a modern flat AuditLog schema with direct event columns.

Key Architectural Decisions

  • Poll-based consumption: The worker uses BasicGet (pull) rather than a push-based consumer, with a configurable poll interval (default 5 s) and a sleep-loop for pacing.
  • Dual schema support: At runtime the worker detects whether the target AuditLog table uses the legacy column layout (with ActionID) or the modern flat layout, and chooses the appropriate INSERT statement accordingly.
  • Environment-first configuration: All connection parameters can be overridden via environment variables (e.g. RABBITMQ_HOST, DB_CONNECTION_STRING), falling back to app.config AppSettings, then to hard-coded defaults.

One internal data class is used as the in-process contract between the parsing and persistence layers:

Serialisation uses System.Web.Script.Serialization.JavaScriptSerializer (JSON) and System.Xml.Linq.XDocument (XML).

Communication Patterns

Asynchronous inbound (RabbitMQ AMQP):

The worker uses BasicGet (pull/polling model) rather than a push-based consumer. On each poll cycle it opens a new IConnection/IModel pair, calls BasicGet on the audit.events queue, and either BasicAcks on success or BasicNacks (with requeue=false) on processing failure. Failed messages are routed to the zava.dlx dead-letter exchange. A configurable sleep interval (default 5 000 ms) paces the polling loop. Queue existence is verified passively with QueueDeclarePassive; if absent the queue is created with x-dead-letter-exchange set.

Synchronous outbound (ADO.NET / SQL Server):

Each successfully parsed event is written to SQL Server synchronously within the same poll cycle using System.Data.SqlClient. The worker opens a fresh SqlConnection per event, executes a schema-detection query (INFORMATION_SCHEMA.COLUMNS), then inserts into AuditLog using either the legacy schema path (with an AuditActions upsert for ActionID) or the modern flat schema path.

Resilience:

No circuit breaker, retry policy, or timeout configuration is implemented. Connection failures to either RabbitMQ or SQL Server bubble up as exceptions and are caught at the poll loop level; the worker logs the error and waits for the next poll interval before retrying. AutomaticRecoveryEnabled is explicitly set to false on the RabbitMQ ConnectionFactory, meaning AMQP connection recovery must be handled by the outer restart mechanism (e.g., container restart policy).

Service discovery:

No service discovery mechanism is used. All hostnames (rabbitmq, sqlserver) are configured statically via environment variables or app.config AppSettings.

Security posture:

No authentication or TLS is configured for the RabbitMQ connection (Encrypt=False on the SQL connection string; no SSL options on ConnectionFactory). The RabbitMQ credentials (zava_app / zava_pass) and SQL Server password (Zava123!) are stored in plaintext in app.config. No transport-layer security, token-based auth, or secrets management integration is present.

Primary Workflows

Workflow 1: Audit Event Ingestion and Persistence

This is the sole primary workflow. The worker runs continuously, polling RabbitMQ for new audit event messages and persisting each one to SQL Server.

Steps:

  1. Poll — the worker calls BasicGet on the audit.events queue.
  2. Empty queue check — if no message is available, log "No audit events available" and sleep for pollIntervalMs (default 5 000 ms) before retrying.
  3. Message receipt — if a message is available, extract the raw message body and delivery tag.
  4. Format detection — inspect the first non-whitespace character of the body: if it is <, treat the payload as XML; otherwise treat it as JSON.
  5. Parse JSON — use JavaScriptSerializer to deserialise the payload; extract eventType/EventType/action, serviceName/ServiceName/service, userId/UserId/userName, timestamp/Timestamp, correlationId/CorrelationId/requestId.
  6. Parse XML — use XDocument to parse the payload; extract EventType/Action/Type, ServiceName/Service, UserId/UserName, Timestamp, CorrelationId/RequestId.
  7. Schema detection — query INFORMATION_SCHEMA.COLUMNS to determine whether the AuditLog table contains an ActionID column (legacy schema) or not (modern schema).
  8. Legacy path (if ActionID column present):
    • Look up the event type in AuditActions by ActionName.
    • If not found, insert a new AuditActions record with Severity = 'Info' and retrieve the generated ActionID.
    • Insert a row into AuditLog using the legacy column set (ActionID, UserName, IPAddress, TableName, RecordID, OldValues, NewValues, Description, Timestamp, SessionID, MachineName).
  1. Modern path (if ActionID column absent):
    • Insert a row directly into AuditLog using the flat column set (EventType, ServiceName, UserId, Timestamp, Details, CorrelationId).
  1. Acknowledge — on success, BasicAck the message so RabbitMQ removes it from the queue.
  2. Error handling — if any step throws an exception, BasicNack (with requeue=false) so the message is routed to the zava.dlx dead-letter exchange; log the error and continue the poll loop.

Cross-Service Data Flows

ZavaAuditWorker is a pure consumer; it does not call any upstream services and does not aggregate data from multiple sources. The data flow is strictly unidirectional:


Upstream ZavaBank Services
        │  publish JSON or XML audit event messages
        ▼
RabbitMQ (audit.events queue)
        │  BasicGet (pull polling, every 5 s)
        ▼

Business Rules & Decision Logic

Validation rules:

  • An empty message body is rejected immediately with InvalidOperationException("Audit message body is empty.") before any parsing is attempted.
  • Timestamp fields that cannot be parsed as a valid DateTime default to DateTime.Now rather than failing the event.
  • Missing or unrecognised field names are tolerated: the parser tries multiple alternative key names (e.g., eventType, EventType, action) and returns an empty string if none match.

Decision logic:

  • Message format routing: The single character test (raw.TrimStart().StartsWith("<")) determines whether JSON or XML parsing is used. There is no explicit content-type header or queue-level schema enforcement.
  • Schema variant routing: The runtime INFORMATION_SCHEMA.COLUMNS check is performed on every saved event; there is no caching of the result, meaning every write opens two round-trips to SQL Server (one for schema detection, one for the insert).
  • Action upsert (legacy path): If an event type is seen for the first time, a new AuditActions record is created automatically with Description = "Auto-created by ZavaAuditWorker" and Severity = "Info". This is an implicit default — there is no validation of the action name against a controlled vocabulary.

Error handling:

  • Exceptions during a poll cycle are caught at the outer loop level; the worker logs the error and sleeps before retrying — it does not terminate.
  • Processing exceptions (inside PollQueue) trigger a BasicNack with requeue=false, routing the message to the dead-letter exchange. There is no retry-before-DLX logic; every failure goes directly to the DLX on the first attempt.

Audit / compliance:

  • The DetailsJson field stores the complete raw message payload (JSON or XML), providing a verbatim audit trail regardless of schema version.
  • The CorrelationId / RequestId field propagates the originating service's request identifier, enabling cross-service request tracing within the audit log.

Authorization:

  • No authorization is implemented within this service. Any process that can publish to the audit.events RabbitMQ queue can inject arbitrary audit records.

Transactions:

  • No explicit transactions are used. The AuditActions upsert and the AuditLog insert (legacy path) are separate SqlCommand executions within the same open SqlConnection but outside a TransactionScope; a failure between the two leaves an orphaned AuditActions record without a corresponding AuditLog entry.

Version & Compatibility Risks

The project targets .NET Framework 4.8, which is in maintenance-only support with no new feature development. RabbitMQ.Client 5.2.0 is significantly outdated — the current stable series is 7.x, and v5.x is no longer receiving security patches. System.Data.SqlClient is the legacy SQL Server driver (part of the .NET Framework BCL); Microsoft now recommends Microsoft.Data.SqlClient, which receives active security and feature updates. System.Web.Script.Serialization (JavaScriptSerializer) has been deprecated in favour of System.Text.Json (built into .NET 5+) or Newtonsoft.Json, and is not available on .NET Core / .NET 5+. The project uses packages.config rather than <PackageReference>, which is the older NuGet format incompatible with SDK-style projects.

Notable Observations

  • Single third-party dependency: The application's external NuGet surface is extremely small (only RabbitMQ.Client), which simplifies dependency management but means most functionality relies on BCL types that will need replacement when migrating away from .NET Framework.
  • Deprecated serialization API: JavaScriptSerializer is unavailable on .NET Core/.NET 5+ without compatibility shims; a migration to System.Text.Json or Newtonsoft.Json will be required for cross-platform portability.
  • Legacy data access driver: System.Data.SqlClient should be replaced by Microsoft.Data.SqlClient to gain TLS 1.3, Azure Active Directory auth, and ongoing security fixes.
  • packages.config format: The project uses the legacy packages.config NuGet format rather than <PackageReference> in the .csproj; this must be migrated as part of any SDK-style project conversion.

ZavaAuditWorker persists audit events to a SQL Server database (ZavaBankDB) using raw ADO.NET (no ORM), and supports two AuditLog table schemas — a legacy layout with an AuditActions lookup table and a modern flat layout.

Caching Strategy

No caching layer is implemented. All reads (schema detection, action lookup) and writes (audit inserts) go directly to SQL Server on every poll cycle. There are no in-memory caches, distributed caches, or query-result caches.

Data Ownership Boundaries

ZavaAuditWorker is the sole service in this repository. It owns the AuditLog and AuditActions tables in ZavaBankDB. No cross-service data access patterns exist within this codebase.

The ZavaBankDB database is presumably shared with other ZavaBank services (the naming convention and the pre-existing schema suggest it is part of a larger monolithic database), but no direct cross-service table access is performed here — the worker only writes to its own audit tables. No read/write segregation (CQRS) is implemented.
