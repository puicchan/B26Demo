Data Storage & External Services

The application writes parsed file records into three SQL Server staging tables (StagingCheckImage, StagingWireConfirmation, StagingRegulatoryFeed) inside the ZavaBankCore database. After each file is processed, a JSON event is published to a RabbitMQ topic exchange (zava.events) using the routing key file.ingested, enabling downstream consumers to react to ingestion outcomes. Configuration is loaded from a classpath properties file with environment variable overrides for all external connection parameters.

Key Architectural Decisions

  • Polling-based ingestion: A tight while(running) loop polls the watch directory at a configurable interval (default 5 s), classifying files by name prefix/suffix rather than using a dedicated file-watch API.
  • Per-file JDBC connection: A new java.sql.Connection is opened and closed for every single INSERT statement, with no connection pooling, which can impact throughput under load.
  • Fire-and-forget RabbitMQ publishing: Messaging failures are logged but do not affect file move outcomes — the file is always moved to processed/ or error/ regardless of whether the event publish succeeds.

DTOs & Contracts

No formal DTO or contract classes exist. The application uses inline String[] arrays derived from CSV parsing (line.split(",", -1)) and fixed-width slicing (safeSlice()) to extract field values. No OpenAPI/Swagger specification, protobuf schema, or GraphQL schema is present.

The RabbitMQ event payload is a hand-constructed JSON string with four fields (eventType, fileName, fileType, status, ingestedAt) and no formal schema class or serialization library backing it.

Communication Patterns

Synchronous (JDBC — SQL Server):

The application writes to SQL Server via raw java.sql.DriverManager.getConnection() for every single INSERT operation. There is no connection pooling, no retry policy, and no timeout configuration. If the database is unavailable, the insert fails silently (the exception is caught and logged) and the file is moved to the error directory.

Asynchronous (AMQP — RabbitMQ):

After processing each file, the application publishes a JSON event to the zava.events topic exchange using routing key file.ingested. A new Connection and Channel are opened and closed for every publish. There is no connection pooling, no retry, and no dead-letter queue. If RabbitMQ is unreachable, the failure is logged and silently swallowed — the file move outcome is unaffected.

File System (Polling):

The primary ingestion trigger is a poll loop reading a shared directory (default /shared/file-ingestion) every 5 seconds using File.listFiles(). There is no file-locking mechanism, no deduplication guard, and no detection of partially-written files.

Resilience: No circuit breaker, bulkhead, or retry library is used. All resilience logic is implicit catch-and-log.

Service discovery: External service endpoints (SQL Server URL, RabbitMQ host/port) are hardcoded in fileingestion.properties with optional environment variable overrides. No service discovery mechanism is used.

Security posture: No authentication or TLS is configured at any communication layer. JDBC connections use a plaintext username/password from properties. RabbitMQ connections use plaintext credentials. The properties file ships with default credentials (sa / YourStrong!Passw0rd, guest / guest). No encryption in transit is enforced.

ZavaFileIngestion serves ZavaBank's back-office operations by automatically ingesting three categories of financial files — check images, wire transfer confirmations, and regulatory feeds — parsing them, recording their contents in staging tables, and notifying downstream systems of ingestion outcomes.

This is a single-service, single-bounded-context application. There are no upstream services providing data; the service is the source-of-truth producer for the staging tables it populates.

Primary Workflows

Workflow 1: File Discovery and Routing

The service runs a continuous poll loop (default every 5 seconds) that scans the configured watch directory for new files. Each file is classified by its name and routed to a type-specific processor. Files that cannot be classified are silently skipped. This workflow runs perpetually until the JVM receives a shutdown signal.

Steps:

  1. Poll the watch directory using File.listFiles().
  2. For each file found, call identifyFileType(fileName) to classify it.
  3. If the file matches a known pattern, delegate to the appropriate processor.
  4. If no pattern matches, skip the file (no move, no event).

Classification rules:

  • Filename contains check and ends with .csv → check_images
  • Filename contains wire and ends with .txt → wire_confirmations
  • Filename contains reg or feed and ends with .csv or .txt → regulatory_feeds
  • Filenames containing processed or error are explicitly excluded to prevent reprocessing of already-handled files.
  • A secondary fallback dictionary (FILE_TYPE_HINTS) maps the keywords check, wire, and reg without strict extension requirements.

Workflow 2: Check Image Ingestion

Processes CSV files containing check image metadata. Each file is expected to contain a single CSV line with three fields. The workflow reads the first (and only first) line; subsequent lines are not processed.

Steps:

  1. Open the file for reading (UTF-8).
  2. Read the first line and split on commas.
  3. Extract: checkNumber (field 0), accountNumber (field 1), imagePath (field 2).
  4. Execute INSERT INTO StagingCheckImage (FileName, CheckNumber, AccountNumber, ImagePath, IngestedAt).
  5. On success: publish file.ingested event with status=processed, move file to processed/.
  6. On failure (parse error or DB error): publish file.ingested event with status=failed, move file to error/.

Workflow 3: Wire Confirmation Ingestion

Processes fixed-width text files containing wire transfer confirmation records. Each line is a separate record. Empty lines are skipped. At least one successful DB insert is required for the file to be marked as processed.

Steps:

  1. Open the file for reading (UTF-8).
  2. For each non-empty line, slice fixed-width fields:
    • Characters 0–15: wireReference
    • Characters 16–27: amount
    • Characters 28–67: beneficiary
  1. Execute INSERT INTO StagingWireConfirmation (FileName, WireReference, Amount, Beneficiary, IngestedAt) per line.
  2. If at least one insert succeeds: publish status=processed, move file to processed/.
  3. If all inserts fail or file is unreadable: publish status=failed, move file to error/.

Workflow 4: Regulatory Feed Ingestion

Processes CSV or plain-text regulatory feed files. Each non-empty line represents one regulatory report item. At least one successful DB insert is required for the file to be marked as processed.

Steps:

  1. Open the file for reading (UTF-8).
  2. For each non-empty line, split on commas.
  3. Extract: feedCode (field 0), recordDate (field 1), payload (field 2).
  4. Execute INSERT INTO StagingRegulatoryFeed (FileName, FeedCode, RecordDate, Payload, IngestedAt) per line.
  5. If at least one insert succeeds: publish status=processed, move file to processed/.
  6. If all inserts fail or file is unreadable: publish status=failed, move file to error/.

Workflow 5: Ingestion Event Publishing

After every file processing attempt (success or failure), the service publishes a JSON event to RabbitMQ. This is a fire-and-forget notification; failures do not affect the file move outcome.

Event payload structure:


{
  "eventType": "file.ingestion",
  "fileName": "<file name>",
  "fileType": "<check_images|wire_confirmations|regulatory_feeds>",
  "status": "<processed|failed>",
  "ingestedAt": "<Unix epoch milliseconds>"
}

Cross-Service Data Flows

ZavaFileIngestion operates as a pure producer. It writes to SQL Server staging tables and publishes AMQP events; it does not consume data from any other service. Downstream consumers (not present in this repository) are expected to:

  • Read from StagingCheckImage, StagingWireConfirmation, and StagingRegulatoryFeed directly or via a separate read service.
  • Subscribe to the zava.events RabbitMQ exchange with binding key file.ingested to react to ingestion outcomes in near-real-time.

There are no gateway aggregation patterns, no cross-service REST calls, and no circuit breaker fallbacks — the service has no downstream service dependencies beyond its two infrastructure components (SQL Server and RabbitMQ), and failures in those are handled as silent per-operation errors.

Business Rules & Decision Logic

File Classification Rules

  • A file is classified as check_images if its lowercase name contains check and ends with .csv, OR if it contains the keyword check (fallback dictionary).
  • A file is classified as wire_confirmations if its lowercase name contains wire and ends with .txt, OR if it contains the keyword wire.
  • A file is classified as regulatory_feeds if its lowercase name contains reg or feed and ends with .csv or .txt, OR if it contains the keyword reg.
  • Files whose names contain processed or error are explicitly excluded from classification to prevent reingestion.
  • If a file matches no classification rule, it is skipped with no side effects.

Ingestion Success Criteria

  • Check images: The file is considered successfully ingested if the single INSERT for the first line succeeds. Only one record per file is expected.
  • Wire confirmations and regulatory feeds: The file is considered successfully ingested (status=processed) if at least one line insert succeeds. Partial success (some lines insert, others fail) still results in status=processed and the file is moved to processed/.

Error Handling Rules

  • Parse errors (malformed CSV, encoding issues) are caught per-file. The file is moved to error/ and a status=failed event is published.
  • Database insert failures are caught per-insert. The failure is logged but does not immediately abort the file — for multi-line files, processing continues for remaining lines.
  • RabbitMQ publish failures are caught and logged but do not affect the file move outcome. A file can be moved to processed/ even if the event was never delivered.

State Transitions (File Lifecycle)


pending (in watch dir)
    ├── recognized + insert OK  →  processed (moved to processed/)
    ├── recognized + insert fail →  error (moved to error/)
    └── unrecognized             →  remains in watch dir (not moved)

Cross-Cutting Concerns

  • Transactions: No transaction management. Each INSERT is a standalone JDBC auto-commit. There is no rollback capability if a multi-line file partially inserts.
  • Deduplication: No deduplication guard. If the same file reappears in the watch directory (e.g., after a failed move), it will be processed again, resulting in duplicate staging records.
  • Audit / Logging: Each operation logs to stdout with prefix [ZavaFileIngestion]. No structured logging, no audit trail, and no log retention mechanism is implemented.
  • Authorization: No authentication or authorization is applied at any point. Any process with filesystem access to the watch directory can introduce files for ingestion.

Version & Compatibility Risks

The project targets Java 8 (sourceCompatibility = JavaVersion.VERSION_1_8), which reached end-of-life for free Oracle builds in 2019. While the Eclipse Temurin JRE 8 image used in the Dockerfile still receives Adoptium community updates, Java 8 is a significant modernization risk for cloud migrations that mandate Java 17 or 21 LTS runtimes. mssql-jdbc 12.6.1.jre8 is a recent driver release and is compatible with SQL Server 2019/2022, so no immediate driver risk exists. amqp-client 5.21.0 is a current release and carries no known end-of-life concern.

Notable Observations

  • No framework dependencies: The application uses no web framework, no dependency-injection container, and no ORM — all logic is in a single plain-Java class. This simplifies migration but also means all cross-cutting concerns (connection pooling, retry, observability) are absent.
  • No connection pooling library: Each SQL INSERT opens and closes a raw java.sql.Connection, which is expensive under load. A migration to cloud SQL (e.g., Azure SQL) would benefit from adding HikariCP or a similar pool.
  • No logging framework: The application uses System.out.println exclusively. A standard logging library (SLF4J + Logback or Log4j 2) should be added before cloud deployment to support structured, level-based logging.
  • Shadow JAR packaging: The Gradle Shadow plugin (7.1.2) bundles all runtime dependencies into a single fat JAR, which is well-suited for the current Docker-based deployment model.

Data Architecture & Persistence Layer

ZavaFileIngestion persists parsed file records into three SQL Server staging tables using raw JDBC with no ORM, no schema migration tooling, and no caching layer.

Caching Strategy

No caching layer is implemented. All database writes are direct, per-row INSERT operations with no in-memory buffering or second-level cache. There is no cache provider (EhCache, Redis, Caffeine, Hibernate second-level cache) configured.

Data Ownership Boundaries

ZavaFileIngestion owns all three staging tables within the shared ZavaBankCore SQL Server database. It writes exclusively to those tables; it issues no SELECT queries and never reads from the database. Downstream consumers are expected to read the staging tables directly or via a separate service — no cross-service data access pattern is defined within this repository.

The service uses a shared database model: SQL Server is also referenced by other Zava Bank services (inferred from the ZavaBankCore database name and the presence of the zava.events RabbitMQ exchange used by other services). There is no schema-per-service isolation; all staging tables reside in the same database instance.
