Data Storage & External Services

The service writes directly to SQL Server staging tables for check image metadata, wire confirmations, and regulatory feeds. It also depends on a RabbitMQ exchange for downstream event notification and a shared filesystem path that acts as the ingestion handoff point from upstream producers.

Key Architectural Decisions

  • Uses a single-process polling worker instead of an HTTP-facing service, so file availability drives execution.
  • Persists records with raw JDBC statements rather than an ORM or repository abstraction.
  • Publishes a lightweight topic event after each file is processed so downstream systems can react asynchronously.

DTOs & Contracts

No controller DTO classes, OpenAPI definitions, protobuf schemas, or GraphQL contracts are present. The service instead works with implicit integration contracts:

  • Check image metadata file: CSV row interpreted as check number, account number, and image path.
  • Wire confirmation file: Fixed-width text lines interpreted as reference, amount, and beneficiary.
  • Regulatory feed file: CSV rows interpreted as feed code, record date, and payload.
  • Published event contract: JSON payload containing eventType, fileName, fileType, status, and ingestedAt.

These contracts are mutable string-based payloads assembled directly in Main.java; no explicit serialization framework or schema registry is used.

Communication Patterns

The service uses a mixed synchronous and asynchronous integration model. File discovery and parsing happen synchronously in the local worker loop, followed by synchronous JDBC writes to SQL Server for each record or file. After processing completes, the worker publishes an asynchronous RabbitMQ topic message so downstream consumers can react without blocking the ingestion loop.

No service discovery, API gateway, retry policy, circuit breaker, timeout policy, or bulkhead pattern is implemented in code. Startup availability depends on the presence of the shared filesystem path and on SQL Server and RabbitMQ being reachable when a file is processed. Because the application has no inbound API surface, there is no API-level authentication or authorization layer; external access control is delegated to filesystem permissions, RabbitMQ credentials, and SQL Server credentials configured via properties or environment variables.

Core Business Workflows

ZavaFileIngestion supports the bank's back-office ingestion domain by turning inbound operational files into normalized staging records and processing events. The core workflow is straightforward but business-critical because it is the handoff point between external data providers and downstream banking systems.

Primary Workflows

Workflow 1: Process check image metadata

  1. The worker detects a file whose name indicates a check image CSV.
  2. The file is opened and the first CSV row is parsed into check number, account number, and image path.
  3. A staging row is inserted into StagingCheckImage.
  4. If the insert succeeds, the worker publishes a processed event and moves the file to the processed folder.
  5. If parsing or insertion fails, the worker publishes a failed event and moves the file to the error folder.

Workflow 2: Process batch wire confirmations or regulatory feeds

  1. The worker detects a .txt wire confirmation file or a regulatory .csv/.txt feed file.
  2. Each non-empty line is parsed according to its file type: fixed-width slicing for wires, CSV splitting for regulatory feeds.
  3. Each parsed row is written to the corresponding staging table.
  4. The workflow succeeds if at least one line is inserted; otherwise the file is treated as failed.
  5. A status event is published and the file is archived to the processed or error folder accordingly.

Cross-Service Data Flows

The business flow crosses infrastructure boundaries rather than application service boundaries. Upstream partners supply files through a shared folder, ZavaFileIngestion transforms those files into SQL Server staging records, and RabbitMQ broadcasts the ingestion outcome so other banking services can continue the broader business process. If SQL parsing or persistence fails, the business degradation is explicit: the file is routed to the error folder and downstream systems receive a failure event instead of staged data.

Business Rules & Decision Logic

  • Classification rules: filenames containing check, wire, reg, or feed, combined with expected extensions, determine which workflow executes.
  • Validation rules: empty files fail immediately; blank lines are skipped for batch-oriented workflows.
  • Decision logic: check-image processing requires one valid first row, while wire confirmations and regulatory feeds require at least one successfully inserted line to count as a successful file.
  • State transition: every file ends in exactly one terminal state — processed or failed — and is moved to the corresponding archive folder.
  • Transactions and error handling: each insert is executed independently through JDBC; failures are logged, and the worker continues polling subsequent files.
  • Authorization and audit: no business-level authorization rules or audit trail beyond console logging and the emitted RabbitMQ status event are implemented in the repository.
