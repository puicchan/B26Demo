Data Storage & External Services

The service stores wire transfer records in a SQL Server WireTransfers table via JDBC. It also synchronously calls an external currency rate API and a ledger transactions API, then publishes transfer status events to RabbitMQ.

Key Architectural Decisions

  • Uses a servlet-only architecture with URL mappings in web.xml instead of Spring MVC.
  • Persists transfer state before returning API responses to keep status lookups queryable.
  • Uses synchronous HTTP calls for FX/ledger integration and asynchronous RabbitMQ events for downstream consumers.

The service supports wire transfer initiation and status tracking, including cross-service integration for FX conversion and ledger posting before publishing transfer outcome events.

Primary Workflows

Workflow 1: Submit Wire Transfer

  1. Client submits transfer request to /api/wire/transfer.
  2. Service validates required fields and routing number checksum.
  3. For non-USD transfers, service retrieves FX rate from currency service.
  4. Service attempts debit posting in ledger service.
  5. Service persists transfer with resulting status and reference number.
  6. Service publishes wire status event to RabbitMQ and returns response payload.

Workflow 2: Check Wire Transfer Status

  1. Client calls /api/wire/{id}/status.
  2. Service validates path pattern and numeric id.
  3. Service loads transfer row from SQL Server.
  4. Service returns status payload or not-found/error response.

Cross-Service Data Flows

Transfer submission orchestrates synchronous calls to currency and ledger services, then asynchronously emits a RabbitMQ event for downstream consumers. If ledger posting fails, the business outcome degrades to PENDING_LEDGER, preserving transfer data and failure reason while still returning a successful HTTP response with the pending status.

Business Workflow Sequence

sequenceDiagram participant User participant TransferApi as "WireTransferServlet" participant Fx as "Currency Service" participant Ledger as "Ledger Service" participant Store as "WireTransfers" participant Bus as "RabbitMQ" User->>TransferApi: Submit wire transfer request TransferApi->>TransferApi: Validate required fields and routing number alt Currency is not USD TransferApi->>Fx: Request FX rate Fx-->>TransferApi: FX rate else Currency is USD TransferApi-->>TransferApi: Use FX rate = 1 end TransferApi->>Ledger: Request debit posting alt Ledger posting succeeds Ledger-->>TransferApi: Debit accepted TransferApi-->>TransferApi: Mark status POSTED else Ledger posting fails Note over TransferApi: Mark status PENDING_LEDGER end TransferApi->>Store: Persist transfer and status Store-->>TransferApi: Generated transfer id TransferApi->>Bus: Publish status event TransferApi-->>User: Return transfer id and status

Business Rules & Decision Logic

  • Required fields rule: fromAccountId, beneficiary/account/routing details, and positive amount must be present.
  • Routing-number rule: ABA checksum must validate before processing.
  • FX decision rule: non-USD transfers call FX service; USD transfers use rate 1.
  • Ledger outcome rule: failed ledger posting does not reject transfer creation; persisted status becomes PENDING_LEDGER with failure reason.
  • Status query rule: path must match /api/wire/{id}/status and id must be numeric.
