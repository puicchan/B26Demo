Data Storage & External Services

The service uses SQL Server (ZavaBankDB) as its primary system of record for accounts, transactions, users, and session tokens. It integrates with zava-auth-gateway over HTTP to resolve .ZAVAAUTH cookies into session tokens and calls zava-ledger over HTTP/XML to post settlement transactions.

Key Architectural Decisions

  • Uses Struts Action mappings (*.do) with server-rendered JSP pages instead of REST controllers.
  • Uses direct JDBC PreparedStatement calls rather than an ORM/repository abstraction.
  • Implements cross-ecosystem SSO by calling an external auth gateway to decode legacy cookie-based authentication.

DTOs & Contracts

Gateway-level/request DTOs are Struts form objects LoginForm and PaymentForm, used as request contract carriers for /login.do and /makePayment.do. Response/view models include SessionUser, AccountOption, and PaymentHistoryRecord, which are mutable POJOs passed to JSP pages for rendering. No OpenAPI/Swagger definitions, GraphQL schema, or protobuf contracts were found. Serialization for downstream service calls is custom/manual: XML payload construction for ledger POST requests and lightweight string extraction from JSON for auth-gateway WhoAmI responses.

Communication Patterns

Communication is synchronous throughout. Inbound browser requests are dispatched by Struts ActionServlet; actions then execute direct JDBC queries against SQL Server and optional HTTP calls to external services. SsoSessionService performs token resolution via DB lookup, with fallback GET to Auth Gateway (WhoAmI.ashx) when only cookie auth is present. PaymentAction sends XML over HTTP POST to ledger and only continues when <status>POSTED</status> is returned. Timeout-based resilience is present (3s for auth-gateway calls, 8s for ledger calls), but no circuit breaker, retry loop, service discovery, or client-side load balancing framework is implemented. Security posture at API contract layer is limited: endpoints do not enforce TLS/authz internally and rely on session token validation logic plus upstream network assumptions.

Core Business Workflows

ZavaPayGateway supports customer payment operations by authenticating banking sessions, submitting payments, and presenting transaction history. Workflows are centered on web-form interactions and synchronous coordination with internal banking systems.

Primary Workflows

Workflow 1: Session-Based Login and Access Establishment

Entry point: GET/POST /login.do

  1. User supplies token directly or arrives with session token sources (query/header/cookie).
  2. Gateway attempts token validation against session/user records.
  3. If cookie-only path is used, gateway calls auth service to translate cookie into session token and retries lookup.
  4. On success, session context is established and user is redirected to payment entry.
  5. On failure, login screen is shown with token-expired/not-found message.

Workflow 2: Submit Payment and Transition to History View

Entry point: POST /makePayment.do

  1. Gateway verifies active session and loads selectable source accounts.
  2. User-submitted payment data is validated for required fields and positive amount.
  3. Gateway composes settlement request with generated reference number and alternate settlement account rule.
  4. Gateway submits transaction to ledger endpoint.
  5. If ledger confirms posted status, user is redirected to history screen; otherwise, error is shown on payment page.

Workflow 3: Review Posted Payment History

Entry point: GET /paymentHistory.do

  1. Gateway ensures the user session is still valid.
  2. Gateway queries recent payment transactions tagged for gateway-originated postings.
  3. Records are normalized into display rows (amount format, payment type extraction, date formatting).
  4. User receives history table or empty-state message.

Cross-Service Data Flows

Cross-system composition is linear and request-bound: identity context is sourced from SQL session tables, optionally enriched by AuthGateway token translation when cookie auth is present, while payment completion depends on downstream ledger acceptance. Data merging is simple and local (gateway maps SQL rows to view models); if auth or ledger services are unavailable, the workflow degrades to user-visible login/payment errors rather than asynchronous retry.

Business Rules & Decision Logic

  • Validation rules: Payment submission requires account, payment type, and amount; amount must be greater than zero.
  • Decision logic: Settlement account must not equal debit account; service switches to a fallback account ID when conflict occurs.
  • State transitions: Payment request transitions from submitted input to posted-history entry only when ledger response indicates posted status.
  • Data integrity rules: Session establishment requires active token, active user, and non-expired token timestamp.
  • Error handling: Auth resolution failures and ledger failures short-circuit workflows into user-facing error messages with no queued retry.
  • Authorization model: Access control is session-token based; no explicit role-based authorization checks were detected in action handlers.
