Data Storage & External Services

ZavaPayGateway uses a single Microsoft SQL Server database (ZavaBankDB) for all persistent storage, including accounts, transactions, session tokens, and users. Database connectivity is managed through PayGatewayConnectionFactory using a plain JDBC DriverManager pattern (no connection pooling). Two external HTTP services are integrated: the ZavaLedger service (http://zava-ledger:8080/api/transactions) receives XML-formatted POST requests to record payment transactions, and the .NET Auth Gateway (http://zava-auth-gateway:8080/WhoAmI.ashx) is called to resolve .ZAVAAUTH FormsAuthentication cookies into session tokens when the Java application cannot decrypt them directly.

Key Architectural Decisions

  • Apache Struts 1.3 MVC: The application uses the classic Struts 1 Action/ActionForm/ActionForward pattern with struts-config.xml as the central routing configuration, mapping URL patterns (*.do) to Action classes.
  • Manual JDBC without ORM or connection pooling: All database interactions use raw PreparedStatement queries opened and closed per request, without a framework like Hibernate or a connection pool like HikariCP.
  • Configuration via properties file and environment variables: PayGatewayConfig reads from paygateway.properties as defaults, with environment variable overrides (e.g., DB_HOST, LEDGER_URL), providing basic externalized configuration suitable for containerized deployments.

DTOs & Contracts

Struts ActionForms (request binding):

  • LoginForm — binds the sessionToken string field from the login POST body.
  • PaymentForm — binds accountId, amount, paymentType, and memo from the payment submission POST body. All fields are raw String; type conversion and range validation are performed inline in PaymentAction.

View model / response objects (mutable Java beans):

  • AccountOption — carries account display data (id, number, type, formatted balance string) populated from SQL query results and placed into request scope for JSP rendering.
  • PaymentHistoryRecord — carries a formatted transaction row (id, account number, formatted amount, payment type, reference number, formatted date, status) placed into request scope for the history JSP.
  • SessionUser — immutable (constructor-initialized, no setters) value object holding the authenticated userId (int) and username (String). Stored in HttpSession under the key zavaPayGateway.user.

External XML contract (outbound — ZavaLedger):

ZavaPayGateway sends a hand-constructed XML body to ZavaLedger:


<transactionRequest>
  <debitAccountId>…</debitAccountId>
  <creditAccountId>…</creditAccountId>
  <amount>…</amount>
  <description>ZavaPayGateway:{paymentType}:{memo}</description>
  <referenceNumber>PGW-{timestamp}</referenceNumber>
</transactionRequest>

The ledger is expected to respond with <status>POSTED</status> on success. No formal schema (XSD, OpenAPI, protobuf) is defined or validated on either side.

No OpenAPI/Swagger, protobuf, or GraphQL definitions are present.

No Jackson, Gson, or JSON serialization library is used for inbound or outbound payloads (JSON parsing in SsoSessionService is implemented manually with string operations).

Communication Patterns

Synchronous — all inter-service communication is synchronous HTTP:

  • ZavaPayGateway → ZavaLedger: HttpURLConnection POST with an XML body; hardcoded connect and read timeouts of 8 000 ms each. No retry policy. Failure is handled by displaying an error message to the user and abandoning the transaction.
  • ZavaPayGateway → ZavaAuthGateway: HttpURLConnection GET forwarding the .ZAVAAUTH cookie; hardcoded connect and read timeouts of 3 000 ms each. No retry policy. Failure silently returns an empty token, falling through to unauthenticated state.

Asynchronous patterns: None. No message queues, event bus, or async processing is present.

Resilience patterns: No circuit breaker, bulkhead, or retry library (Resilience4j, Hystrix, etc.) is configured. Both outbound calls absorb exceptions silently (catch (IOException ignored)), providing a "fail-soft" behavior at the cost of silent failure visibility.

Service discovery: Services are addressed by hardcoded base URLs read from PayGatewayConfig (properties file key ledger.url / auth.gateway.whoami.url), overridable via environment variables (LEDGER_URL / AUTH_GATEWAY_WHOAMI_URL). No dynamic service registry (Eureka, Consul, Kubernetes DNS) is used.

Security posture: There is no transport-layer security (HTTPS/TLS) configured for any inbound or outbound connection. Authentication is entirely custom: the application resolves a session token from three sources (form POST field, X-Session-Token header, or Authorization: Bearer header) and falls back to calling the auth gateway when a .ZAVAAUTH cookie is present. Session tokens are validated against the SessionTokens table in SQL Server. There is no role-based authorization — once a session is established, any authenticated user can access all payment functions. No CSRF protection, SameSite cookie policy, or HttpOnly/Secure flags are set on any session attribute.

Startup Dependency Chain

ZavaPayGateway has two hard runtime dependencies and one soft dependency at startup:

  1. SQL Server (ZavaBankDB) — hard dependency. Session validation and all page rendering require a live database connection. No readiness probe, retry mechanism, or startup health-check is configured; the application will fail requests silently if the database is unavailable.
  2. ZavaLedger (http://zava-ledger:8080/api/transactions) — soft dependency for payment posting only. The app starts without it, but payment submissions will fail with a user-facing error.
  3. ZavaAuthGateway (http://zava-auth-gateway:8080/WhoAmI.ashx) — soft dependency for cookie-based SSO fallback. The app starts without it; users without a direct session token will be unable to authenticate if this service is down.

No Docker Compose depends_on, Kubernetes readiness probes, dockerize TCP wait, or any other startup ordering mechanism is configured. Services are expected to be available when the application container starts.

ZavaPayGateway operates as a read-heavy orchestrator: it does not own any domain entity. It reads account and session data from the shared database and delegates transaction writes to ZavaLedger via XML REST.

Primary Workflows

Workflow 1: User Authentication (Session Token Login)

A user arrives at the portal and must establish a session before accessing any payment functionality.

Steps:

  1. Browser requests /login.do — the login form is displayed.
  2. User submits a session token (via the form field, X-Session-Token header, Authorization: Bearer header, or through the .ZAVAAUTH cookie SSO path).
  3. The application validates the token against the SessionTokens table: the token must exist, be active (IsActive = 1), not expired (ExpiresAt > GETDATE()), and belong to an active user (u.IsActive = 1).
  4. Branch A — Token valid: A SessionUser (userId + username) is placed in the HTTP session; user is redirected to the payment form.
  5. Branch B — Token not found, .ZAVAAUTH cookie present: The app forwards the cookie to ZavaAuthGateway (GET /WhoAmI.ashx), extracts the sessionToken from the JSON response, and re-validates against the database (step 3).
  6. Branch C — All token sources exhausted: User sees an error message and remains on the login page.

Workflow 2: Payment Submission

An authenticated user submits a payment, which is validated, routed, and posted to the ZavaLedger.

Steps:

  1. Browser requests GET /makePayment.do — the app loads up to 30 active accounts from the database and displays the payment form with the user's name.
  2. User selects a debit account, enters an amount, chooses a payment type, and optionally provides a memo, then submits POST /makePayment.do.
  3. Validation (all fields required; amount must be a positive decimal):
    • Account, amount, and payment type must not be blank → error if missing.
    • Amount must parse as a valid BigDecimal and be greater than zero → error if invalid.
  1. Settlement account resolution: The app reads the configured settlement account ID. If the selected debit account matches the settlement account, the settlement account is switched to an alternate (account 1 → use 2; any other → use 1).
  2. A unique reference number is generated: PGW-{System.currentTimeMillis()}.
  3. The app posts an XML transactionRequest to ZavaLedger (connect and read timeout: 8 seconds) containing: debit account ID, credit (settlement) account ID, amount, description (ZavaPayGateway:{paymentType}:{memo}), and reference number.
  4. Branch A — Ledger confirms <status>POSTED</status>: User is redirected to the payment history page with a success message including the reference number and formatted amount.
  5. Branch B — Ledger unavailable, returns a non-200 response, or times out: User sees the error "Unable to post payment to ZavaLedger." on the payment form; no transaction is recorded.

Workflow 3: Payment History

An authenticated user views their recent payment transactions.

Steps:

  1. Browser requests GET /paymentHistory.do — session is verified (or established via SSO if not already present).
  2. The app queries the Transactions table for the 75 most recent rows where Description LIKE 'ZavaPayGateway:%', joined with Accounts to resolve account numbers, ordered by descending transaction date.
  3. Each row is parsed to extract the payment type from the description prefix (ZavaPayGateway:{paymentType}:…).
  4. Results are rendered as an HTML table with columns: Transaction ID, Account Number, Amount, Payment Type, Reference Number, Date, and Status.

Workflow 4: Health Check

The /health.do endpoint maps to HealthAction, which unconditionally returns the health.jsp page displaying "ZavaPayGateway - Online". No database connectivity check, downstream service probe, or JVM metrics are included. This is a static availability indicator only.

Cross-Service Data Flows

Payment submission flow (ZavaPayGateway → ZavaLedger → shared DB):

ZavaPayGateway constructs a debit/credit pair from user input and configuration, assigns a reference number, and posts it to ZavaLedger as XML. ZavaLedger writes the Transactions record to the shared ZavaBankDB database. ZavaPayGateway subsequently reads those same records back directly via JDBC when displaying the payment history. This creates an implicit read-after-write dependency between the two services through the shared database.

Fallback when ZavaLedger is unavailable: The user sees an error on the payment form and the transaction is not recorded. No retry, compensation, or queue-based fallback exists; the user must resubmit manually.

Authentication SSO flow (ZavaPayGateway → ZavaAuthGateway → shared DB):

When no direct session token is available but a .ZAVAAUTH FormsAuthentication cookie is present, ZavaPayGateway calls ZavaAuthGateway's WhoAmI.ashx endpoint (3-second timeout) to resolve the cookie to a session token. This token is then validated against the SessionTokens table in the shared database, same as a directly submitted token. Fallback when ZavaAuthGateway is unavailable: The resolved token is empty; the user is redirected back to the login page. Cookie-based SSO silently fails.

Business Rules & Decision Logic

Validation Rules (Payment Submission):

  • Account ID, amount string, and payment type are all required (blank check before parsing).
  • Amount must be parseable as a BigDecimal and strictly greater than zero. Non-numeric input or zero/negative amounts produce a user-facing error.
  • Memo is optional; it is included in the transaction description but does not affect submission validity.

Settlement Account Conflict Rule:

If the user-selected debit account ID equals the configured settlement account ID, the settlement account is automatically reassigned: if it was account 1, use account 2; otherwise, use account 1. This prevents a self-debit/credit transaction but relies on a naive fallback rather than a proper account selection strategy.

Reference Number Generation:

Every payment submission generates a reference number using PGW- concatenated with System.currentTimeMillis(). This is not collision-resistant under concurrent submissions (millisecond-precision only) and is not backed by a database sequence or UUID.

Transaction Description Convention:

Transactions posted via this gateway always carry a description of the form ZavaPayGateway:{paymentType}:{memo}. The history query depends on this prefix (LIKE 'ZavaPayGateway:%') to filter records; any transactions written by other systems using this prefix would appear in the history view.

Session Validation Rules:

A session token is considered valid only when all of the following hold in the SessionTokens table: IsActive = 1, ExpiresAt > GETDATE(), and the associated user has IsActive = 1. Only the most recently expiring matching token is used (ORDER BY st.ExpiresAt DESC TOP 1).

Transaction Management:

No @Transactional annotation or programmatic transaction scope wraps any operation. Each JDBC connection is opened and closed per request in auto-commit mode. There is no distributed transaction spanning ZavaPayGateway's database reads and ZavaLedger's transaction write — it is possible for the ledger to succeed but the subsequent history read to fail, with no reconciliation mechanism.

Error Handling:

All SQLException and IOException exceptions in data access and outbound HTTP calls are silently caught and ignored (catch (... ignored)). Business failures surface only as user-facing message attributes set in the HTTP request scope. No structured logging, audit trail, or alerting is produced on failure.

Authorization:

Any user with a valid session token can access all payment functionality. There is no role-based authorization, account ownership check, or limit on payment amounts beyond the positive-value constraint.

Version & Compatibility Risks

Apache Struts 1.3.10 is the most significant risk: Struts 1 has been end-of-life since 2013 and receives no security patches. It is incompatible with Jakarta EE 9+ (which moved from javax. to jakarta. namespaces), meaning a future container upgrade would break the application. The javax.servlet-api 3.1.0 (provided scope) reinforces that the application targets an older Servlet 3.1 container (e.g., Tomcat 8.x); moving to Tomcat 10+ would also require a namespace migration. The Microsoft SQL Server JDBC driver at 12.8.1 is current and does not pose compatibility concerns.

Notable Observations

  • Struts 1 is end-of-life: Apache Struts 1 has had no active maintenance or security releases since 2013. Migration to Spring MVC, Jakarta EE MVC, or a modern REST-based framework is strongly recommended before deploying to production.
  • No ORM or connection pooling: All database access uses raw JDBC via DriverManager.getConnection(). There is no HikariCP, c3p0, or similar connection pool, and no ORM layer (Hibernate/JPA), which creates risk of connection exhaustion under load and increases maintenance burden.
  • No logging framework declared: The project has no SLF4J, Log4j 2, or Logback dependency. Any diagnostics rely solely on stdout/stderr or servlet container logging.
  • Minimal dependency footprint: With only 4 declared dependencies, the application is self-contained but heavily manual in areas (JSON parsing, HTTP client calls) that modern libraries would simplify safely.

Caching Strategy

No caching layer is implemented. There is no Redis, EhCache, Caffeine, Hibernate second-level cache, or Spring Cache (@Cacheable) configuration in this project. Every request performs live database queries regardless of data volatility.

The account list query (SELECT TOP 30 … Accounts WHERE Status = 'Active') and session token validation query are executed on every page request with no result caching, which may contribute to latency under moderate load given the absence of a connection pool.

Data Ownership Boundaries

ZavaPayGateway and ZavaLedger share a single SQL Server database (ZavaBankDB). ZavaPayGateway reads from Accounts, AccountTypes, SessionTokens, and Users, and reads from Transactions for history display. ZavaLedger writes Transactions records via its REST API; ZavaPayGateway does not perform direct INSERT or UPDATE on Transactions — it delegates writes to ZavaLedger and then reads the results back directly from the shared database. This creates an implicit coupling between ZavaPayGateway and ZavaLedger's internal schema: the Description column format (ZavaPayGateway:{paymentType}:{memo}) is a convention that both services must honor.

The SessionTokens and Users tables are owned by the broader ZavaBank identity platform (written by ZavaAuthGateway); ZavaPayGateway treats them as read-only.
