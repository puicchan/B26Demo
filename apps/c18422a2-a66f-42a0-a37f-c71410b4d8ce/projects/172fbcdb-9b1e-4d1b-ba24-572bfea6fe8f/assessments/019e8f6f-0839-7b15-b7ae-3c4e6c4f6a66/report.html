Data Storage & External Services

The application persists batch execution logs to a Microsoft SQL Server database (ZavaBankCore) using the official Microsoft JDBC driver over a plain JDBC connection (no ORM). Three downstream HTTP microservices are called on each batch cycle: the interest accrual service, the statement generation service, and the daily reconciliation service — all reached via configurable base URLs and endpoints loaded from batchscheduler.properties or environment variable overrides.

Key Architectural Decisions

  • Single-class monolith: All scheduling, HTTP communication, JDBC persistence, and configuration loading are implemented in a single Main class with static helper methods, reflecting a minimal, dependency-light design.
  • Environment variable overrides: Every critical configuration value (service URLs, DB credentials, scheduler interval) can be overridden via environment variables, making the application container-friendly without code changes.
  • Run-to-completion batch model: Each scheduled execution runs all three service calls sequentially, aggregates their results, and writes a single BatchRunLog record — failures in individual services are recorded but do not abort the remaining calls.

DTOs & Contracts

No formal DTO or contract classes are defined. All request bodies are inline JSON string literals hardcoded in executeBatchRun():

  • {"job":"interest_accrual"} — sent to the interest accrual service
  • {"job":"statement_generation"} — sent to the statement generation service
  • {"job":"daily_reconciliation"} — sent to the reconciliation service

No response bodies are deserialized; the application reads the raw response body only for logging purposes and uses the HTTP status code (2xx vs non-2xx) as the sole success indicator. No OpenAPI/Swagger specification, protobuf schemas, or GraphQL schemas are present. Serialization is handled via raw String.getBytes(StandardCharsets.UTF_8).

Communication Patterns

Synchronous HTTP (outbound only): All three downstream service calls are synchronous HTTP POSTs using java.net.HttpURLConnection. Calls are made sequentially — interest accrual first, then statement generation, then reconciliation. A failure in one call does not abort subsequent calls; the result is recorded and the run continues.

No resilience patterns: There is no retry logic, circuit breaker, timeout escalation, or bulkhead pattern. A single configurable socket timeout (http.timeout.ms, default 15 000 ms) applies to both connection and read phases. If a downstream service is slow or unavailable, the scheduler blocks for up to 15 seconds per call before marking it as failed.

Service discovery: No service discovery mechanism is used. All downstream service hostnames are resolved via Docker / container DNS (plain hostnames such as zavainterestcalculator). URLs are configurable via batchscheduler.properties or environment variable overrides, but no dynamic registry is consulted.

No API gateway: Calls are made directly to each service without any intermediary gateway or load balancer.

Security posture: No authentication, no TLS, and no authorization are configured. All three downstream HTTP calls are plain HTTP with no Authorization header or token. The SQL Server connection uses a hardcoded username/password from the properties file (overridable via SQLSERVER_USERNAME / SQLSERVER_PASSWORD environment variables). There is no network-level or application-level security on the outbound calls.

Startup dependency chain: The scheduler starts immediately on container launch; it does not wait for downstream services to become healthy. If a downstream service is unavailable at the first scheduled run, the call is marked as failed and logged — no startup health gate is enforced.

Core Business Workflows

ZavaBatchScheduler is a financial batch orchestrator that drives three core end-of-day banking operations — interest accrual, account statement generation, and ledger reconciliation — on a configurable schedule and records the outcome of each cycle for audit purposes.

ZavaBatchScheduler acts as the orchestration context only — it does not own or read any domain data from the downstream services. Each downstream service is the sole source of truth for its own domain entities.

Primary Workflows

Workflow 1: Periodic Batch Execution Cycle

Trigger: Timer fires after scheduler.interval.ms (default: every 5 minutes; intended for nightly/periodic banking operations).

Steps:

  1. The scheduler wakes up and checks the running flag; if shutdown has been requested, the run is skipped.
  2. Record the batch run start time.
  3. Interest Accrual Job: Send an HTTP POST to the interest accrual service. Receive success or failure outcome.
  4. Statement Generation Job: Send an HTTP POST to the statement generation service. Receive success or failure outcome. This step executes regardless of the outcome of step 3.
  5. Daily Reconciliation Job: Send an HTTP POST to the reconciliation service. Receive success or failure outcome. This step executes regardless of the outcomes of steps 3 and 4.
  6. Aggregate outcome: Derive overall batch status — SUCCESS if all three jobs returned a 2xx response; FAILED if any job returned a non-2xx response or threw an exception.
  7. Audit log: Insert a BatchRunLog record into SQL Server with the start time, end time, aggregated status, and per-job boolean results.
  8. Wait for the next timer tick.

Business rule involved: All three jobs must succeed for the batch run to be marked SUCCESS. A partial success (e.g., only interest accrual failed) is still recorded as FAILED, with the details field identifying which jobs succeeded and which did not.

Workflow 2: Graceful Shutdown

Trigger: JVM receives SIGTERM or SIGINT (e.g., docker stop, kubectl delete pod, OS signal).

Steps:

  1. The JVM shutdown hook fires and sets running = false.
  2. scheduler.shutdownNow() is called, cancelling any pending scheduled task.
  3. If a batch run is currently in progress, it completes its current HTTP call (no mid-run interruption is enforced), but the run loop exits at the next iteration check.
  4. The application logs "Shutdown requested" and then "ZavaBatchScheduler stopped."

Cross-Service Data Flows

ZavaBatchScheduler uses a fire-and-forget orchestration pattern rather than data aggregation. It sends a minimal trigger payload to each downstream service and receives only a success/failure signal (HTTP status code) in return. No domain data is read back, merged, or forwarded between services.

Flow:


ZavaBatchScheduler → POST (trigger payload) → Interest Accrual Service → performs accrual internally
ZavaBatchScheduler → POST (trigger payload) → Statement Generation Service → generates statements internally
ZavaBatchScheduler → POST (trigger payload) → Reconciliation Service → reconciles ledger internally

Each downstream service operates independently on its own data store. ZavaBatchScheduler records only the aggregate outcome.

Fallback behavior: There is no circuit breaker. If a downstream service is unavailable (connection refused, timeout, or 5xx response), the call is marked as failed after the configured timeout (default 15 seconds). The scheduler records the failure in BatchRunLog and continues to the next scheduled cycle — there is no retry, no compensating action, and no alerting mechanism beyond stdout logging.

Business Rules & Decision Logic

Decision logic:

  • Overall batch status: The batch run is SUCCESS only when all three downstream jobs return a 2xx HTTP response. Any individual job failure results in an overall FAILED status for the run, regardless of how many jobs succeeded.
  • Independent job execution: A failure in one job does not abort the remaining jobs. All three are always attempted in the same cycle, ensuring that a transient failure in one service does not prevent the other two from being triggered.
  • Skip on shutdown: If the JVM is shutting down (running = false), in-progress timer invocations exit immediately without executing any batch work.

Error handling:

  • Individual job failures are caught as exceptions or non-2xx HTTP responses. The error message (exception text or HTTP body) is logged to stdout. No dead-letter queue, retry queue, or alerting system is in place.
  • A failure to insert the BatchRunLog record (e.g., SQL Server unavailable) is caught and logged to stdout; it does not re-throw or affect the scheduler lifecycle.

Audit/logging:

  • Every batch run — successful or failed — results in one BatchRunLog row providing a full audit trail: run type, start and end timestamps, overall status, and per-job success flags.
  • Supplementary diagnostic messages are written to stdout with ISO 8601 timestamps.

Authorization:

  • No authorization controls are applied. The scheduler runs as an unattended process with no user identity, session, or role-based access checks. Access to downstream services is unrestricted (plain HTTP, no authentication header).

Transactions:

  • No distributed transaction spans the three downstream service calls. Each call is independent; there is no saga or compensating transaction if one service succeeds and another fails.
  • The BatchRunLog INSERT is a single-row, auto-committed JDBC operation. It is not transactionally linked to the downstream service calls.

Transaction management: each INSERT opens a new DriverManager.getConnection() and closes it in a finally block. No explicit transaction boundary or rollback is configured; JDBC auto-commit applies.

Caching Strategy

No caching layer is used. The application has no @Cacheable, Redis, EhCache, Caffeine, or in-memory map caching. Each batch run reads configuration from properties/env at startup and interacts with SQL Server only for write operations.

Data Ownership Boundaries

ZavaBatchScheduler owns the BatchRunLog table as a write-only audit log. No other service in the observed codebase is expected to write to this table. Downstream microservices (zavainterestcalculator, zavastatementservice, zavaledger) manage their own data stores independently; this scheduler has no direct database access to those services and coordinates with them solely over HTTP.

The ZavaBankCore database is shared infrastructure — the BatchRunLog table co-exists alongside tables owned by other banking services. There is no schema-per-service isolation or logical namespace separation visible from this project. Cross-service data access is avoided: the scheduler reads no data from tables owned by downstream services.

However, the database connection credentials (db.username, db.password) are stored in plaintext in batchscheduler.properties with a default value of sa / YourStrong!Passw0rd.
