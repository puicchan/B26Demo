Data Storage & External Services

The application connects to a single SQL Server database (ZavaBankCore) via plain JDBC to insert records into the BatchRunLog table after each batch execution. There is no caching layer. Three downstream HTTP microservices are called per batch run cycle: the Interest Calculator Service, the Statement Generation Service, and the Reconciliation (Ledger) Service — all communicated with using JSON POST requests over HTTP. The amqp-client library (RabbitMQ) is declared as a dependency but is not actively used in the current codebase.

Key Architectural Decisions

  • Single-class monolith: The entire application logic resides in Main.java with no framework or DI container — scheduling, HTTP calls, JDBC, and config loading are all inline methods.
  • Configuration via properties + environment variables: A classpath properties file (batchscheduler.properties) provides defaults; environment variables (e.g. SQLSERVER_URL, INTEREST_SERVICE_URL) take precedence, enabling container-friendly configuration.
  • Fat-JAR deployment: The Shadow plugin bundles all dependencies into a single executable JAR, containerized with a minimal eclipse-temurin:8-jre base image.

DTOs & Contracts

ZavaBatchScheduler uses no DTO classes. All outbound request payloads are inline hardcoded JSON strings (e.g., {"job":"interest_accrual"}). Response bodies are read and logged as plain strings; no deserialization or response modelling is performed.

There is no OpenAPI/Swagger specification, no protobuf schema, and no GraphQL schema. Serialization is handled entirely by raw byte writes to the HTTP output stream (OutputStream.write(payload.getBytes(StandardCharsets.UTF_8))).

Communication Patterns

Synchronous HTTP (POST)

All three service calls are performed synchronously using java.net.HttpURLConnection. Calls are made sequentially within a single executeBatchRun() invocation: interest accrual → statement generation → daily reconciliation. A configurable timeout (default 15 000 ms, key http.timeout.ms) applies to both connect and read phases of each call.

No resilience patterns

There is no circuit breaker, retry logic, bulkhead, or fallback mechanism. A network timeout or non-2xx response is treated as a simple boolean failure (false), recorded in the log, and the overall batch cycle is marked FAILED. Subsequent cycles will still execute normally; there is no back-off or alerting.

No asynchronous messaging

Despite amqp-client (RabbitMQ) being declared as a dependency, no AMQP connections or channels are opened. All communication is strictly synchronous HTTP.

Service discovery

Service addresses are hardcoded URLs read from batchscheduler.properties (e.g., http://zavainterestcalculator:8080). Environment variables (INTEREST_SERVICE_URL, STATEMENT_SERVICE_URL, RECON_SERVICE_URL) can override these at runtime, enabling Docker/Kubernetes DNS-based resolution. There is no dynamic service registry (Eureka, Consul, etc.).

Security posture

No transport security (TLS/HTTPS) is configured — all HTTP calls use plaintext http://. No authentication, API keys, or authorization headers are sent to downstream services. The SQL Server connection string explicitly sets encrypt=false and trustServerCertificate=true, disabling TLS for database traffic as well. All service-to-service communication is fully unauthenticated.

Primary Workflows

Workflow 1: Scheduled Batch Cycle (Recurring)

Trigger: ScheduledExecutorService fires every scheduler.interval.ms milliseconds (default: 5 minutes / 300 000 ms). The first execution begins immediately at startup (initialDelay = 0).

Steps:

  1. The scheduler thread wakes and checks the running flag; if the application is shutting down, the cycle is skipped.
  2. executeBatchRun() records the cycle start timestamp.
  3. Interest Accrual: HTTP POST to {interest.service.url}{interest.service.endpoint} with payload {"job":"interest_accrual"}. A 2xx response sets interestOk = true; any other outcome (timeout, non-2xx) sets interestOk = false.
  4. Statement Generation: HTTP POST to {statement.service.url}{statement.service.endpoint} with payload {"job":"statement_generation"}. Same success/failure logic as above.
  5. Daily Reconciliation: HTTP POST to {reconciliation.service.url}{reconciliation.service.endpoint} with payload {"job":"daily_reconciliation"}. Same success/failure logic as above.
  6. Overall status is computed: SUCCESS only if all three jobs returned true; otherwise FAILED.
  7. A BatchRunLog record (RunType=nightly, StartedAt, EndedAt, Status, Details) is inserted into SQL Server.
  8. All results and any errors are emitted to stdout.

Business constraints:

  • Jobs run sequentially, not in parallel; a slow or timed-out call blocks the subsequent jobs in the same cycle.
  • There is no skip/compensate mechanism — all three jobs are always attempted regardless of earlier failures within the same cycle.
  • If the BatchRunLog insert fails, the failure is logged to stdout but does not affect the next cycle.

Workflow 2: Graceful Shutdown

Trigger: JVM shutdown signal (SIGTERM / Ctrl-C).

Steps:

  1. The shutdown hook sets running = false and calls scheduler.shutdownNow().
  2. Any in-progress batch cycle completes its current HTTP call; the scheduler does not start a new cycle.
  3. The main thread exits its polling loop and the JVM terminates.

Cross-Service Data Flows

ZavaBatchScheduler operates as a fire-and-forget orchestrator: it sends an HTTP POST to each downstream service and interprets a 2xx response as success. No data is extracted from the response body beyond reading it into a log string — there is no aggregation, merging, or cross-service join.

The three downstream calls are independent from a data flow perspective. The scheduler does not pass output from one service as input to the next. Each service performs its own domain logic internally and is solely responsible for any state changes within its own data store.

Fallback behaviour: If a downstream service is unavailable or returns an error, the scheduler records false for that job in the Details field of BatchRunLog and continues to the next job. There is no retry, dead-letter mechanism, or alerting — the degraded batch run is simply written to the audit log with status FAILED.

Business Rules & Decision Logic

Validation Rules

  • No input validation is performed — the scheduler generates all payloads internally as hardcoded strings; there is no user-supplied input.

Decision Logic

  • Batch cycle status: SUCCESS if and only if all three service calls return HTTP 2xx; FAILED if any one or more calls fail or time out.
  • Details field format: A concatenated string recording per-job boolean results, e.g. interest=true, statements=true, reconciliation=false, enabling per-job failure analysis in the audit log.
  • Shutdown guard: At the start of each scheduled invocation, the running flag is checked. If false (shutdown in progress), the cycle body is skipped — preventing a new batch run from starting during shutdown.

State Transitions

  • BatchRunLog record lifecycle: Append-only; each cycle produces exactly one immutable audit record. There are no status updates or deletions.

Business Constraints

  • Sequential execution: Interest accrual must complete (or time out) before statement generation begins; statement generation must complete before reconciliation begins. A 15-second timeout on each call means a worst-case cycle can take up to 45 seconds of HTTP blocking time plus JDBC write time.
  • No idempotency or deduplication: If the JVM restarts mid-cycle, the downstream services may have already processed some jobs; the scheduler has no mechanism to detect or skip already-executed jobs.
  • Single instance assumption: The scheduler is designed to run as a single instance. Running multiple instances concurrently would trigger duplicate batch jobs in each downstream service.

Audit / Logging

  • Every batch cycle produces one BatchRunLog database record and multiple stdout log lines (job responses, final status, any errors).
  • Database insert failures are caught and logged to stdout but do not propagate as exceptions or block future cycles.

Authorization

  • No authentication or authorisation is applied to outbound HTTP calls. Downstream services are assumed to be reachable only within a trusted network boundary.
  • No role-based or attribute-based access control is implemented.

ZavaBatchScheduler writes exclusively to its own BatchRunLog table in the shared ZavaBankCore SQL Server database. The three downstream services (zavainterestcalculator, zavastatementservice, zavaledger) are expected to own their own tables within the same database or separate stores; ZavaBatchScheduler does not read from any of those service-owned tables. All cross-service interaction occurs via HTTP (see api-service-contracts.md), not via direct database access. This is a write-only, append-only audit boundary with no read path and no join patterns.
