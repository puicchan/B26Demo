Data Storage & External Services

The application connects to a Microsoft SQL Server instance (ZavaBankCore database) via JDBC to persist BatchRunLog records after every batch run. It calls three external HTTP microservices over plain HTTP POST with JSON payloads: zavainterestcalculator for interest accrual, zavastatementservice for statement generation, and zavaledger for daily reconciliation. All service URLs and database credentials are configurable via batchscheduler.properties and can be overridden with environment variables, making the application container-friendly.

Key Architectural Decisions

  • Single-class monolith: All scheduling, HTTP calling, database writing, and configuration loading are implemented in one Main class, reflecting a simple, self-contained batch daemon design.
  • Environment variable overrides: Every critical configuration value (service URLs, DB credentials, interval) can be overridden via named environment variables, enabling twelve-factor app deployment in containers or Kubernetes.
  • Fixed-delay scheduling: Uses scheduleWithFixedDelay (not scheduleAtFixedRate) so that a slow or hung batch run does not cause overlapping executions.

DTOs & Contracts

No formal DTO classes are defined. The only data structures exchanged are:

  • Outbound request payloads: Plain JSON strings ({"job":"<job_name>"}) constructed as hard-coded string literals in callService(). No serialization library (Jackson, Gson) is used — payloads are written as raw UTF-8 bytes.
  • Inbound responses: Response bodies from downstream services are read as raw strings and written to the console log only; the HTTP status code (2xx vs non-2xx) is the sole contract tested.
  • BatchRunLog row: Not a DTO class — parameters are passed directly to a PreparedStatement in insertBatchRunLog().

No OpenAPI/Swagger specifications, protobuf schemas, or GraphQL schemas are present.

Communication Patterns

Synchronous HTTP: All inter-service communication uses java.net.HttpURLConnection with synchronous blocking reads. Connect timeout and read timeout are both controlled by http.timeout.ms (default 15 000 ms). Calls are sequential within each batch run: interest accrual → statement generation → daily reconciliation.

No resilience patterns: There is no retry logic, circuit breaker (Resilience4j, Hystrix, etc.), or bulkhead configuration. A transient downstream failure causes the job step to be marked as failed immediately, with the overall batch status set to FAILED if any step fails.

Service discovery: None. Downstream service URLs are hard-coded in batchscheduler.properties (e.g., http://zavainterestcalculator:8080) and resolved via container/host DNS. There is no Eureka, Consul, or Kubernetes-native service mesh integration.

Startup dependency chain: The application begins scheduling immediately on start. There is no wait-for-service readiness probe for the downstream HTTP services or the SQL Server database; connection failures are caught per-invocation and logged.

Security posture: No authentication, authorization, or TLS is configured for any outbound call. All HTTP calls are plain HTTP (http://) with no bearer token, API key, or mutual TLS. The SQL Server connection uses hardcoded credentials in batchscheduler.properties (username sa, password YourStrong!Passw0rd). No transport security or secret management mechanism is in place.

ZavaBatchScheduler orchestrates the nightly financial batch processing cycle for ZavaBank by sequentially triggering interest accrual, statement generation, and daily ledger reconciliation across three downstream microservices, then recording the overall run outcome for audit purposes.

ZavaBatchScheduler is the source of truth only for BatchRunLog; it does not own or read any domain data from the three downstream services. Communication is one-way: a fire-and-check HTTP POST with no response payload consumed beyond the HTTP status code.

Primary Workflows

Workflow 1: Nightly Batch Run

The core scheduled workflow runs every scheduler.interval.ms milliseconds (default 5 minutes) and orchestrates three financial processing jobs in sequence.

Steps:

  1. ScheduledExecutorService fires the batch task after the configured interval (initial delay = 0, so first run is immediate on startup).
  2. Record batch run start time.
  3. Interest Accrual: POST {"job":"interest_accrual"} to zavainterestcalculator. Mark interestOk = true if HTTP 2xx received within 15 s; otherwise interestOk = false and log the failure.
  4. Statement Generation: POST {"job":"statement_generation"} to zavastatementservice. Mark statementOk = true if HTTP 2xx received; otherwise statementOk = false.
  5. Daily Reconciliation: POST {"job":"daily_reconciliation"} to zavaledger. Mark reconOk = true if HTTP 2xx received; otherwise reconOk = false.
  6. Compute overall status: SUCCESS only if all three jobs returned true; otherwise FAILED.
  7. Record batch run end time and INSERT a BatchRunLog row with type nightly, timestamps, overall status, and the per-job boolean details string.
  8. Scheduler waits for the configured interval after task completion before triggering the next run (fixed-delay semantics).

Business rules involved: All three jobs must succeed for the batch to be marked SUCCESS. There is no partial-success state. Steps are strictly sequential — statement generation never starts until interest accrual has finished (or timed out).

Workflow 2: Graceful Shutdown

Steps:

  1. JVM receives SIGTERM (e.g., docker stop or kubectl delete pod).
  2. Registered ShutdownHook thread sets running = false and calls scheduler.shutdownNow(), interrupting any in-progress batch task.
  3. Main thread's while (running) loop exits.
  4. Process terminates.

Risk: An in-progress batch run interrupted by shutdown will not write a BatchRunLog row for that run, leaving a gap in the audit trail.

Cross-Service Data Flows

ZavaBatchScheduler does not aggregate or transform data across services. Each downstream call is a one-way trigger:

  • Trigger → Acknowledge: Scheduler sends a minimal JSON payload identifying the job; downstream service performs the actual work and responds with an HTTP status code only.
  • No response payload consumed: The body of the HTTP response is read and logged to stdout but is not parsed or forwarded to any other system.
  • Fallback behavior: When a downstream service is unavailable or returns a non-2xx status, the corresponding job flag is set to false, the batch run is logged as FAILED, and the scheduler continues to the next job without any retry. There is no circuit breaker, dead-letter queue, or compensating transaction.
  • No cross-service data join: Each trigger is independent; no data returned from one service is passed as input to the next.

Business Rules & Decision Logic

Validation rules:

  • Service URL and endpoint must both be non-null; if either is missing the job is skipped and the flag is set to false (no exception thrown).
  • HTTP response status code 200–299 is treated as success; any other code (including redirects) is treated as failure.

Decision logic:

  • Overall batch status is a strict AND of all three job results: a single job failure marks the entire run as FAILED.
  • Jobs execute unconditionally and sequentially regardless of prior job outcome — a failed interest accrual does not prevent statement generation or reconciliation from being attempted.

State transitions:

  • Batch run lifecycle: started → (all jobs attempted) → SUCCESS or FAILED → written to BatchRunLog.
  • No retry, no intermediate states, no manual re-run mechanism.

Scheduling constraint:

  • scheduleWithFixedDelay ensures the next run starts only after the current run fully completes (including the audit INSERT), preventing overlapping executions even if a run exceeds the configured interval.

Cross-cutting concerns:

  • Transactions: No database transaction wraps the batch execution or the BatchRunLog INSERT; each JDBC call is auto-committed. No saga or distributed transaction pattern is used.
  • Error handling: All exceptions (HTTP connection failures, SQL exceptions) are caught at the method level and logged to stdout. Errors do not propagate to the scheduler, so the scheduler remains alive and continues firing subsequent runs.
  • Audit / logging: Every batch run produces a console log line and a BatchRunLog row. Individual job HTTP responses (status code and body) are also logged to stdout.
  • Authorization: No authentication or authorization is applied to the outbound batch triggers; calls are plain unauthenticated HTTP POST requests.

Data Ownership Boundaries

ZavaBatchScheduler owns only the BatchRunLog audit table in the shared ZavaBankCore SQL Server database. The three downstream microservices (zavainterestcalculator, zavastatementservice, zavaledger) presumably own their own tables within the same or separate databases, but this application does not access those tables directly — it only calls their HTTP APIs. There is no CQRS pattern, no shared read model, and no cross-service direct database access from this application.
