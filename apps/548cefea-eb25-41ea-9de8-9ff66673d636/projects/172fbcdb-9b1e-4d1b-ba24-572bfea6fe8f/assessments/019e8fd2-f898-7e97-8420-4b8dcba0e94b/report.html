Data Storage & External Services

The application persists loan data in a Microsoft SQL Server database (ZavaBankDB) using two tables: LoanApplications (application lifecycle and decision data) and LoanDecisions (audit trail of decisions). The LoanBootstrapServlet creates these tables on startup if they do not exist. Three downstream REST services are called synchronously during loan application processing: a KYC Service for identity verification, a Risk Engine for credit scoring, and a Ledger Service for creating loan accounts once a loan is approved.

Key Architectural Decisions

  • Servlet-per-endpoint pattern: Each HTTP endpoint is implemented as a dedicated HttpServlet subclass with no MVC framework or dependency injection container.
  • Direct JDBC with manual connection management: Data access uses raw JDBC (DriverManager.getConnection) through a single LoanConnectionFactory; connections are opened per request and closed in finally blocks.
  • Configuration from properties file and environment variables: LoanOriginationConfig reads settings from loan-origination.properties with environment variable overrides, enabling container-based configuration injection.

DTOs & Contracts

The application has no formal DTO classes. All request and response payloads are handled ad-hoc using org.json.JSONObject deserialized directly from the raw HTTP request body and serialized to the response writer.

Inbound request fields (POST /api/loans/apply) — parsed inline within LoanApplyServlet.doPost:

  • customerId (int), loanProductId (int), requestedAmount (double), termMonths (int), purpose (string), fullName (string), idNumber (string), loanAccountTypeId (int)

Outbound response fields (POST /api/loans/apply):

  • status, applicationId, kycStatus, riskScore, riskLevel, approvedAmount, ledgerAccountCreated, ledgerAccountId

Status query response (GET /api/loans/{id}/status):

  • applicationId, customerId, status, requestedAmount, approvedAmount, interestRate, termMonths, decisionDate, decisionNotes, loanAccountId

There are no OpenAPI/Swagger specifications, protobuf schemas, or GraphQL schemas. No Jackson, JAXB, or custom serialization configuration is present — all marshaling is done manually via JSONObject.

Communication Patterns

Synchronous REST (outbound): LoanApplyServlet makes three sequential, blocking HTTP POST calls using java.net.HttpURLConnection with a 7-second connect timeout and 7-second read timeout. Calls are made in strict order — KYC first, then Risk Engine (only if KYC passes), then Ledger Service (only if risk engine approves). Base URLs are resolved from environment variables at runtime.

No resilience patterns: There is no circuit breaker, retry logic, bulkhead, or fallback mechanism. If any downstream service times out or returns an error, the exception is silently caught and an empty JSONObject is returned, causing the loan to be processed with default/zero values rather than failing explicitly.

No service discovery: Downstream service base URLs are hardcoded via environment variables (KYC_SERVICE_BASE_URL, RISK_SERVICE_BASE_URL, LEDGER_SERVICE_BASE_URL). No Kubernetes DNS, Eureka, or Consul integration is present.

No API gateway: The application is a single service with no gateway layer.

Startup dependency chain: LoanBootstrapServlet runs with load-on-startup=1 and attempts to create the LoanApplications and LoanDecisions tables before any requests are served. If the SQL Server is unavailable at startup, the application will fail to initialize.

Security posture: No authentication or TLS is configured. All four endpoints are publicly accessible with no authorization checks, no JWT/OAuth2 validation, no API keys, and no Spring Security or equivalent framework in place.

Core Business Workflows

ZavaLoanOriginationAPI automates the loan origination process for ZavaBank: accepting loan applications from customers, orchestrating KYC identity verification and credit risk scoring, and provisioning ledger accounts for approved loans.

Primary Workflows

Workflow 1: Loan Application Submission

The core business workflow. A client submits a loan request; the application is recorded and then routed through identity verification, risk assessment, and account provisioning in a strictly sequential pipeline.

Steps:

  1. Input validation — customerId, loanProductId, requestedAmount, and termMonths must all be present and greater than zero. Returns 400 Bad Request immediately if any are missing or invalid. fullName defaults to "Customer {customerId}" if blank.
  2. Application creation — A new LoanApplication record is inserted with Status = "Submitted".
  3. KYC verification — The KYC Service is called with the customer's ID, full name, and ID number. The response status gates further processing:
    • PASS → continue to risk assessment
    • Any other status (e.g., REVIEW, FAIL) → application is set to KYC_REVIEW, a ManualReview decision is recorded, and processing stops
  1. Risk scoring — The Risk Engine is called with the application ID, customer ID, requested amount, and term. The response determines approval:
    • score >= 620 or approved = true → continue to ledger provisioning with the engine's approvedAmount and interestRate
    • score < 620 or approved = false → application is set to Declined, a Decline decision is recorded, and processing stops
  1. Ledger account creation — The Ledger Service is called to open a new loan account for the customer. Outcome:
    • Account created (created = true) → application is set to Approved, a Approve decision is recorded
    • Account creation fails or service is unreachable → application is set to APPROVED_PENDING_LEDGER, an Approve decision is still recorded (the credit decision stands)
  1. Atomic persistence — All database writes (INSERT application + UPDATE application + INSERT decision) are committed in a single JDBC transaction. On any SQLException, the transaction is rolled back.

Workflow 2: Loan Status Query

A client queries the current status of a submitted loan application by its numeric ID. The system performs a single SQL lookup and returns the application record including decision outcome, approved amount, interest rate, and linked ledger account ID. Returns 404 Not Found if no record exists for the given ID.

Workflow 3: Schema Bootstrap (Startup)

On application startup, LoanBootstrapServlet runs before any request-handling servlets are available. It establishes a database connection and creates the LoanApplications and LoanDecisions tables if they do not already exist. If the database is unreachable, the servlet throws a ServletException and the application fails to start — no requests can be served.

Cross-Service Data Flows

The loan origination pipeline is a sequential synchronous orchestration: this service acts as the orchestrator, calling KYC → Risk → Ledger in strict order. Each call's result controls whether the next call is made.

  • KYC → Risk: The KYC status field is the gate. Only "PASS" (case-insensitive) advances to risk scoring; any other value short-circuits the workflow.
  • Risk → Ledger: The risk engine's approved boolean (or score >= 620 as the fallback heuristic) is the gate. Only an approved risk result advances to ledger account creation.
  • Ledger fallback: Unlike KYC and risk failures (which halt the process as business rejections), a ledger failure is treated as a temporary operational issue — the credit decision (Approved) is preserved and the application enters APPROVED_PENDING_LEDGER state, requiring a separate remediation path.
  • Data carried across boundaries: applicationId is generated on the first DB insert and passed to the Risk Engine. The Risk Engine returns approvedAmount and interestRate that are stored on the application record. The Ledger Service returns accountId that is stored as LoanAccountID.

Failure handling gaps: If the KYC, Risk, or Ledger services return an HTTP error or the connection times out (7-second timeout), the exception is silently swallowed and an empty response object is returned. This means:

  • A KYC service outage results in kycStatus = "REVIEW" (as if the customer failed KYC), routing the application to manual review instead of surfacing an error.
  • A Risk Engine outage results in riskScore = 0 and approved = false, causing all applications to be auto-declined during outages.

Transaction Boundaries

All three database writes in the apply workflow (INSERT application, UPDATE application, INSERT decision) are wrapped in a single JDBC transaction (setAutoCommit(false) / commit() / rollback()). This ensures atomicity: either all three writes succeed or none are persisted.

Business Constraints

  • Interest rate fallback: If the Risk Engine does not return an interest rate, the configured default (loan.default.interestRate, default 7.25%) is used.
  • Approved amount fallback: If the Risk Engine does not return an approved amount, the originally requested amount is used as the approved amount.
  • Silent downstream failure: Failures in KYC, Risk, or Ledger HTTP calls are silently absorbed (exceptions caught and empty responses returned), meaning service outages result in business-logic side effects rather than explicit errors to the caller.

Audit Trail

Every terminal state transition writes a LoanDecision record with:

  • DecisionBy = "LoanOriginationAPI" (hardcoded)
  • DecisionDate = GETDATE()
  • CreditScoreAtTime — risk score at decision time
  • RiskLevel — Low / Medium / High
  • Reason — human-readable explanation of the decision outcome
  • Conditions — ledger account creation status

No application-level logging (SLF4J, Log4j, etc.) is present. All error paths use empty catch blocks with no logging, making operational troubleshooting reliant entirely on the database audit trail.

org.json:json:20140107 is a 2014 release — over a decade old — and predates many API improvements and bug fixes in subsequent releases (current is 20240303+). It has no known CVEs at this version but using such an old artifact introduces risk if newer versions address correctness or security issues. javax.servlet-api:4.0.1 targets the Java EE 8 / Servlet 4.0 specification, which uses the legacy javax. namespace; the Jakarta EE 9+ ecosystem migrated to jakarta., meaning any future upgrade to a modern servlet container (Tomcat 10+, Jetty 11+) will require a namespace migration from javax.servlet to jakarta.servlet.

Notable Observations

  • Extremely minimal dependency footprint: Only 3 declared dependencies — the application relies entirely on direct JDBC and raw HttpURLConnection calls rather than any framework, ORM, or HTTP client library, which reduces surface area but also means all infrastructure concerns are implemented manually.
  • No logging framework declared: There is no SLF4J, Log4j 2, or Logback dependency; all errors are silently swallowed with empty catch blocks, creating an observability gap that will need to be addressed for production cloud deployments.
  • No security library: Authentication and authorization are entirely absent from the declared dependencies and source code — the API endpoints are unprotected.
  • org.json v20140107 is extremely outdated: This library version is from January 2014. While functional, it should be upgraded to a maintained release or replaced with a modern alternative (e.g., Jackson Databind, Gson) before cloud migration.

Data Architecture & Persistence Layer

ZavaLoanOriginationAPI persists data in two SQL Server tables (LoanApplications, LoanDecisions) accessed exclusively via raw JDBC with no ORM framework. Schema is bootstrapped programmatically at application startup.

Caching Strategy

No caching layer is configured. Every request opens a new JDBC connection, executes queries directly against SQL Server, and closes the connection. There is no connection pool (HikariCP, DBCP, C3P0), no query result cache, no second-level cache, and no distributed cache (Redis, EhCache, Caffeine).

The absence of connection pooling is a significant performance concern for any production or cloud-scaled deployment. Each DriverManager.getConnection(...) call creates a new TCP connection to SQL Server, which is expensive and will not scale under concurrent load.

Data Ownership Boundaries

ZavaLoanOriginationAPI is the sole owner of both LoanApplications and LoanDecisions within a single shared SQL Server database (ZavaBankDB). There is no database-per-service isolation; all tables reside in one database instance.

Cross-service data access is strictly one-directional: this service reads from and writes to its own tables only. References to customers (CustomerID), loan products (LoanProductID), and ledger accounts (LoanAccountID) are stored as plain integers; there is no foreign key enforcement or join logic reaching into other services' databases.

CQRS / read-write patterns: No CQRS pattern is implemented. The same application handles both command (apply/create) and query (status lookup) paths using the same JDBC connection against the same tables.
