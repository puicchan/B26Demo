The service persists loan applications and decisions in SQL Server using JDBC. It integrates with three external HTTP services: KYC verification, risk scoring, and ledger account creation, all configured through externalized base URLs.

  • Performs synchronous orchestration to KYC, risk, and ledger systems during loan submission.

DTOs & Contracts

The contract is JSON-object based with dynamic JSONObject usage rather than typed DTO classes. LoanApplyServlet accepts request-body JSON fields and returns a composed response that includes orchestration output (status, risk, approval amount, ledger status). LoanStatusServlet returns a JSON projection of database-backed application fields. No OpenAPI, protobuf, or GraphQL schema files were detected.

Communication Patterns

Communication is synchronous REST over HTTP via HttpURLConnection with 7000ms connect/read timeouts. The application orchestrates sequential downstream calls (KYC → Risk → Ledger) in a single request path and applies inline decision fallback logic: KYC non-pass yields review status, risk decline yields declined status, and ledger creation failure yields APPROVED_PENDING_LEDGER. Service discovery is static via configured base URLs; no gateway/service registry pattern is present. No TLS enforcement, authentication, or authorization checks were identified at API contract level.

Primary Workflows

Workflow 1: Submit Loan Application

  1. Client submits POST /api/loans/apply with customer and loan data.
  2. Service validates required request fields and persists initial application as Submitted.
  3. KYC verification is invoked; non-pass routes to manual review status.
  4. If KYC passes, risk scoring runs and determines approve/decline path.
  5. Approved applications attempt ledger account creation.
  6. Application and decision records are updated and committed transactionally.

Workflow 2: Check Loan Status

  1. Client calls GET /api/loans/{id}/status.
  2. Service validates URL pattern and numeric ID.
  3. Service retrieves current application snapshot from persistence and returns decision details.

Cross-Service Data Flows

The loan submission flow is an orchestration pattern inside LoanApplyServlet: request data is enriched progressively with KYC, then risk, then ledger outcomes. Each downstream service contributes decision-critical data (identity status, risk/approval metadata, account creation result). If any downstream stage yields a non-success business outcome (e.g., failed KYC, declined risk, ledger creation pending), the workflow degrades to a business-specific status without abandoning the entire request lifecycle.

Business Workflow Sequence

sequenceDiagram participant Customer participant LoanAPI as "Loan Origination" participant LoanDB as "Loan Database" participant KYC as "KYC Service" participant Risk as "Risk Engine" participant Ledger as "Ledger Service" Customer->>LoanAPI: Submit loan application LoanAPI->>LoanDB: Save application as Submitted LoanDB-->>LoanAPI: applicationId LoanAPI->>KYC: Verify customer identity KYC-->>LoanAPI: KYC status alt KYC requires review LoanAPI->>LoanDB: Update status KYC_REVIEW LoanAPI-->>Customer: Return manual review outcome else KYC passed LoanAPI->>Risk: Request risk score and decision Risk-->>LoanAPI: score and approval data alt Risk declined LoanAPI->>LoanDB: Update status Declined LoanAPI-->>Customer: Return decline outcome else Risk approved LoanAPI->>Ledger: Create loan ledger account Ledger-->>LoanAPI: account creation result alt Ledger created LoanAPI->>LoanDB: Update status Approved else Ledger pending Note over LoanAPI: Business fallback with pending ledger state LoanAPI->>LoanDB: Update status APPROVED_PENDING_LEDGER end LoanAPI-->>Customer: Return approved outcome end end

Business Rules & Decision Logic

  • Required-field validation enforces customerId, loanProductId, requestedAmount, and termMonths before processing.
  • KYC is a gating rule: only PASS proceeds to risk analysis; otherwise status becomes KYC_REVIEW.
  • Risk decision rule: explicit approved response governs outcome, otherwise score fallback (>=620) is used.
  • Ledger creation affects final approved state: success yields Approved, failure yields APPROVED_PENDING_LEDGER.
  • Persistence for submit flow is transactional (insert application + update + decision record) with rollback on SQL failures.
