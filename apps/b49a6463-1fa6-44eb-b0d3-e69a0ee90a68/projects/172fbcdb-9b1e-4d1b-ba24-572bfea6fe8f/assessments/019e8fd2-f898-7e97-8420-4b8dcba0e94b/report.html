Data Storage & External Services

The application does not manage its own database but depends on two external services. The Ledger HTTP API (zavaledger:8080) receives individual ACH transaction records over HTTP POST as JSON. RabbitMQ acts as the event bus: successful file processing publishes to the zava.events topic exchange with routing key ach.processed, while failures publish to the dead-letter exchange zava.dlx with routing key ach.failed. ACH files are staged on a shared local or mounted file system with three directories: incoming, processed, and error.

Key Architectural Decisions

  • Poll-based file ingestion: The service runs a tight polling loop (default 5 s) over a shared file-system directory rather than using a file-watch API, making it straightforward to run in any environment that can mount a volume.
  • Single-class monolith: All logic (parsing, HTTP posting, RabbitMQ publishing, file management) lives in Main.java with private static inner classes for domain objects (AchBatch, AchEntry).
  • Environment-variable configuration override: achprocessor.properties provides defaults; a fixed set of well-known environment variables (e.g., RABBITMQ_HOST, LEDGER_URL) can override any property at runtime without modifying the properties file.

Communication Patterns

Synchronous (HTTP REST):

For each ACH entry in a parsed file, the processor makes a blocking HttpURLConnection POST to the Ledger API. Both connect and read timeouts are set to 15 000 ms (configurable via http.timeout.ms). There is no retry logic, circuit breaker, or backoff policy — a single failure marks the entire ACH file as failed.

Asynchronous (AMQP — RabbitMQ):

After each successful entry post the processor opens a new AMQP connection, declares the target topic exchange as durable, and publishes a ach.processed event to the zava.events exchange. On file-level parse or post failure it publishes an ach.failed event to the zava.dlx dead-letter exchange. Each publish creates and tears down its own Connection and Channel.

Service discovery:

Services are located by hardcoded hostnames/ports in achprocessor.properties (e.g., zavaledger:8080, localhost:5672). Environment variables (LEDGER_URL, RABBITMQ_HOST, etc.) can override these at runtime. No service-discovery registry (Eureka, Consul, Kubernetes DNS) is used.

Resilience:

No circuit breaker, bulkhead, or structured retry is implemented. Transient HTTP or AMQP failures result in immediate failure of the in-flight ACH file and a failure event publication. The polling loop continues with the next file on the next tick.

Security posture:

No TLS/HTTPS is configured for the outbound Ledger HTTP call (plain http://). RabbitMQ credentials are stored in achprocessor.properties as plaintext (username/password). No authentication tokens, OAuth2, or JWT are present. All communication is unencrypted and unauthenticated beyond the RabbitMQ username/password.

Startup Dependency Chain

ZavaACHProcessor does not implement any startup dependency ordering mechanism. It starts immediately and begins polling the file system. If RabbitMQ or the Ledger API is unavailable at startup:

  1. RabbitMQ unavailable — A new AMQP connection is opened per publish call. If RabbitMQ is unreachable when an event needs to be published, the exception is caught and logged; the file processing result (moved to processed or error) is unaffected.
  2. Ledger API unavailable — Each transaction POST is attempted once. A connection exception or non-2xx response is caught and logged; the entry is counted as failed, and if any entry in the file fails, the file is moved to the error directory.
  3. File system unavailable — If the incoming directory does not exist, the polling loop logs a warning and waits for the next tick (no crash).

No dockerize, Kubernetes readiness probes, depends_on health checks, or startup-retry logic is configured.

Primary Workflows

Workflow 1: ACH File Ingestion and Transaction Posting

This is the single end-to-end business process of the application. It is triggered automatically by the polling loop on a configurable interval (default 5 seconds).

Steps:

  1. File discovery — The processor scans the incoming directory. Files whose names match processed or error (case-insensitive) are ignored; all other regular files are dispatched for processing.
  2. NACHA parsing — The file is read line by line. Record-type 5 lines (batch header) populate AchBatch fields; record-type 6 lines (entry detail) create AchEntry objects; record-type 8 lines (batch control) populate totals. If no entries are found or an exception occurs, the file is treated as a parse failure.
  3. Parse failure path — If parsing fails, the file is moved to the error directory and an ach.failed event is published with reason parse_error.
  4. Transaction posting — For each AchEntry in the batch, a JSON payload is POSTed to the Ledger API. The HTTP timeout is 15 seconds. A 2xx response records a success; any other response or exception records a failure.
  5. Success event publishing — For each successfully posted entry, an ach.processed event is immediately published to the zava.events RabbitMQ exchange.
  6. File archival (all entries succeeded) — If every entry in the batch posted successfully, the file is moved to the processed directory.
  7. File archival (any entry failed) — If one or more entries failed to post, the file is moved to the error directory and an ach.failed event is published with reason transaction_post_failed.

Workflow 2: Graceful Shutdown

When a JVM shutdown signal (SIGTERM) is received, the registered shutdown hook sets the running flag to false. The polling loop exits after its current sleep interval, and the process terminates cleanly. Any file currently being processed completes before shutdown.

Cross-Service Data Flows

ZavaACHProcessor is the sole initiator of all cross-service data flows. There is no data aggregation or composition — all flows are outbound and one-directional:

  • ZavaACHProcessor → Ledger API: For each AchEntry, a flat JSON document containing routing number, account number, amount, trace number, batch number, and file name is POSTed. The Ledger API's response body is not consumed by the processor (only the HTTP status code is checked). There is no read-back or reconciliation path.
  • ZavaACHProcessor → RabbitMQ: Events are published fire-and-forget. There is no consumer registration, no reply queue, and no acknowledgment consumed by this service. If RabbitMQ is unavailable at publish time the exception is logged but does not affect file archival.
  • No data flows into ZavaACHProcessor from external services: The application is a pure producer/writer. It does not call the Ledger API to read back posted transactions, and it does not consume any RabbitMQ queues.

Business Rules & Decision Logic

Validation rules:

  • An ACH file must contain at least one record-type 6 (entry detail) line to be considered valid. Files with zero entries are rejected with a parse error.
  • Only regular files are processed; subdirectories, and files whose names are exactly processed or error (case-insensitive), are skipped.
  • NACHA fields are sliced from fixed-width positions (as defined by the NACHA standard). Fields that fall outside the string length are silently truncated to an empty string — no length-validation exception is thrown.

Decision logic (all-or-nothing file semantics):

  • A file is considered fully successful only if every entry in the batch is posted to the Ledger without error. A single failure causes the entire file to be archived to the error directory and a failure event to be emitted.
  • Individual entry failures within a batch do not trigger an entry-level failure event; only the file-level ach.failed event is published.

State transitions (ACH file lifecycle):


incoming/ → [parse] → [post each entry] → processed/   (all entries OK)
incoming/ → [parse] → [post each entry] → error/        (any entry fails)
incoming/ → [parse fails]               → error/        (no valid entries / exception)

Error handling:

  • Parse exceptions and HTTP exceptions are caught, logged to stdout, and treated as failures — no retry, no re-queue.
  • RabbitMQ publish failures are caught and logged; they do not affect file archival or the polling loop.
  • File move failures are caught and logged; the original file may remain in the incoming directory and be re-processed on the next poll tick (no deduplication mechanism exists).

Audit / logging:

  • All processing steps are logged to stdout using a [ZavaACHProcessor] prefix. Logs include file name, success/failure counts, HTTP status codes, and exception messages. No structured logging format (JSON, MDC) is used.

Authorization:

  • No access control, user authentication, or role-based authorization is implemented. Any process with file-system write access to the incoming directory can inject ACH files for processing.

Version & Compatibility Risks

The application targets Java 8, which reached Oracle's end-of-life for free commercial support in 2019; the Eclipse Temurin 8 JRE used in the Docker image still receives community security updates from Adoptium, but upgrading to Java 17 or 21 LTS would widen the support window and improve performance. amqp-client 5.21.0 is a recent release and compatible with RabbitMQ 3.x/4.x. mssql-jdbc 12.6.1.jre8 is current, though the jre8 classifier means it cannot take advantage of newer JDBC 4.3 features available on Java 11+; upgrading to a jre11 or newer-classifier build after the Java runtime upgrade would close this gap. The Shadow (fat-JAR) plugin 7.1.2 is stable but not the latest release.

Notable Observations

  • Declared-but-unused database driver: mssql-jdbc is listed as a compile-time dependency but no SQL connection logic exists in the current source code. This adds unnecessary classpath weight to every container image and may confuse future maintainers; it should either be used or removed.
  • Minimal dependency surface: With only two runtime libraries the application has a very small attack surface and simple upgrade path — all three packages (Java, amqp-client, mssql-jdbc) can be bumped independently.
  • No logging framework: The application uses raw System.out.println wrapped in a log() helper rather than SLF4J or Logback, making structured/level-filtered logging unavailable without refactoring.
  • No dependency management BOM: There is no Bill of Materials (BOM) import; each dependency version is pinned individually, which is manageable at this scale but would benefit from a BOM if the dependency count grows.

Entity Model

ZavaACHProcessor uses two private static inner classes as in-memory transfer objects. They are not persisted by this application and carry no ORM annotations.

> Note: These classes are populated from fixed-width NACHA record fields (record types 5, 6, and 8) and exist only for the duration of a single file-processing iteration. They are not persisted to any database by this service.

Caching Strategy

No caching layer is implemented. ZavaACHProcessor is a stateless polling processor; each file is read fresh from disk, each transaction is POSTed synchronously, and each event is published over a new AMQP connection. There is no in-memory cache, second-level cache, or distributed cache.

Data Ownership Boundaries

ZavaACHProcessor acts as a data producer only — it reads NACHA files from a shared mount and forwards transaction records to the Ledger service via HTTP. It does not own or manage any database.

  • File system: The shared directory (/shared/ach-incoming) is the only persistent store this service directly accesses. Files move between three subdirectories (incoming → processed or incoming → error) to track processing state.
  • Ledger service: The Ledger API is the authoritative owner of the persisted transaction records. ZavaACHProcessor is a writer client; it has no read-back path and no visibility into the Ledger's internal schema.
  • RabbitMQ: Events published to the zava.events and zava.dlx exchanges are consumed by downstream subscribers (unknown to this service). ZavaACHProcessor is a pure publisher with no queue bindings or consumption.

There is no shared database, no schema-per-service topology, and no CQRS pattern. Cross-service data flow is entirely outbound and one-directional (file system → ZavaACHProcessor → Ledger + RabbitMQ).

Data Classification & Sensitivity

Entity / Field GroupSensitive FieldsClassificationControls in Place
AchEntryaccountNumber, routingNumber, individualName, individualIdPCI / PIINone — fields are transmitted in plaintext HTTP POST bodies and plaintext AMQP messages; no encryption-at-rest, no field masking, no TLS
AchBatchcompanyName, effectiveDate, totalDebit, totalCreditPCI (financial totals)None — included in AMQP event payloads without encryption

Bank account numbers (accountNumber) and routing numbers (routingNumber) are PCI-sensitive; individual names and IDs are PII. Neither the HTTP channel to the Ledger API nor the RabbitMQ channel uses TLS, and no field-level masking or encryption-at-rest is applied. This represents a significant security gap that should be addressed before any cloud migration.
