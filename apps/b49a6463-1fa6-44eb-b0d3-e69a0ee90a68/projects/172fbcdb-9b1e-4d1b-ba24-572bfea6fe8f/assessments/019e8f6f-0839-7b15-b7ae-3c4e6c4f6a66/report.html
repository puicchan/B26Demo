Data Storage & External Services

The service does not manage its own database; it relies on two external integrations. Processed ACH transactions are forwarded to the ZavaLedger service via HTTP POST (http://zavaledger:8080/api/ledger/ach). Outcome events (ach.processed, ach.failed) are published to a RabbitMQ broker (zava.events and zava.dlx topic exchanges) so that downstream consumers can react to payment results. File-system directories (/shared/ach-incoming, /processed, /error) serve as the durable input/output medium.

Key Architectural Decisions

  • Pull-based file polling: The service scans the incoming directory on a configurable interval (default 5 s) rather than using an OS file-watch API, keeping the integration simple and portable across container environments.
  • All configuration via properties + environment overrides: achprocessor.properties provides defaults, and every key can be overridden by a corresponding environment variable (e.g. RABBITMQ_HOST, LEDGER_URL), enabling twelve-factor-style deployment.
  • Per-entry fan-out with atomic file disposition: Each ACH entry is posted to the Ledger individually; the file is moved to processed/ only when all entries succeed, otherwise it is routed to error/, ensuring no partial acknowledgments.

Communication Patterns

Synchronous (HTTP)

ZavaACHProcessor uses the JDK's built-in HttpURLConnection to POST each ACH entry to http://zavaledger:8080/api/ledger/ach. Connect and read timeouts are both set to 15 000 ms (configurable via http.timeout.ms / HTTP_TIMEOUT_MS). No retry logic, circuit breaker, or backoff strategy is implemented — a failed HTTP call is logged and the entry is counted as a failure; if any entry in a file fails, the whole file is moved to the error directory.

Asynchronous (RabbitMQ AMQP)

Events are published via the RabbitMQ Java client (amqp-client 5.21.0). A new Connection and Channel are opened for every publish call and closed in a finally block — there is no connection pooling or persistent channel. Two topic exchanges are used: zava.events (routing key ach.processed) for success notifications, and zava.dlx (routing key ach.failed) for failure notifications. Both exchanges are declared as durable in exchangeDeclare.

Service Discovery

ZavaLedger is addressed by a hardcoded hostname (zavaledger) with no service registry (no Eureka, Consul, or Kubernetes DNS abstraction). The hostname and port are overridable via environment variables LEDGER_URL and LEDGER_ACH_ENDPOINT.

Resilience

No circuit breaker, retry policy, or bulkhead pattern is implemented. The only resilience mechanism is a configurable HTTP timeout (15 s default). Failures are surfaced only through log output (System.out.println).

Security Posture

No TLS/HTTPS, authentication, or authorization is configured. HTTP requests to ZavaLedger are sent over plain HTTP with no credentials, API keys, or tokens. RabbitMQ credentials (guest/guest by default) are stored in achprocessor.properties as plaintext and are overridable via environment variables, but no secret management mechanism (vault, Kubernetes secret, etc.) is enforced at the code level. All endpoints are effectively unauthenticated.

Core Business Workflows

ZavaACHProcessor automates NACHA-format Automated Clearing House batch payment processing for the Zava Bank platform, ingesting ACH files from a shared directory and routing each transaction to the core ledger while publishing outcome events for downstream consumers.

Primary Workflows

Workflow 1: ACH File Ingestion and Transaction Posting

Trigger: Timer-based — the polling loop wakes every ach.poll.interval.ms (default 5 000 ms) and lists all files in the incoming directory.

Steps:

  1. Discover files — Scan /shared/ach-incoming for regular files, skipping sub-directories and items whose name is literally "processed" or "error" (the managed output subdirectories).
  2. Parse NACHA file — Read the fixed-width NACHA record file. Extract batch header (record type 5), individual entries (record type 6), and batch control (record type 8). Build an in-memory AchBatch with its AchEntry list.
  3. Validate parse output — If no entries were found (empty or malformed file), classify the file as a parse failure immediately.
  4. Post entries to Ledger — For each AchEntry, issue a synchronous HTTP POST to ZavaLedger (/api/ledger/ach, 15 s timeout). Track success and failure counts.
  5. Publish per-entry events — For every successful ledger post, publish an ach.processed event to the zava.events RabbitMQ exchange.
  6. Determine file outcome — If all entries succeeded, the batch is complete. If any entry failed, the batch is in error.
  7. Dispose file — Move the source file atomically: to /processed on full success, or to /error on any failure or parse error.
  8. Publish batch-level failure event — If the file was moved to /error, publish an ach.failed event to the zava.dlx dead-letter exchange.

Workflow 2: Startup Initialization

Trigger: Application start (Main.main()).

Steps:

  1. Load configuration from achprocessor.properties, then apply any environment variable overrides.
  2. Register a JVM shutdown hook that sets the running flag to false, signalling the polling loop to terminate cleanly.
  3. Ensure the /processed and /error directories exist (create them if absent).
  4. Start the polling loop — enter Workflow 1 on each iteration.

Cross-Service Data Flows

ZavaACHProcessor uses a push-only data flow; it never reads data back from ZavaLedger or RabbitMQ.

ZavaACHProcessor → ZavaLedger (synchronous, per-entry):

  • For every AchEntry, ZavaACHProcessor sends the routing number, account number, amount, trace number, and batch number to ZavaLedger via HTTP POST.
  • ZavaLedger is the system of record: it persists the transaction and returns an HTTP 2xx to confirm acceptance.
  • If ZavaLedger returns a non-2xx status or times out, the entry is marked as failed. No retry is attempted; the calling code immediately records the failure.

ZavaACHProcessor → RabbitMQ (asynchronous, fire-and-forget):

  • On each successful ledger post, an ach.processed event is published. Downstream services (audit, notifications, reconciliation) consume these events independently.
  • On batch failure, an ach.failed event is published to the dead-letter exchange, allowing operations teams or automated retry services to react.
  • If RabbitMQ is unavailable, the publish exception is caught and logged to stdout; file disposition proceeds regardless.

Fallback behavior:

  • ZavaLedger unavailable → entire ACH file is moved to /error; an ach.failed event is attempted.
  • RabbitMQ unavailable → event publish silently fails; no impact on file disposition or ledger posting.

Business Rules & Decision Logic

Validation Rules

Decision Logic

All-or-nothing batch disposition: A file is only moved to /processed when every single AchEntry in the batch is successfully posted to ZavaLedger. A single failed entry causes the entire file to be classified as an error and moved to /error. This conservative rule avoids partial-acknowledgment scenarios in a payment context.

Per-entry success tracking: Success and failure counts are accumulated across all entries. The final disposition decision is based solely on failCount == 0.

Failure reason codes (published in ach.failed event):

  • parse_error — file could not be parsed or contained no entries.
  • transaction_post_failed — at least one entry's HTTP POST to ZavaLedger returned a non-2xx status or timed out.

State Transitions


ACH File lifecycle:
  INCOMING → [parse] → PARSING
  PARSING  → [no entries] → ERROR (reason: parse_error)
  PARSING  → [entries found] → PROCESSING
  PROCESSING → [all entries POST to ledger succeed] → PROCESSED
  PROCESSING → [any entry POST fails] → ERROR (reason: transaction_post_failed)

Cross-Cutting Concerns

  • Transactions: No database transactions are used. The file-move operation serves as the durable commit point — it is atomic at the OS level (Files.move with REPLACE_EXISTING).
  • Error handling: Exceptions from HTTP calls, RabbitMQ publishing, and file I/O are all caught individually and logged to stdout. There is no global exception handler, retry mechanism, or dead-letter re-queue strategy.
  • Audit / logging: All significant events (file discovered, entry success/failure counts, file disposition, errors) are logged to stdout using a simple [ZavaACHProcessor] prefix. No structured logging framework (SLF4J, Logback) is used.
  • Authorization: No authorization checks — the service runs as a trusted internal process with full access to the shared file system, ZavaLedger, and RabbitMQ. Any process that can write to the incoming directory can initiate payment processing.
  • Concurrency: The polling loop is single-threaded; files are processed sequentially one at a time. There is no parallel processing or locking mechanism to prevent duplicate processing in multi-instance deployments.

Version & Compatibility Risks

The application targets Java 8 (sourceCompatibility = JavaVersion.VERSION_1_8), which reached end-of-life for most commercial distributions and is far behind the current LTS releases (Java 21, Java 17). The RabbitMQ AMQP Client 5.21.0 is a recent release and is compatible with current RabbitMQ broker versions. The MS SQL Server JDBC driver 12.6.1.jre8 is also a recent release; however, the jre8 classifier locks the driver to the Java 8 runtime — upgrading to a modern JDK would require switching to the jre11 or jre17 classifier. The Shadow (fat-JAR) Plugin 7.1.2 is compatible with Gradle 7.x but has not been tested against Gradle 8.x; the project's cached Gradle wrapper metadata references version 8.9, meaning a compatibility check would be required on upgrade.

Notable Observations

  • Minimal dependency footprint: With only two declared runtime dependencies, the service avoids common framework-induced complexity (no Spring Boot, no Quarkus) but also lacks structured logging, metrics, health-check endpoints, or retry/circuit-breaker libraries, which would be expected in a production-grade microservice.
  • Unused database driver: mssql-jdbc is listed as a compile-time dependency but no SQL or JDBC code is present in the single source file. This dead dependency should be removed to reduce image size and avoid inadvertent transitive vulnerabilities.
  • No dependency locking: There is no Gradle lockfile (gradle.lockfile) or version catalog (libs.versions.toml), so dependency resolution is non-deterministic across builds. Adding a lockfile or a BOM would improve build reproducibility.
  • Java 8 compiler target: Targeting Java 8 prevents use of newer language features (records, sealed classes, text blocks, virtual threads) and modern API improvements available in Java 11+, and will become a growing maintenance burden as library vendors drop Java 8 support.

Data Architecture & Persistence Layer

ZavaACHProcessor has no persistent data layer of its own; it operates as a stateless file processor that reads NACHA ACH files from a shared directory, holds data in transient in-memory models during a single processing pass, and delegates all persistence to the downstream ZavaLedger service.

ZavaACHProcessor defines two in-memory, non-persistent data models as private static inner classes of Main.java. They exist only for the duration of a single file-processing pass and are never serialized to a database.

Caching Strategy

No caching layer is implemented. Every polling cycle reads directly from the file system. There are no in-memory caches, Redis clients, EhCache configurations, or Spring @Cacheable annotations. Given the service's stateless, event-driven nature, caching is not applicable to the current design.

Data Ownership Boundaries

ZavaACHProcessor enforces a strict no-local-persistence boundary. It does not own or manage any database tables. The file system directories (/shared/ach-incoming, /processed, /error) are the only durable artefacts it touches, and even these are shared volumes supplied by the container orchestration environment.

All transaction data that needs to be durably recorded is forwarded to ZavaLedger via HTTP POST. ZavaLedger is the system of record for ACH transactions within the Zava Bank platform. ZavaACHProcessor never reads back from ZavaLedger — the relationship is strictly write-only (fire-and-forget per entry).

Cross-service data flow:

  1. ACH file arrives on the shared volume (written by an upstream partner feed or the orchestration platform).
  2. ZavaACHProcessor reads it, parses it, and for each entry makes a synchronous POST to ZavaLedger.
  3. ZavaACHProcessor publishes events to RabbitMQ for any downstream consumers (audit, notifications, reconciliation).
  4. No data is read back across service boundaries.

Summary of concerns: ACH data is highly sensitive — it contains bank routing/account numbers that are regulated under NACHA Operating Rules and PCI DSS. ZavaACHProcessor transmits this data over unauthenticated, unencrypted HTTP to ZavaLedger and publishes it to RabbitMQ exchanges with no field-level masking, encryption-at-rest, or access control. Although the data is not persisted locally, any interception of the HTTP channel or the RabbitMQ messages could expose full payment credentials. Encrypting the transport layer (TLS), redacting account numbers in event payloads, and securing RabbitMQ credentials are the minimum remediation steps required before production use.
