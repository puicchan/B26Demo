The application reads reporting data from a SQL Server database using direct SQL queries over ADO.NET. It also depends on an external authentication gateway (configured via AuthGatewayLoginUrl and AuthGatewayLogoutUrl) for login/logout redirection and session handoff.

Key Architectural Decisions

  • Uses Web Forms code-behind with direct SQL statements instead of repository/ORM abstractions.
  • Enforces Forms Authentication and redirects unauthenticated users to login before rendering report pages.
  • Keeps authorization and data access in-process within a single deployable web application.

Core Business Workflows

The application supports management users by providing operational loan reports with configurable date filters and authenticated access control. Its core workflows focus on secure dashboard access and on-demand report aggregation from banking transaction data.

Primary Workflows

Workflow 1: Authenticated Dashboard Access and Initial Report Load

  1. User requests /Default.aspx.
  2. System checks authentication state.
  3. If unauthenticated, user is redirected to /Login.aspx with a return URL.
  4. Login page directs user to external auth gateway.
  5. After authentication, user returns to dashboard.
  6. System initializes a default 7-day window and runs report aggregation.
  7. Dashboard renders portfolio summary, delinquency report, and daily transaction volume.

Workflow 2: Date-Filtered Report Refresh

  1. Authenticated user submits new start/end dates from dashboard filter controls.
  2. System validates the date range (start <= end and parseable dates).
  3. If invalid, workflow stops with a validation message.
  4. If valid, system executes report queries and updates all report sections.
  5. UI status is updated to indicate successful refresh.

Cross-Service Data Flows

This repository contains a single business service boundary with two external dependencies: authentication gateway and SQL Server. The auth gateway provides session establishment via redirect flow, while SQL Server supplies all reporting data. No multi-service aggregation, asynchronous event choreography, or circuit-breaker fallback pattern is implemented.

Business Rules & Decision Logic

  • Validation rules: Date inputs must parse as valid dates and end date cannot be earlier than start date.
  • Decision logic: Authentication gate determines whether user can access report generation workflow.
  • State transitions: Session state transitions from unauthenticated to authenticated before dashboard rendering.
  • Business constraints: Delinquency report only includes unpaid overdue entries; daily volume query is constrained to selected date window.
  • Computed values: Portfolio totals are aggregated by loan status; transaction volume is summed by day.
  • Transactions and errors: Report queries are read-only and handled independently; invalid input returns user-facing status without database calls.
  • Authorization: Global anonymous denial with explicit allow rules only for login/logout pages enforces protected dashboard access.
