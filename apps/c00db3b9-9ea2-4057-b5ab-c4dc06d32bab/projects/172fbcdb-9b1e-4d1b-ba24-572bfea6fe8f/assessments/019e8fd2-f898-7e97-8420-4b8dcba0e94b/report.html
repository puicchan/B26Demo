ZavaAuthGateway is a single-service authentication gateway implemented as an ASP.NET Web Forms application on .NET Framework 4.8. It serves interactive login pages and lightweight HTTP handlers that validate and expose authenticated user session state to other Zava Bank applications.

Data Storage & External Services

The application persists identity and session state in a SQL Server database referenced as ZavaBankDB. No cache, queue, or message broker is configured; the only externalized integration is the database connection, which may be sourced from DB_* environment variables or the web.config connection string, and the service is hosted through Mono xsp4 on port 8080.

Key Architectural Decisions

  • Uses server-rendered Web Forms pages for employee login while exposing dedicated HTTP handlers for machine-to-machine session checks.
  • Centralizes all persistence in a single AuthRepository, so page handlers and HTTP handlers share the same database access path.
  • Relies on encrypted Forms Authentication tickets that carry the raw session token in UserData for cross-application SSO validation.

DTOs & Contracts

The application does not define dedicated API DTO classes; instead it serializes lightweight JSON responses directly from handler code. AuthUser models the user record loaded from persistence, while SessionValidationResult represents the token validation response payload before it is written as JSON. The key contract artifact is the encrypted FormsAuthenticationTicket, which carries the session token in UserData and is exchanged through the .ZAVAAUTH cookie.

Communication Patterns

All interactions are synchronous HTTP requests handled in-process. Login.aspx validates credentials against SQL Server, creates a session token row, and then issues a forms auth cookie for subsequent browser requests. Downstream applications can call /api/auth/validate with a session token or /WhoAmI.ashx with the auth cookie to resolve identity information. No asynchronous messaging, retry policies, circuit breakers, or service discovery mechanisms are configured. Security is implemented with Forms Authentication and authorization rules in web.config; TLS termination is not defined in the repo, and machine clients rely on bearer-like session token exchange rather than OAuth or JWT.

Core Business Workflows

ZavaAuthGateway manages employee authentication and single sign-on session validation for the broader Zava Bank platform. Its business value is concentrated in credential verification, session issuance, identity lookup, and logout-driven session revocation.

Primary Workflows

Workflow 1: Employee login and session issuance

  1. An employee opens Login.aspx and submits username and password.
  2. The application loads the matching user record and rejects the request if the account is missing, inactive, or locked out.
  3. The entered password is combined with the stored salt and hashed with SHA-1 for comparison with the persisted password hash.
  4. On success, a new session token is created in the database, the user's last login timestamp is updated, and a forms auth ticket is issued in the .ZAVAAUTH cookie.
  5. The employee is redirected to Default.aspx, where the authenticated principal name is displayed.

Workflow 2: Session validation for downstream applications

  1. A downstream application calls /api/auth/validate with a session token through the query string or X-Session-Token header.
  2. The gateway verifies that the token exists, remains active, has not expired, and belongs to an active user.
  3. The gateway returns a compact JSON response indicating whether the token is valid and, when valid, includes the user identifier, username, and expiry timestamp.

Workflow 3: Identity lookup and logout

  1. A client with a valid .ZAVAAUTH cookie requests /WhoAmI.ashx to retrieve the current user profile.
  2. The gateway decrypts the forms auth ticket, resolves the user and role membership, and returns an authenticated JSON payload with display name, session token, and roles.
  3. When the user navigates to Logout.aspx, the gateway deactivates the stored session token, signs the user out, expires the auth cookie, and redirects back to the login page.

Cross-Service Data Flows

The repository contains a gateway-style integration pattern where other Zava applications consume this service's identity and session state over HTTP instead of reading authentication tables directly. The most explicit cross-service flow is WhoAmI.ashx, which returns the raw session token from forms auth ticket UserData so non-.NET applications can continue an SSO flow without being able to decrypt the .ZAVAAUTH cookie. If validation fails or the cookie cannot be decrypted, the service returns an unauthenticated response instead of partial user data.

Business Rules & Decision Logic

  • Username and password must both be present before login processing starts.
  • A user can authenticate only when the account exists, IsActive is true, and IsLockedOut is false.
  • Password verification compares the stored salted hash with a freshly computed SHA-1 hash of Salt + password.
  • Session validation succeeds only when the token is active, unexpired, and belongs to an active user.
  • Logout is authoritative: it deactivates the persisted session token and expires the browser cookie.
  • ApiAuthValidateHandler accepts only GET requests and returns a client error when no token is supplied.
  • WhoAmIHandler falls back to { "authenticated": false } whenever the cookie is missing, invalid, expired, or the user record is unusable.

Version & Compatibility Risks

The project targets .NET Framework 4.8 and relies on built-in framework assemblies instead of modern NuGet-managed packages, which limits portability and modernization options. The Dockerfile hosts the application on Mono 6.12, so compatibility risks are concentrated around legacy Web Forms and System.Web rather than third-party libraries.

Notable Observations

  • packages.config is empty, so dependency governance is implicit in the target framework rather than explicit package versioning.
  • The project depends on System.Web, which is tightly coupled to classic ASP.NET and cannot be carried directly into modern .NET.
  • SQL access is implemented with raw ADO.NET rather than an ORM package, reducing package count but increasing migration effort for data access.
