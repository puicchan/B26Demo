Data Storage & External Services

The application uses one SQL Server database (ZavaBankDB) for user identities, role mappings, and active session tokens. It exposes lightweight HTTP endpoints that are consumed by browser clients and upstream services that need session validation or identity lookup.

Key Architectural Decisions

  • Uses server-side Forms Authentication cookies and stores the raw session token in authentication ticket UserData for cross-service SSO handoff.
  • Uses a single repository (AuthRepository) with explicit SQL commands instead of an ORM.
  • Keeps health and token-validation endpoints as dedicated handlers for low-overhead checks.

The API contracts are manually serialized JSON payloads in handlers rather than formal OpenAPI DTO classes. Key contract models include AuthUser and SessionValidationResult in AuthRepository.cs, where SessionValidationResult is the response backing for token validation (valid, userId, username, expiresAt). No immutable record types, OpenAPI files, protobuf schemas, or GraphQL schemas were detected.

Communication Patterns

All communication is synchronous request/response HTTP from client to gateway, followed by synchronous SQL calls through AuthRepository. No asynchronous messaging, service discovery, circuit breaker, retry, timeout policy, or gateway composition across multiple backend services was identified. Security posture is partial: forms-auth cookie encryption and authorization rules exist in web.config, but API handlers such as /api/auth/validate and /WhoAmI.ashx are publicly reachable and rely on token/cookie validation within handler code; TLS termination is not configured in repository files.

Core Business Workflows

The application's core business function is authenticating users and issuing short-lived session tokens that can be reused by dependent services for SSO checks.

Primary Workflows

Workflow 1: User login and session creation

  1. User submits credentials on Login.aspx.
  2. Service loads user account and validates account status.
  3. Service validates salted password hash.
  4. On success, service creates a session token record and updates last login timestamp.
  5. Service issues forms-auth cookie containing username and session token reference.
  6. User is redirected to default page as authenticated session.

Workflow 2: Session token validation for dependent clients

  1. Client calls GET /api/auth/validate with token.
  2. Service validates token existence, active flag, expiry, and user active status.
  3. Service returns valid=true with identity summary when token is active.
  4. Service returns valid=false when token is missing, expired, or revoked.

Workflow 3: Identity lookup for active cookie session

  1. Client calls GET /WhoAmI.ashx with .ZAVAAUTH cookie.
  2. Service decrypts forms-auth ticket and validates expiry.
  3. Service resolves user profile and role list.
  4. Service returns authenticated identity payload with role list and session token.

Cross-Service Data Flows

Cross-service flow is token-based rather than direct orchestration: upstream services call this gateway to validate tokens and retrieve user identity details. When token checks fail, the gateway returns unauthenticated responses so callers can enforce access denial. No multi-service data aggregation workflow or asynchronous event flow is implemented.

Business Rules & Decision Logic

  • Authentication requires non-empty username and password and a matching salted SHA1 hash.
  • User account must be active and not locked out to proceed with login or identity retrieval.
  • Session tokens are valid only when marked active and not expired.
  • Logout revokes session token by setting inactive status and immediate expiration.
  • Access control defaults to deny anonymous users, with explicit allow-rules only for login and health endpoints.

Version & Compatibility Risks

The project targets .NET Framework 4.8 and relies on legacy ASP.NET WebForms and System.Web APIs. This stack is not supported on modern .NET runtime targets, so migration to net10.0 requires architectural and API-level replacement rather than in-place package updates.

Notable Observations

  • packages.config is empty; dependencies are mostly framework assemblies instead of NuGet-managed packages.
  • Data access uses inline SQL in repository methods, so migration effort will focus on data access abstraction updates.
  • Security is bound to FormsAuthentication cookie handling in System.Web, which has no direct equivalent on modern .NET.
