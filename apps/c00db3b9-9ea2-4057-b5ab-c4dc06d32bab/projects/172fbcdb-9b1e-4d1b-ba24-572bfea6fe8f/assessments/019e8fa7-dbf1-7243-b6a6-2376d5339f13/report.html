Data Storage & External Services

The application uses a single SQL Server database (ZavaBankDB) as its only persistence store, holding four logical tables: Users (credentials and profile), Roles, UserRoles (many-to-many mapping), and SessionTokens (active sessions with expiry and IP/user-agent metadata). Connection strings are resolved at runtime by DbConfig — environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) take priority over the web.config connection string, enabling container-based deployment. There are no external service integrations, caches, or message brokers.

Key Architectural Decisions

  • Direct ADO.NET data access: AuthRepository uses raw SqlConnection/SqlCommand with parameterized queries rather than an ORM, keeping the dependency footprint minimal.
  • Dual session token model: On login, a GUID session token is stored in SQL Server and also embedded in the encrypted Forms Authentication ticket (UserData), allowing Java/non-.NET applications to perform SSO validation via the api/auth/validate endpoint without decrypting the cookie.
  • Environment-variable-first configuration: DbConfig checks environment variables before falling back to web.config, supporting both traditional IIS hosting and containerized deployments (Dockerfile is present in the repository).

DTOs & Contracts

The application uses no formal DTO classes or OpenAPI/Swagger specification. Response bodies are serialized manually via string concatenation and HttpUtility.JavaScriptStringEncode.

Internal models used at the API boundary:

  • AuthUser — Internal entity populated from the Users table on login and by WhoAmI.ashx; not serialized directly (fields are projected into manual JSON strings). See data-architecture.md for field details.
  • SessionValidationResult — Internal DTO returned by AuthRepository.ValidateSessionToken; its UserId, Username, and ExpiresAt fields are serialized manually into the /api/auth/validate JSON response.

Response shapes (informal contracts):

GET /api/auth/validate — valid token:


{ "valid": true, "userId": 42, "username": "jsmith", "expiresAt": "2026-06-03T23:15:55.0000000Z" }

Invalid token: { "valid": false } — Missing token: { "valid": false, "error": "token is required" }

GET /WhoAmI.ashx — authenticated:


{ "authenticated": true, "username": "jsmith", "displayName": "John Smith", "sessionToken": "GUID", "roles": ["Admin"] }

Unauthenticated: { "authenticated": false }

No protobuf, GraphQL, or OpenAPI schemas exist in this repository.

Communication Patterns

Synchronous only: All communication is synchronous HTTP. There is no inter-service messaging, event streaming, or asynchronous queue integration.

Data access: AuthRepository communicates directly with SQL Server over ADO.NET SqlConnection. No ORM, connection pooling configuration, or retry policy is applied.

Gateway aggregation: Not applicable — this application is itself the authentication gateway. Downstream services (e.g., Java banking services) call /api/auth/validate passing a session token to verify identity without decrypting the .ZAVAAUTH cookie.

Resilience: No circuit breaker, retry policy, or timeout configuration (e.g., Polly) is present. SQL connection timeouts default to ADO.NET defaults (30 s command timeout).

Service discovery: None. The SQL Server address is resolved from environment variables (DB_HOST, DB_PORT) or web.config connection string — no service registry is used.

Security posture:

  • All pages and handlers are protected by ASP.NET Forms Authentication by default (<deny users="?" /> in web.config).
  • Public paths explicitly whitelisted: Login.aspx, Default.aspx, Logout.aspx, /api/auth/validate, /health, /WhoAmI.ashx.
  • Auth cookie (.ZAVAAUTH) uses AES encryption and SHA-1 HMAC via <machineKey> with hardcoded keys in web.config — a significant security concern for production deployments.
  • No TLS is configured at the application layer; TLS termination is expected at the load balancer or IIS level.
  • No OAuth2, JWT, or API key authentication — the token-based flow uses opaque GUID tokens stored in SQL Server.
  • WhoAmI.ashx sets Access-Control-Allow-Origin: *, permitting cross-origin reads of identity data.

Secrets Provisioning Workflow

Currently there is no automated secrets provisioning workflow. Secrets are managed via two mechanisms:

  1. Static web.config: The default SQL Server connection string and machine key are committed to the repository as plaintext. This is suitable only for local development and is a critical security risk if used in production — the SA credentials and machine key should be rotated and removed from source control.
  2. Runtime environment variables: When all five DB_* variables are present at container startup, DbConfig.GetConnectionString() constructs the connection string dynamically, bypassing the web.config entry. This is the intended container deployment path. No secret manager, vault, or managed identity integration is implemented.

Gap: The machineKey is not configurable via environment variables — it is always read from web.config. To achieve per-environment isolation, a web.config transform or a separate web.Production.config with a unique key per environment is required.

Core Business Workflows

ZavaAuthGateway is the centralized authentication service for the Zava Bank platform, responsible for verifying user identities, issuing session credentials, and enabling cross-ecosystem single sign-on so that Java and other non-.NET services can validate user sessions without accessing the proprietary cookie format.

This is a single-service application. Downstream banking services (e.g., Java loan or account services) are external consumers — they depend on ZavaAuthGateway for authentication but own their own domain data independently.

Primary Workflows

Workflow 1: User Login

A user submits their username and password on the Login page. The gateway verifies credentials against the Users table, creates a new SQL-backed session token, and issues an encrypted Forms Authentication cookie containing the session GUID.

Steps:

  1. User submits form with username and password.
  2. Input is validated: both fields must be non-empty.
  3. AuthRepository.GetUserByUsername fetches the user record.
  4. Account state is checked: user must exist, be active, and not be locked out.
  5. Password is verified: SHA-1 of (salt + password) is compared to the stored PasswordHash.
  6. A random GUID session token is generated and inserted into SessionTokens with a 30-minute expiry.
  7. LastLoginDate is updated on the user record.
  8. A Forms Authentication ticket is created, encrypted with the machine key, and set as the .ZAVAAUTH cookie.
  9. The user is redirected to Default.aspx.

Workflow 2: User Logout

A user requests logout. The gateway deactivates their SQL session token and clears the auth cookie.

Steps:

  1. The .ZAVAAUTH cookie is read and decrypted.
  2. The embedded GUID (ticket.UserData) is extracted.
  3. AuthRepository.DeactivateSessionToken sets IsActive=0 and ExpiresAt=NOW() on the token row.
  4. FormsAuthentication.SignOut() removes the server-side session state.
  5. An expired replacement cookie is written to overwrite the browser's cookie.
  6. The user is redirected to Login.aspx?loggedOut=1.

Workflow 3: Cross-Service Session Token Validation

A downstream service (e.g., a Java loan-processing service) receives a GUID token from the browser and needs to verify it without decrypting the .ZAVAAUTH cookie.

Steps:

  1. Downstream service calls GET /api/auth/validate with the token as a query parameter or X-Session-Token header.
  2. The gateway looks up the token in SessionTokens: it must be active, not expired, and the owner must be an active user.
  3. On success: returns { valid: true, userId, username, expiresAt }.
  4. On failure: returns { valid: false } (no expiry information is leaked).

Workflow 4: Identity Discovery (WhoAmI)

A browser-side script or downstream service needs to know who the currently authenticated user is, along with their roles.

Steps:

  1. Client calls GET /WhoAmI.ashx with the .ZAVAAUTH cookie present.
  2. The cookie is decrypted via FormsAuthentication.Decrypt.
  3. If the ticket is expired or missing, returns { authenticated: false }.
  4. AuthRepository.GetUserByUsername fetches the current user record.
  5. Account state is re-checked (active, not locked out).
  6. AuthRepository.GetUserRoles fetches all role names for the user.
  7. Returns { authenticated: true, username, displayName, sessionToken (GUID), roles[] }.

Cross-Service Data Flows

ZavaAuthGateway is the upstream provider in all cross-service flows — it does not consume other services. The two integration patterns are:

  1. Cookie-based SSO (browser clients): The browser presents the .ZAVAAUTH cookie to each service. .NET services can decrypt this directly using the shared machine key. Non-.NET services cannot.
  2. Token-based SSO (service-to-service): Non-.NET services extract the raw GUID session token from the browser (passed as ?sessionToken= by the portal) and call /api/auth/validate to verify identity. This is the primary cross-ecosystem integration point.

Fallback behavior: If ZavaAuthGateway is unreachable or /api/auth/validate returns valid: false, downstream services are expected to deny access. No circuit breaker or retry is implemented within this gateway — resilience is the responsibility of the calling service.

Business Rules & Decision Logic

Validation Rules

  • Login inputs: Username and password must both be non-empty strings (trimmed); empty submissions are rejected immediately without a database call.
  • Token input: Token for /api/auth/validate must be non-empty; returns HTTP 400 with { valid: false, error: "token is required" } if absent.
  • HTTP method guard: /api/auth/validate only accepts GET; all other methods receive HTTP 405.

Decision Logic

  • Account eligibility: On login, the user record must satisfy three conditions simultaneously — IsActive = true AND IsLockedOut = false AND password hash matches. Any failure produces the same generic error message to prevent username enumeration.
  • Session validity: A session token is valid only when IsActive = 1 AND ExpiresAt > GETDATE() AND the owning user's IsActive = 1. All three must hold.
  • Cross-ecosystem SSO routing: The sessionToken field in the WhoAmI response exposes the raw GUID so that portals can pass it to Java services as ?sessionToken= — enabling SSO without sharing the machine key.

Business Constraints

  • Session lifetime: Hard-coded to 30 minutes from creation (DATEADD(MINUTE, 30, GETDATE())). Sliding expiration is configured in Forms Authentication (slidingExpiration="true") for the cookie, but the underlying SQL token expiry is not extended — this creates a mismatch where the cookie may still be valid but the SQL token has expired.
  • Single source system: SourceSystem is hard-coded to "DotNet" when creating tokens; no provision for distinguishing tokens created by different client types.

Error Handling

  • Login, logout, and WhoAmI catch all exceptions silently with empty catch {} blocks, returning "not authenticated" rather than surfacing errors to the user. This makes diagnosing production failures difficult.
  • The /api/auth/validate endpoint does not catch exceptions — a SQL failure will propagate as an unhandled HTTP 500.

Authorization

  • All pages and handlers are protected by Forms Authentication by default (<deny users="?" />). Only explicitly whitelisted paths allow anonymous access.
  • No role-based authorization is enforced within this gateway; role data is returned to callers for downstream authorization decisions.
  • WhoAmI.ashx sets Access-Control-Allow-Origin: *, allowing any origin to read identity data — this should be restricted to known portal origins.

Version & Compatibility Risks

All runtime dependencies are built-in .NET Framework 4.8 assemblies with no external NuGet packages. .NET Framework 4.8 is in long-term support but is Windows-only and is the last major .NET Framework release — Microsoft will not ship new feature versions. System.Data.SqlClient (the ADO.NET SQL Server driver bundled with .NET Framework) has been superseded by Microsoft.Data.SqlClient, which receives active feature development, TLS updates, and security patches. System.Web (Web Forms, IHttpHandler, Forms Authentication) has no equivalent in .NET 6+/ASP.NET Core and represents the highest migration risk: Web Forms pages and .ashx handlers must be rewritten as Razor Pages/Minimal API endpoints, and Forms Authentication must be replaced with ASP.NET Core cookie authentication middleware.

Notable Observations

  • Zero third-party NuGet dependencies: The application relies exclusively on in-box .NET Framework 4.8 assemblies, which simplifies package management but also means the team is not benefiting from community libraries (e.g., Dapper for data access, Serilog for structured logging, or ASP.NET Core Identity for authentication).
  • SHA-1 used for password hashing: Login.aspx.cs uses SHA1.Create() for password hashing (salted), but SHA-1 is cryptographically weak for password storage — a migration concern independent of the .NET version upgrade.
  • Raw ADO.NET with inline SQL: AuthRepository uses raw SqlConnection/SqlCommand with embedded SQL strings. There is no ORM or query builder, which increases future maintenance cost and makes schema migration tracking harder.
  • No logging library: The application contains no logging or observability dependency. Failures are silently caught and swallowed in several catch {} blocks, making production diagnostics difficult.

Caching Strategy

No caching layer is implemented. Every request performs a live SQL Server query. There are no MemoryCache, IDistributedCache, Redis, or application-level cache decorators. For a read-heavy authentication endpoint (e.g., /api/auth/validate called per-request by downstream services), this means each call incurs a full round-trip to SQL Server.

Data Ownership Boundaries

ZavaAuthGateway owns all four tables exclusively. There is no other service in this repository with direct database access. Downstream services (e.g., Java banking applications) access user identity data only via the /api/auth/validate REST endpoint — they do not have direct database access, establishing a clean authentication boundary.

The SessionTokens table is the primary integration point: the GUID token created at login is embedded in the Forms Authentication cookie and also stored in SQL Server, allowing non-.NET services that cannot decrypt the cookie to call /api/auth/validate to verify identity.
