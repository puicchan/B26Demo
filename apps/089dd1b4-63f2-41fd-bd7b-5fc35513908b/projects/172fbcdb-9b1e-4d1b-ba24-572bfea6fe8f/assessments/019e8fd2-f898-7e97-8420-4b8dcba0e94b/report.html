Data Storage & External Services

The service stores metadata and binary document payloads in a SQL Server Documents table and communicates with the database over JDBC. No message broker, cache, or external API integration is defined in this repository.

Key Architectural Decisions

  • Uses a simple servlet-based monolith with one deployable WAR rather than layered framework abstractions.
  • Uses a startup bootstrap servlet to create the Documents table when the app initializes.
  • Reads database settings from environment variables first, then falls back to bundled property values.

DTOs & Contracts

The API contract is primarily JSON object construction within servlets using org.json.JSONObject and org.json.JSONArray, rather than dedicated DTO classes. Multipart upload contracts are defined by request part names (customerId, documentType, file) and response JSON keys (status, documentId, customerId, documentType, fileName).

Gateway-level DTOs are not present because this repository contains a single service and no aggregation gateway. Service-level contracts are mutable JSON objects created in request handlers. Serialization behavior relies on direct JSON string output and servlet response writers.

Communication Patterns

  • Synchronous patterns: Browser/client sends HTTP requests to servlet endpoints; servlets execute synchronous JDBC SQL operations.
  • Asynchronous patterns: None detected (no queue/topic/event handler usage).
  • Resilience patterns: No explicit retry, circuit breaker, timeout policy, or bulkhead library usage detected in code/build files.
  • Service discovery: None detected; database host is read from env/property configuration.
  • API gateway/composition: Not applicable for this single-service project.
  • Startup dependency chain: API availability depends on database reachability during bootstrap table creation.
  • Security posture: No API authentication, authorization checks, or TLS configuration is defined in this repository; endpoints appear publicly callable at the application level.

Core Business Workflows

ZavaDocVault supports the business process of storing and retrieving customer documents for banking operations. Core workflows cover document intake, customer-level document lookup, and secure file retrieval by identifier.

Primary Workflows

Workflow 1: Upload Customer Document

Entry point: client submits document upload form to POST /api/documents/upload.

  1. Service validates customerId and file presence.
  2. Service normalizes missing document type to General and sanitizes file name.
  3. Service reads uploaded file bytes.
  4. Service writes document record to persistence and returns uploaded confirmation.
  5. On validation failure, returns a client error; on persistence failure, returns server error.

Workflow 2: View Customer Documents

Entry point: client requests GET /api/documents/customer/{customerId}.

  1. Service validates path format and numeric customerId.
  2. Service queries persistence for customer documents ordered by upload recency.
  3. Service returns count and list payload for UI/API consumers.

Workflow 3: Download Document

Entry point: client requests GET /api/documents/{id}.

  1. Service validates numeric document ID.
  2. Service loads document metadata and binary content.
  3. If found, service streams file response as attachment.
  4. If missing, returns not found response.

Cross-Service Data Flows

No multi-service aggregation flow is implemented in this repository. All business flows are contained in the single ZavaDocVault service, which directly composes responses from its own SQL queries. When the database is unavailable, workflows degrade to error responses rather than fallback data composition.

Business Workflow Sequence

sequenceDiagram participant User participant App as "ZavaDocVault Service" participant DB as "SQL Server" User->>App: Submit document upload request App->>App: Validate customer and file input alt Validation fails App-->>User: Return error for invalid request else Validation succeeds App->>DB: Save document and metadata alt Save succeeds DB-->>App: Confirm document id App-->>User: Return uploaded confirmation else Save fails App-->>User: Return persistence error end end User->>App: Request customer document list App->>DB: Query documents by customer DB-->>App: Return matching rows App-->>User: Return document list User->>App: Request document download by id App->>DB: Query document content alt Document exists DB-->>App: Return file payload App-->>User: Return downloadable file else Document missing App-->>User: Return not found end

Business Rules & Decision Logic

  • customerId must be numeric for upload and list workflows; invalid values return client errors.
  • Upload requires a non-empty file; missing/empty file is rejected.
  • Empty or missing document type defaults to General.
  • Uploaded file name is sanitized to avoid path separator characters in stored output name.
  • Download workflow distinguishes between missing documents (not found) and storage failures (server error).
  • Transaction handling is implicit per JDBC operation; no explicit multi-step saga or cross-service transaction pattern is present.
