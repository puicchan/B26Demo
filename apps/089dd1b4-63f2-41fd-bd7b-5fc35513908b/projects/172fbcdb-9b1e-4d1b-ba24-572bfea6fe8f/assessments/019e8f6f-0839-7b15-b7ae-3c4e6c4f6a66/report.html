Data Storage & External Services

ZavaDocVault uses a single Microsoft SQL Server database (ZavaBankDB) as its only persistence store. The Documents table holds all document metadata (CustomerID, DocumentType, FileName, ContentType, UploadedDate) and the raw binary file content (VARBINARY(MAX)). The schema is bootstrapped automatically on startup via DocVaultBootstrapServlet using a DDL-safe IF OBJECT_ID … IS NULL CREATE TABLE statement. No external messaging, caching, or third-party API services are used.

Key Architectural Decisions

  • Direct JDBC without ORM: The application bypasses ORM frameworks and uses raw PreparedStatement calls for all SQL operations, keeping dependencies minimal but requiring manual resource management.
  • Bootstrap servlet for schema creation: DocVaultBootstrapServlet (load-on-startup=1) auto-creates the Documents table on first deploy, eliminating the need for a separate migration tool.
  • Environment variable configuration override: DocVaultConfig reads from doc-vault.properties first and allows any property to be overridden by a matching environment variable (e.g., DB_HOST, DB_PASSWORD), making the container image portable across environments.

DTOs & Contracts

ZavaDocVault does not use dedicated DTO classes. All request data is read directly from HttpServletRequest parameters and parts; all responses are assembled inline using org.json.JSONObject and JSONArray. There are no OpenAPI/Swagger, protobuf, or GraphQL specifications.

Request contracts (implicit):

  • Upload: customerId (form field, required integer), documentType (form field, optional string, defaults to "General"), file (multipart part, required, non-empty)
  • Customer list: {customerId} path segment (required integer)
  • Download: {id} path segment (required long integer)

Response shapes (inline JSON):

  • Upload success: {"status":"UPLOADED","documentId":N,"customerId":N,"documentType":"...","fileName":"..."}
  • Customer list success: {"customerId":N,"count":N,"documents":[{"documentId":N,"customerId":N,"documentType":"...","fileName":"...","contentType":"...","uploadedDate":"..."}]}
  • Error (any endpoint): {"status":"ERROR","message":"..."}

Serialization is handled directly by org.json with no Jackson or custom serializer configuration.

Communication Patterns

Synchronous only: All communication is direct HTTP request/response with no message queues, event streams, or async patterns.

No inter-service communication: ZavaDocVault is a standalone monolith with no calls to other services, no Feign clients, no RestTemplate/WebClient, and no service discovery.

No resilience patterns: No circuit breaker (Resilience4j, Hystrix), retry policy, or timeout configuration is present beyond JDBC default timeouts.

Security posture: No authentication or authorization is configured. All API endpoints (/api/documents/*, /health) are publicly accessible with no credential checks, no JWT/OAuth2, no Spring Security, and no TLS termination at the application level. Startup Dependency Chain

  1. SQL Server must be reachable on DB_HOST:DB_PORT before Tomcat starts — there is no wait mechanism in the Dockerfile or web.xml.
  2. DocVaultBootstrapServlet (load-on-startup=1) executes immediately when Tomcat loads the application context, connecting to SQL Server and running the CREATE TABLE IF NOT EXISTS DDL.
  3. If SQL Server is unavailable at startup, DocVaultBootstrapServlet.init() throws a ServletException, which causes Tomcat to mark the servlet context as failed and the application will not serve requests.
  4. The remaining servlets (Upload, CustomerDocuments, Download) are loaded on first request (no load-on-startup set) and have no explicit readiness wait.

> There is no dockerize, Kubernetes readiness probe, Docker Compose depends_on with health check, or retry logic. In a containerized environment, a startup ordering mechanism (e.g., initContainer, depends_on: condition: service_healthy) should be added.

Core Business Workflows

ZavaDocVault is a document management service for ZavaBank that allows banking staff or customers to upload, list, and download customer-associated documents (e.g., ID scans, bank statements, signed forms) stored in a central SQL Server repository.

Primary Workflows

Workflow 1: Document Upload

A caller (banking staff or authenticated client) submits a document file for a specific customer. The servlet validates the required fields (customerId, non-empty file), reads the file bytes into memory, and persists the full binary payload plus metadata to the Documents table. A generated DocumentID is returned to the caller.

Steps:

  1. Caller POSTs to /api/documents/upload with customerId, optional documentType (defaults to "General"), and a file multipart part.
  2. Servlet validates that customerId is a valid integer and that the file part is non-null and non-empty.
  3. File bytes are read entirely into memory (in-process buffer, max 10 MB per file).
  4. A single INSERT statement stores all fields plus binary data; GETDATE() sets UploadedDate.
  5. The generated DocumentID (SQL Server IDENTITY) is retrieved via RETURN_GENERATED_KEYS.
  6. Response JSON: {status:"UPLOADED", documentId, customerId, documentType, fileName}.

Workflow 2: List Customer Documents

A caller requests all documents belonging to a customer. The servlet looks up document metadata (no binary data) ordered by most recent upload.

Steps:

  1. Caller sends GET /api/documents/customer/{customerId}.
  2. Servlet validates that the path segment is a valid integer.
  3. SELECT on Documents filtered by CustomerID, ordered by UploadedDate DESC, DocumentID DESC.
  4. Response JSON: {customerId, count, documents:[{documentId, documentType, fileName, contentType, uploadedDate}]}.

Workflow 3: Document Download

A caller retrieves the binary content of a specific document by its ID.

Steps:

  1. Caller sends GET /api/documents/{id}.
  2. Servlet validates the path segment as a valid long integer.
  3. SELECT FileName, ContentType, DocumentData for the given DocumentID.
  4. If not found, returns 404.
  5. If found, sets Content-Disposition: attachment; filename="{FileName}" and streams the binary payload.

Workflow 4: Application Bootstrap (Schema Initialization)

On every Tomcat startup, DocVaultBootstrapServlet (load-on-startup=1) ensures the Documents table exists before any API request is served.

Steps:

  1. Tomcat loads DocVaultBootstrapServlet during context initialization.
  2. Connects to SQL Server using configured credentials.
  3. Executes IF OBJECT_ID('Documents', 'U') IS NULL CREATE TABLE Documents (...).
  4. If SQL Server is unreachable, throws ServletException and aborts startup.

Cross-Service Data Flows

ZavaDocVault is a standalone service with no outbound calls to other services. There is no gateway aggregation, no event-driven data flow, and no shared database with another service. The CustomerID field is a caller-supplied opaque integer; the application does not validate it against an external customer registry.

Business Rules & Decision Logic

Validation rules:

  • customerId must be a non-null, non-empty integer value; returns 400 if missing or non-numeric.
  • The uploaded file part must be non-null and have size > 0; returns 400 otherwise.
  • documentType defaults to "General" if not supplied or empty.
  • fileName is sanitized with safeFileName(): backslashes and forward slashes are replaced with underscores to prevent path-traversal in Content-Disposition headers; empty names fall back to "upload.bin".

Constraints:

  • Maximum file size: 10 MB (10485760 bytes) enforced by Tomcat @MultipartConfig.
  • Maximum total request size: 12 MB (12582912 bytes).

State transitions:

  • Documents have no lifecycle state — once inserted, they cannot be updated or deleted through the current API. There are no soft-delete, archive, or versioning states.

Error handling:

  • SQLException from any JDBC operation returns a 500 Internal Server Error with a generic JSON error message. The original exception is not propagated to the client.
  • Errors are not logged to any logging framework — exception details are silently swallowed.

Data integrity:

  • No referential integrity between CustomerID in Documents and any external customer table (no foreign key, no validation call).
  • Auto-commit mode is active on all connections; there is no explicit transaction boundary.

Authorization:

  • No role-based or attribute-based access controls exist. Any caller who can reach the network can upload, list, or download any document for any CustomerID.

Version & Compatibility Risks

The most significant version risk is Java 8 (sourceCompatibility = JavaVersion.VERSION_1_8): Java 8 is in long-term maintenance mode and is flagged by AppCAT as requiring an upgrade for all three Azure targets (AKS, App Service, Container Apps). The org.json 20140107 artifact is from 2014 — while functionally stable, it predates many security and API improvements and should be updated. The javax.servlet-api 3.1.0 artifact is the Servlet 3.1 specification, which is compatible with Tomcat 9 but is superseded by Jakarta EE's jakarta.servlet-api for Tomcat 10+. Any upgrade of the container to Tomcat 10 would require a namespace migration from javax. to jakarta..

Notable Observations

  • No ORM or connection pooling: All database access is via raw DriverManager.getConnection() with no connection pool (no HikariCP, c3p0, or DBCP). Under concurrent load, each request opens and closes a new physical connection, which is a significant performance risk.
  • org.json 20140107 is over 10 years old: The declared version predates JSON security patches and API improvements; upgrading to a recent org.json (e.g., 20240303) is low-risk and recommended.
  • No logging framework declared: The application has no SLF4J, Log4j, or Logback dependency; errors are silently swallowed with catch blocks that ignore exceptions, making production diagnosis difficult.
  • Minimal dependency footprint: The extremely lean dependency set simplifies modernization; migrating to Spring Boot or Quarkus would require adding framework BOM/starter dependencies but the existing logic is straightforward to port.

Caching Strategy

No caching layer is present. Each HTTP request opens a new JDBC connection, executes a SQL query, and closes the connection. There is no second-level cache, query result cache, or in-memory store (no EhCache, Redis, Caffeine, or Spring Cache annotations).

Data Ownership Boundaries

ZavaDocVault is a single-service monolith that is the sole owner of the Documents table. There is no shared database, no database-per-service concern, and no cross-service data access to document. All reads and writes to the Documents table originate from the four servlet classes within the same WAR deployment.

Read/write patterns: Simple request/response — upload writes one row (with binary blob), customer list reads metadata rows for a given CustomerID, download reads the full row (including blob) for a given DocumentID. No CQRS, event sourcing, or async write patterns are in use.
