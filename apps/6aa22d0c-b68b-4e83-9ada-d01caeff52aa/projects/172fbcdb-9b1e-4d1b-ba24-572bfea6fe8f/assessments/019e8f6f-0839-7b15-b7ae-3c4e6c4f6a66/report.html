Data Storage & External Services

The service relies on a SQL Server database (ZavaBankDB) as its primary data store, accessed via ADO.NET SqlClient with raw parameterized SQL. Four tables are used: Transactions, Accounts, AlertRules, AccountAlerts, and AlertHistory. The connection string is resolved dynamically from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) with a fallback to the web.config connection string named ZavaBankDb. For messaging, the service integrates with RabbitMQ by calling the RabbitMQ HTTP Management API directly (not using an AMQP client library), posting alert events as JSON to the zava.notifications exchange under the /zavabank virtual host.

Key Architectural Decisions

  • WCF BasicHttpBinding over SOAP: The service exposes operations via WCF using basicHttpBinding, making it accessible to SOAP clients; service metadata (WSDL) is enabled via httpGetEnabled=true.
  • RabbitMQ via HTTP Management API: Instead of using an AMQP client library (e.g., RabbitMQ.Client), alert publication is performed via HTTP calls to the RabbitMQ management REST endpoint, avoiding a native broker dependency but limiting throughput and error visibility.
  • Environment-variable-first configuration: Both database and broker settings are resolved from environment variables at runtime, falling back to web.config values, making the service suitable for containerized deployments.

Communication Patterns

Synchronous (inbound — SOAP/HTTP): Clients invoke service operations over HTTP using SOAP envelopes. No REST or gRPC interfaces are present. No API gateway layer sits in front of this service.

Synchronous (outbound — SQL Server): All database operations are performed synchronously via ADO.NET SqlConnection/SqlCommand. There is no connection pooling configuration beyond ADO.NET defaults, and no ORM or repository abstraction.

Asynchronous (outbound — RabbitMQ): When an alert rule is triggered, AlertService calls RabbitMqPublisher.PublishAlert, which performs a synchronous HTTP POST to the RabbitMQ HTTP Management API (/api/exchanges/{vhost}/zava.notifications/publish). Despite using a message broker, the publish call is blocking — the service waits for the HTTP response before proceeding. Failures are silently swallowed (the method returns false on any exception).

Resilience patterns: None. There is no retry logic, circuit breaker, timeout configuration, or bulkhead pattern for either the SQL Server or RabbitMQ integrations. The RabbitMQ publish call wraps exceptions in a bare catch { return false; } block.

Service discovery: None. The RabbitMQ host and SQL Server host are resolved from environment variables at runtime with hardcoded default values (rabbitmq/sqlserver) — no service discovery registry is used.

API gateway: None. The service is directly accessible.

Security posture: No authentication or authorization is configured. All four WCF service operations and the /health endpoint are publicly accessible with no credentials required. basicHttpBinding does not enforce transport security (TLS) by default; HTTPS would require IIS binding configuration outside this codebase. The web.config allows all users on the /health path explicitly (<allow users="*" />), and no equivalent restriction exists on the WCF endpoint. No JWT, OAuth2, Windows Authentication, or API key mechanism is implemented.

Startup dependency chain: The service has no startup readiness logic; it becomes available as soon as IIS/ASP.NET initializes the WCF service host. SQL Server and RabbitMQ availability are not checked at startup — failures surface only during the first operation call.

Core Business Workflows

ZavaAlertService is a banking alert engine that evaluates financial transactions against customer-configured alert rules and publishes triggered alerts to a notification queue for downstream delivery.

Domain Entities

EntityService / Bounded ContextDescriptionKey Relationships
TransactionExternal (Banking Core)A financial movement on a bank account — debit or credit with amount, balance impact, and originating channelBelongs to one Account; evaluated by alert rules
AccountExternal (Banking Core)A bank account owned by a customerHas many Transactions; has many AccountAlerts
AlertRuleAlert ManagementA global, reusable rule template defining an alert condition (e.g., "low balance", "large transaction") by rule code and default thresholdApplied to accounts via AccountAlerts
AccountAlertAlert ManagementAn account-level activation of a rule, with optional custom threshold and notification channel preferenceLinks one Account to one AlertRule; generates AlertHistory entries
AlertHistoryAlert ManagementAn immutable audit record of when a rule fired for an account, including the trigger value, notification status, and contextual detailsBelongs to one AccountAlert

Service-to-Domain Mapping

ServiceDomain ContextOwned EntitiesExternal Dependencies
ZavaAlertServiceAlert ManagementAlertRule, AccountAlert, AlertHistoryReads Transaction and Account from shared ZavaBankDB (owned by external Banking Core); writes alert events to RabbitMQ zava.notifications exchange

This is a single-service application. Transaction and Account data are accessed directly from a shared database rather than through a banking core API, creating a direct schema dependency on the external system.

Primary Workflows

Workflow 1: Transaction Alert Evaluation

The core workflow. A caller submits a transaction ID; the service loads the transaction and its account's configured alert rules, evaluates each rule, publishes events for any triggered rules, and records the evaluation outcome.

Steps:

  1. Receive evaluation request — A SOAP call with a transactionId is received on the WCF endpoint.
  2. Validate input — transactionId must be greater than 0; otherwise a fault is returned immediately.
  3. Load transaction context — The service queries Transactions joined with Accounts to retrieve accountId, amount, balanceAfter, customerId, and channel.
  4. Handle missing transaction — If no transaction is found for the given ID, a fault is returned.
  5. Load active alert rules — All enabled AccountAlerts with active AlertRules for the account are fetched.
  6. Evaluate each rule — For each rule, the appropriate condition is tested (see Business Rules below). If triggered, the rule code is added to the triggered list.
  7. Publish alert event — For each triggered rule, an AlertEvent is serialized to JSON and posted to the RabbitMQ zava.notifications exchange via the HTTP Management API.
  8. Record alert history — For each triggered rule, an AlertHistory row is written recording the trigger value, notification sent status, and a descriptive detail string.
  9. Return evaluation response — The response includes the list of triggered rule codes, whether any event was published to the queue, and a human-readable message.

Workflow 2: Alert Rule Management (Create)

An authorized caller creates a new global alert rule and simultaneously activates it for a specific customer account.

Steps:

  1. Receive create request — A SOAP call with an AlertRuleDefinition containing rule metadata and account/customer identifiers.
  2. Validate input — RuleName, RuleCode, AccountId, and CustomerId must be present and non-empty.
  3. Insert global rule — A new AlertRules row is inserted with the provided name, code, threshold, category, and active status. The generated RuleID is captured.
  4. Link to account — A new AccountAlerts row is inserted linking the new rule to the specified account, with the notification channel and threshold.
  5. Atomic transaction — Both inserts are wrapped in a single SQL transaction; if either fails, both are rolled back.
  6. Return mutation response — The response includes the new RuleID, AccountAlertID, and a success message.

Workflow 3: Alert Rule Management (Update)

An authorized caller updates an existing global rule definition and optionally updates the account-level configuration for that rule.

Steps:

  1. Receive update request — A SOAP call with a ruleId and an AlertRuleDefinition.
  2. Validate input — ruleId must be greater than 0; RuleName and RuleCode are required.
  3. Update global rule — The AlertRules row is updated in place. If no row is affected (rule does not exist), a fault is returned.
  4. Upsert account-level config (if AccountId > 0) — If an AccountAlerts row exists for this rule and account, it is updated; otherwise a new row is inserted.
  5. Atomic transaction — All mutations are wrapped in a single SQL transaction.
  6. Return mutation response — The response includes the RuleID, affected AccountAlertID, and a success message.

Workflow 4: Retrieve Alert Rules for Account

A read-only query returning all alert rules (enabled and disabled) configured for a specific account.

Steps:

  1. Receive query — A SOAP call with an accountId.
  2. Validate input — accountId must be greater than 0.
  3. Query rules — AccountAlerts joined with AlertRules is queried for the account, ordered by rule name.
  4. Return rules response — An AlertRulesResponse with the full list of AlertRuleInfo items is returned.

Cross-Service Data Flows

ZavaAlertService is a single-service application with no service-to-service API calls. Cross-boundary data flows occur at the infrastructure level:

  • Inbound data (Banking Core → ZavaAlertService): Transaction and Account data are read directly from the shared ZavaBankDB SQL Server database. There is no API abstraction — the alert service queries tables owned by the banking core directly. This means there is no circuit breaker or fallback if the banking core's schema changes or data is unavailable.
  • Outbound events (ZavaAlertService → RabbitMQ): When a rule fires, an AlertEvent JSON payload is published to the zava.notifications exchange. The downstream consumer (a notification delivery service, not part of this repository) is responsible for routing and delivering the notification to the customer. If RabbitMQ is unavailable, the publish silently fails and NotificationSent = false is recorded in AlertHistory — the alert evaluation still succeeds and history is still written.

Business Rules & Decision Logic

Alert Rule Conditions

The service evaluates five named rule codes:

Rule CodeTrigger ConditionThreshold Meaning
LOW_BALbalanceAfter < thresholdMinimum balance before alert fires
LARGE_TXNabs(amount) >= thresholdMinimum transaction size to trigger alert
OVERDRAFTbalanceAfter < 0No threshold — fires on any negative balance
INTL_WIREChannel contains "WIRE" AND abs(amount) >= thresholdMinimum wire transfer size; channel match is case-insensitive
DAILY_SPENDsum of today's debits >= thresholdCumulative daily spend limit; queries all same-day debit transactions

Any unrecognized rule code is treated as never triggered (default case returns false).

Threshold Resolution

For each AccountAlert, the effective threshold is: the account-level Threshold if non-empty, otherwise the AlertRule.DefaultThreshold. Threshold values are stored as strings and parsed as decimals using InvariantCulture; parse failures default to 0.0.

Input Validation Rules

  • transactionId must be > 0 (fault returned on violation)
  • accountId must be > 0 for GetAlertRules (fault returned on violation)
  • ruleId must be > 0 for UpdateAlertRule (fault returned on violation)
  • RuleName and RuleCode are required for both CreateAlertRule and UpdateAlertRule (fault returned if blank)
  • AccountId and CustomerId are required for CreateAlertRule (fault returned if ≤ 0)
  • RuleCode is normalized to uppercase trimmed form before persistence

Notification Channel

Each AccountAlert stores a preferred notification channel (e.g., Email, SMS). The channel is recorded in AlertHistory.Details and included in the published AlertEvent for the downstream notification service to use. Defaults to "Email" if not specified.

Transaction Boundaries

  • CreateAlertRule and UpdateAlertRule each wrap their DB writes in an explicit SqlTransaction. If either INSERT/UPDATE fails, both are rolled back, ensuring AlertRules and AccountAlerts remain consistent.
  • EvaluateTransaction does not use a transaction — each AlertHistory INSERT is committed independently. A failure inserting one history record does not prevent evaluation of subsequent rules.

Error Handling

  • All business validation failures throw FaultException (WCF fault), which is serialized as a SOAP fault to the caller.
  • RabbitMQ publish failures are silently caught; the evaluation continues and NotificationSent = false is stored in history.
  • There is no retry logic, dead-letter queue, or compensating action for failed publish attempts.
  • includeExceptionDetailInFaults=true in web.config means internal exception details are exposed to SOAP callers — this is a security concern in production.

Authorization

No authentication or authorization is implemented. All operations are accessible to any caller without credentials.

Version & Compatibility Risks

All referenced assemblies ship with .NET Framework 4.8, which is a Windows-only runtime currently in maintenance mode (security fixes only). The most critical risk is System.ServiceModel (WCF) — the WCF server stack is not available in .NET Core / .NET 5+; migrating to .NET 10 would require replacing the WCF service host with ASP.NET Core (e.g., minimal APIs, gRPC, or CoreWCF). System.Data.SqlClient (the SqlConnection namespace used) is also the legacy driver; the recommended replacement for modern .NET is Microsoft.Data.SqlClient. System.Web.Extensions (used for JavaScriptSerializer) is unavailable on .NET Core — System.Text.Json or Newtonsoft.Json are the modern alternatives. There are no third-party NuGet packages with independent CVE exposure, but the entire .NET Framework 4.8 runtime surface inherits Microsoft's maintenance-mode support timeline.

Notable Observations

  • No NuGet packages at all: The packages.config is empty and no <PackageReference> items exist in the .csproj. All functionality is delivered entirely through .NET Framework built-in assemblies, making the dependency footprint extremely small but tightly coupled to .NET Framework.
  • RabbitMQ without a client library: Instead of the standard RabbitMQ.Client NuGet package, messaging is implemented via raw HttpWebRequest calls to the RabbitMQ HTTP Management API. This is unconventional, adds network overhead per message, and makes migration to a managed AMQP client necessary when modernizing.
  • Legacy System.Data.SqlClient: The service uses System.Data.SqlClient.SqlConnection (the .NET Framework namespace), which is frozen for new features. On .NET 5+, Microsoft.Data.SqlClient is the supported package for SQL Server access.
  • Old-style project format: The .csproj uses the MSBuild 4.0 / non-SDK format. A migration to the modern SDK-style .csproj is required before targeting .NET 5+ or .NET 10.

Caching Strategy

No caching layer is present. Every service operation opens a new SqlConnection and executes SQL directly against the database. There is no in-memory cache, distributed cache (Redis, Memcached), or query result caching. ADO.NET connection pooling is active by default but is not explicitly configured in the connection string.

Data Ownership Boundaries

Shared vs isolated data store: ZavaAlertService uses a single shared SQL Server database (ZavaBankDB). The Transactions and Accounts tables are owned and written by an external banking core system; ZavaAlertService reads them as reference data. The AlertRules, AccountAlerts, and AlertHistory tables are managed exclusively by this service.

Cross-service data access: ZavaAlertService reads Transactions and Accounts directly via SQL — there is no API abstraction over these tables. This creates a tight coupling between this service and the banking data schema. Any schema change to Transactions or Accounts in the shared database would require corresponding updates to the inline SQL in this service.

Read/write patterns: The service is primarily read-heavy during EvaluateTransaction (multiple SELECT queries per call) with targeted writes to AlertHistory for each triggered rule. Rule management operations (CreateAlertRule, UpdateAlertRule) use explicit SQL transactions to ensure atomic writes across AlertRules and AccountAlerts.
