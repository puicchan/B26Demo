Data Storage & External Services

The service uses SQL Server (ZavaBankDB) as its primary data store, accessed via raw ADO.NET without an ORM. Key tables include Transactions, Accounts, AlertRules, AccountAlerts, and AlertHistory. Connection details are resolved from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) with a fallback to web.config connection strings.

RabbitMQ is used as the event messaging system. Alert events are published to the zava.notifications exchange in the /zavabank virtual host via the RabbitMQ HTTP Management API (port 15672). Connection credentials are configured through environment variables (RABBITMQ_HOST, RABBITMQ_USER, RABBITMQ_PASSWORD, RABBITMQ_VHOST).

Key Architectural Decisions

  • WCF over BasicHttpBinding: The service exposes a SOAP endpoint using BasicHttpBinding, a legacy protocol choice well-suited for enterprise banking integrations but not portable to .NET Core/modern .NET without migration.
  • ADO.NET without ORM: All database interactions use raw SqlCommand/SqlDataReader with inline SQL — no Entity Framework or Dapper — giving fine-grained control but increasing migration effort.
  • RabbitMQ via HTTP Management API: Instead of the AMQP client library, the publisher uses HTTP POST to the RabbitMQ Management REST API, avoiding a NuGet dependency but coupling to the management port configuration.

Communication Patterns

Synchronous (inbound): Clients communicate with the service via SOAP over HTTP using basicHttpBinding. There is no HTTPS/TLS, no authentication, and no authorization configured — all four WCF operations and the health endpoint are publicly accessible with no credential or identity checks.

Synchronous (outbound — database): The service uses ADO.NET SqlConnection / SqlCommand opened per-request (no connection pooling configuration beyond ADO.NET defaults). Every WCF operation opens one or more SQL connections inline, with no retry policy and no circuit breaker protecting the database calls. A transient SQL failure will propagate as a FaultException to the caller.

Asynchronous (outbound — messaging): When an alert rule is triggered during EvaluateTransaction, an AlertEvent is published to RabbitMQ via an HTTP POST to the RabbitMQ Management REST API (/api/exchanges/{vhost}/zava.notifications/publish). The call uses HttpWebRequest with basic authentication. This is a best-effort fire-and-forget: publish failures are caught silently (the outer catch block returns false) and do not roll back the AlertHistory INSERT or fail the WCF operation. There is no retry mechanism, no dead-letter queue, and no acknowledgement.

Resilience: No circuit breaker, retry policy, bulkhead, or timeout is configured on any outbound call (SQL or RabbitMQ). The HttpWebRequest to RabbitMQ has no explicit timeout set, relying on the .NET default (100 seconds).

Service discovery: Both SQL Server and RabbitMQ endpoints are resolved from environment variables at request time (no service registry). If environment variables are absent, the service falls back to web.config values for SQL Server. The RabbitMQ host defaults to the hostname rabbitmq.

Security posture: No TLS/HTTPS, no authentication, and no authorization are configured. The WCF endpoint uses basicHttpBinding over plain HTTP. All operations are callable by any network client that can reach port 8080. Credentials for SQL Server and RabbitMQ are stored in environment variables, with hard-coded defaults visible in source code and web.config.

Core Business Workflows

ZavaAlertService is a banking alert engine that evaluates financial transactions against a set of per-account configurable alert rules and publishes triggered alerts to a downstream notifications queue for customer communication.

Service-to-Domain Mapping

This is a single-service application. The service operates inside a shared database owned by the broader banking platform — it does not own the Transaction or Account tables but reads them directly via SQL.

Primary Workflows

Workflow 1: Transaction Alert Evaluation

This is the core business workflow. When a banking transaction occurs, an upstream system calls EvaluateTransaction with the transaction ID. The service loads the transaction and account details, retrieves all active alert rules for the account, evaluates each rule, publishes events for triggered rules, records history, and returns a summary.

Steps:

  1. Validate that transactionId > 0 (SOAP FaultException thrown otherwise).
  2. Load the transaction record — fetch amount, balance-after, account ID, customer ID, and channel from the shared database.
  3. Load all active and enabled alert rules configured for the account.
  4. For each rule, evaluate whether the threshold condition is met (see Business Rules section).
  5. For each triggered rule: publish an AlertEvent to RabbitMQ and insert an AlertHistory record.
  6. Return AlertEvaluationResponse listing all triggered rule codes and whether notifications were dispatched.

Business outcome: Customers with configured alert thresholds are notified in near-real-time when their account activity matches a rule (e.g., balance drops below a floor, a large debit occurs, an overdraft happens).

Workflow 2: Alert Rule Management

This workflow allows clients (bank staff or a self-service portal) to create and maintain the alert rules attached to customer accounts.

Create Alert Rule (CreateAlertRule):

  1. Validate that RuleName, RuleCode, AccountId, and CustomerId are all provided.
  2. Insert a new AlertRule record with the supplied name, code, threshold, category, and activation status.
  3. Insert a new AccountAlert binding linking the rule to the specified customer account with a notification channel preference.
  4. Both inserts are wrapped in a single database transaction — either both succeed or both roll back.
  5. Return the generated RuleId and AccountAlertId.

Update Alert Rule (UpdateAlertRule):

  1. Validate that ruleId > 0, RuleName, and RuleCode are present.
  2. Update the AlertRule record.
  3. If AccountId > 0 is provided: upsert the AccountAlert binding (update if exists; insert if not).
  4. Both operations are wrapped in a single database transaction.
  5. Return updated IDs and success status.

Get Alert Rules (GetAlertRules):

  1. Validate that accountId > 0.
  2. Return all AlertRule and AccountAlert records for the account (enabled and disabled).

Workflow 3: Daily Spend Calculation (Sub-workflow)

When the DAILY_SPEND alert rule is evaluated during Workflow 1, the service executes a separate SQL aggregate query to sum all debit amounts on the account for the current calendar day. This sub-workflow is embedded inline in the evaluation loop and is not exposed as a standalone operation.

Cross-Service Data Flows

ZavaAlertService is a single service with no outbound REST or gRPC calls to peer services. Cross-service data flow occurs in two directions:

Inbound (shared database reads): The service reads Transactions and Accounts tables that are owned by the Core Banking platform. There is no API abstraction — data is fetched via direct SQL joins. If the Core Banking team changes the schema of these tables, the alert service breaks silently at runtime.

Outbound (message publishing): When alert rules are triggered, AlertEvent payloads are published to the RabbitMQ zava.notifications exchange (virtual host /zavabank) via the HTTP Management API. Downstream consumers (e.g., an email/SMS notification service) are expected to subscribe to this exchange. If the publish fails, the failure is silently swallowed and the AlertHistory row is recorded with NotificationSent = false. There is no retry, no dead-letter queue, and no compensating action — triggered alerts that fail to publish are permanently lost from the messaging perspective.

Fallback behavior: If RabbitMQ is unavailable, the evaluation workflow completes successfully from the WCF caller's perspective. The AlertEvaluationResponse.PublishedToNotificationsQueue flag will be false, indicating that no notifications were dispatched, but no error is surfaced to the calling system.

Business Rules & Decision Logic

Alert Trigger Conditions

Five named rule codes are supported. Each code maps to a specific threshold comparison:

Rule CodeTrigger ConditionThreshold Meaning
LOW_BALbalanceAfter < thresholdMinimum acceptable balance (e.g., 500.00)
LARGE_TXNABS(amount) >= thresholdMaximum single transaction amount (e.g., 10000.00)
OVERDRAFTbalanceAfter < 0No threshold needed — triggers on any negative balance
INTL_WIREChannel contains "WIRE" AND ABS(amount) >= thresholdMinimum wire transfer amount to alert on
DAILY_SPENDToday's total debit spend >= thresholdMaximum daily spending limit
(unknown code)Always false — silently ignoredN/A

Threshold values are stored as strings. If a threshold cannot be parsed as a decimal, it defaults to 0.

Validation Rules

  • transactionId must be > 0; otherwise a FaultException is thrown before any database call.
  • accountId must be > 0 for GetAlertRules and UpdateAlertRule.
  • ruleId must be > 0 for UpdateAlertRule.
  • For CreateAlertRule: rule must not be null; RuleName and RuleCode must not be empty; AccountId and CustomerId must be > 0.
  • For UpdateAlertRule: rule must not be null; RuleName and RuleCode must not be empty. AccountId is optional (account binding only upserted if AccountId > 0).

Transaction Boundaries

  • CreateAlertRule and UpdateAlertRule use a SqlTransaction to atomically write the AlertRules + AccountAlerts pair. A failure in either insert rolls back both.
  • EvaluateTransaction does not use a database transaction. The RabbitMQ publish and the AlertHistory INSERT are not atomic — a partially completed evaluation loop (e.g., process crash between publish and INSERT) leaves the state inconsistent.

Error Handling

  • All validation failures surface as WCF FaultException with a human-readable message.
  • RabbitMQ publish failures are caught and return false — they do not abort the evaluation or surface to the caller.
  • SQL errors are unhandled at the service level and propagate as uncaught exceptions, which WCF converts to SOAP Fault responses (with full stack traces visible due to includeExceptionDetailInFaults = true in web.config).
  • If a transaction ID is not found in the database, a FaultException("Transaction was not found.") is thrown.
  • If an UpdateAlertRule rule ID is not found in the database, a FaultException("Alert rule was not found.") is thrown.

Audit Trail

AlertHistory provides a lightweight append-only audit log. Each record captures: which account-alert rule fired, the trigger value, whether the customer was notified, the notification timestamp (or null), and a plaintext Details string embedding the rule code, account ID, balance-after, and channel. There is no separate event sourcing or change-data-capture mechanism.

Authorization

No authorization is implemented. Any caller that can reach the WCF endpoint on port 8080 may invoke all four operations, including creating and modifying alert rules.

Version & Compatibility Risks

.NET Framework 4.8 is the final version of the .NET Framework line; Microsoft maintains it in security-fix-only mode with no new feature development. Migrating to .NET 8+ (LTS) or .NET 10 would require addressing all framework-specific dependencies below:

  • System.ServiceModel (WCF): WCF is not included in .NET Core / .NET 5+. The CoreWCF open-source project provides a partial server-side port, but migration requires refactoring service hosting and configuration.
  • System.Web / System.Web.Extensions: The System.Web assembly does not exist in .NET Core. System.Web.Extensions.JavaScriptSerializer must be replaced with System.Text.Json or Newtonsoft.Json. The ASHX health handler must be replaced with a minimal API endpoint or middleware.
  • System.Configuration: ConfigurationManager is available via the System.Configuration.ConfigurationManager NuGet package for compatibility, but the modern pattern uses Microsoft.Extensions.Configuration.
  • System.Data.SqlClient: The System.Data.SqlClient namespace is superseded by Microsoft.Data.SqlClient (NuGet package), which receives active support and security patches.
  • Mono 6.12: Mono is effectively end-of-life for new projects. The recommended migration path is to the official .NET runtime.

Notable Observations

  • No third-party NuGet packages: The project has zero NuGet dependencies, making it unusually self-contained. This simplifies the NuGet package migration surface but means all external integrations (RabbitMQ, SQL Server) are implemented from scratch using raw framework APIs.
  • RabbitMQ without AMQP client: Instead of the standard RabbitMQ.Client NuGet package, the service communicates with RabbitMQ through the HTTP Management REST API using HttpWebRequest. This is non-standard, slower than AMQP, and requires the management plugin to be enabled on the broker.
  • Legacy serialization: JavaScriptSerializer (from System.Web.Extensions) is a legacy API that was superseded by System.Text.Json and Newtonsoft.Json. It lacks support for modern JSON features and is not available outside the .NET Framework.
  • WCF as the only API surface: The entire service API is exposed via SOAP/WCF with no REST/HTTP endpoints (except the trivial health check). Modernization would benefit from adding an HTTP/REST surface via ASP.NET Core minimal APIs or Web API.

Caching Strategy

No caching layer is present. Every request to EvaluateTransaction or GetAlertRules issues live SQL queries against SQL Server with no in-memory, distributed, or query-result cache. Alert rule definitions are reloaded on every transaction evaluation, which will create contention at scale. Adding a short-TTL cache (e.g., 30–60 seconds) for the rule set per account would reduce database load significantly.

Data Ownership Boundaries

ZavaAlertService operates on a shared database model. It does not own the Transactions and Accounts tables — it reads from them but never writes to them. These tables are presumably owned by an upstream banking core service. There is no API abstraction between services; ZavaAlertService accesses all data via direct SQL on the shared ZavaBankDB database.

Cross-service data access: The Transactions and Accounts tables are accessed by direct SQL query. There is no REST/gRPC call to a separate service to retrieve transaction or account data. This tight coupling to the shared database is the primary data boundary risk for a microservices migration.

Read/write pattern: Transactions and Accounts are read-only for this service. AlertRules and AccountAlerts are read-write (full CRUD). AlertHistory is append-only (insert, never updated or deleted by this service).
