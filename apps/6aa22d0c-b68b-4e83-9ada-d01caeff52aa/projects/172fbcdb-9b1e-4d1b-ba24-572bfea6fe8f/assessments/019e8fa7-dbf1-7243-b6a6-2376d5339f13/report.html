Data Storage & External Services

The service relies on SQL Server (configured as ZavaBankDB at sqlserver:1433) for all persistent data — transactions, accounts, alert rule definitions, account-level alert subscriptions, and alert history. Connection details are resolved from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) with a fallback to web.config connection strings.

RabbitMQ is used as the outbound messaging layer. When an alert rule is triggered, the service calls the RabbitMQ Management HTTP API to publish a JSON-encoded AlertEvent message to the zava.notifications exchange. RabbitMQ connection settings (RABBITMQ_HOST, RABBITMQ_MANAGEMENT_PORT, RABBITMQ_USER, RABBITMQ_PASSWORD, RABBITMQ_VHOST) are all resolved from environment variables.

Key Architectural Decisions

  • WCF with BasicHttpBinding: The service exposes a single SOAP endpoint using basicHttpBinding, making it interoperable with any SOAP-capable client while retaining strong typed contracts via [ServiceContract] / [OperationContract].
  • Direct ADO.NET without ORM: All database operations use raw SqlCommand with parameterized queries — no ORM is employed, keeping the data access layer explicit and close to the SQL.
  • RabbitMQ via Management HTTP API: Instead of using an AMQP client library, the service publishes messages through the RabbitMQ Management REST API, avoiding a native client dependency at the cost of higher-latency messaging.

Communication Patterns

Synchronous (inbound): Clients call the service over SOAP/HTTP using WCF basicHttpBinding. No TLS is configured — all traffic is plain HTTP. There is no API gateway; clients connect directly to port 8080.

Asynchronous (outbound): When a rule is triggered, AlertService calls RabbitMqPublisher.PublishAlert, which makes a synchronous HTTP POST to the RabbitMQ Management REST API (/api/exchanges/{vhost}/zava.notifications/publish). This is fire-and-forget: the method returns a boolean indicating whether the message was routed, and failures are silent (bare catch {} that returns false).

Resilience: There are no circuit-breaker, retry, or timeout policies. The HttpWebRequest to RabbitMQ uses the .NET default timeout. No fallback behavior is defined beyond returning PublishedToNotificationsQueue = false in the response.

Service discovery: No service discovery mechanism. RabbitMQ and SQL Server are resolved from environment variables (RABBITMQ_HOST, DB_HOST, etc.) with hardcoded defaults (rabbitmq, sqlserver).

Security posture: No authentication or authorization is configured for any endpoint. All WCF operations and the health endpoint are publicly accessible with no token, basic auth, or role checks. Transport security (TLS) is not configured. serviceDebug includeExceptionDetailInFaults="true" is enabled, which leaks internal error details to callers — this is a security concern in production.

Core Business Workflows

ZavaAlertService is the alert evaluation engine for Zava Bank, responsible for detecting when a banking transaction breaches a customer's subscribed alert rules and publishing triggered alert events to a downstream notifications channel.

Domain Entities

EntityService / Bounded ContextDescriptionKey Relationships
TransactionCore Banking (external — read-only)A debit or credit event on an account, with amount, resulting balance, and channelBelongs to an Account; triggers alert evaluation
AccountCore Banking (external — read-only)A bank account linked to a customerHas many Transactions; subscribed to many AlertRules via AccountAlerts
AlertRuleAlert ManagementA named, reusable rule definition (e.g., "LOW_BAL", "LARGE_TXN") with a default threshold and categoryCan be subscribed to by many accounts via AccountAlerts

Primary Workflows

Workflow 1: Transaction Alert Evaluation

Entry point: IAlertService.EvaluateTransaction(transactionId) — called by a client (e.g., the transaction-processing service) after a transaction is posted.

Steps:

  1. Validate input: transactionId must be > 0; otherwise a FaultException is returned immediately.
  2. Load transaction context: Fetch the transaction's Amount, BalanceAfter, Channel, and the owning AccountID and CustomerID from the database.
  3. Load active rules: Retrieve all AccountAlerts where IsEnabled = 1 and the linked AlertRule.IsActive = 1 for the account.
  4. Evaluate each rule: For each rule, run the rule-code dispatch logic (IsTriggered) to determine whether the transaction breaches the threshold.
  5. Publish alert event (if triggered): Serialize an AlertEvent to JSON and POST it to the RabbitMQ Management API. Record whether routing succeeded.
  6. Record alert history: Insert an AlertHistory row for each triggered rule, logging the trigger value, details, and notification outcome.
  7. Return evaluation response: Return AlertEvaluationResponse with the list of triggered rule codes, whether any events were published, and a summary message.

Workflow 2: Alert Rule Management

Entry points:

  • GetAlertRules(accountId) — retrieve all configured rules for an account
  • CreateAlertRule(rule) — create a new rule definition and link it to an account
  • UpdateAlertRule(ruleId, rule) — update an existing rule and upsert its account-level configuration

Create flow:

  1. Validate that RuleName, RuleCode, AccountId, and CustomerId are all provided.
  2. Insert a new AlertRules row with the given name, code, threshold, category, and active status.
  3. Insert a corresponding AccountAlerts row linking the rule to the specified account, with the customer's notification channel preference.
  4. Both inserts run within a single SQL transaction — either both succeed or neither is committed.
  5. Return the new RuleId and AccountAlertId.

Update flow:

  1. Validate ruleId > 0 and that RuleName/RuleCode are provided.
  2. Update the AlertRules row. If no row is affected, raise a FaultException("Alert rule was not found.").
  3. If AccountId > 0 is provided, upsert the AccountAlerts entry (update if exists, insert if not).
  4. Both operations run within a single SQL transaction.

Cross-Service Data Flows

ZavaAlertService is a consumer of data from the Core Banking domain. The Transactions and Accounts tables are owned externally and ZavaAlertService reads them but never writes to them. There is no API call between services — both live in the same shared ZavaBankDB SQL Server database, so the coupling is at the database level rather than through an API contract.

The outbound data flow is via RabbitMQ: when an alert is triggered, AlertEvent is published to the zava.notifications exchange on the /zavabank virtual host. The downstream consumer of this exchange (e.g., an email/SMS notification service) is not part of this service. If RabbitMQ is unavailable, the failure is silent — the alert is still recorded in AlertHistory with NotificationSent = 0, but no retry or dead-letter mechanism is in place.

Business Rules & Decision Logic

Alert Rule Evaluation Logic

The IsTriggered method dispatches on the rule code to apply the appropriate threshold comparison:

Rule CodeTrigger ConditionThreshold Meaning
LOW_BALbalanceAfter < thresholdMinimum acceptable balance (e.g., 500.00)
LARGE_TXNabs(amount) >= thresholdTransaction amount that qualifies as "large"
OVERDRAFTbalanceAfter < 0Threshold is irrelevant — balance going negative always triggers
INTL_WIREChannel contains "WIRE" AND abs(amount) >= thresholdMinimum wire transfer amount to alert on
DAILY_SPENDSum of debit amounts for the account on today's date >= thresholdDaily spending limit
_(unknown code)_Never triggersUnrecognized rule codes are silently skipped

Threshold Handling

  • Thresholds are stored as strings and parsed to decimal at evaluation time.
  • If a Threshold string cannot be parsed, it defaults to 0m.
  • Per-account Threshold on AccountAlerts overrides the AlertRule.DefaultThreshold if non-empty.
  • The SQL expression ISNULL(NULLIF(aa.Threshold, ''), ar.DefaultThreshold) implements this override at the database layer.
  • RuleCode values are normalized to uppercase before comparison.

Transactions & Data Integrity

  • CreateAlertRule and UpdateAlertRule use explicit SqlTransaction to ensure atomicity across the AlertRules and AccountAlerts tables.
  • EvaluateTransaction does NOT wrap the entire operation in a single transaction — it is possible for AlertHistory rows to be partially inserted if a failure occurs mid-loop.

Error Handling & Observability

  • All FaultException instances are surfaced to SOAP clients with the exception detail (because includeExceptionDetailInFaults="true" is set in web.config).
  • RabbitMqPublisher.PublishAlert swallows all exceptions silently and returns false. No logging, no retry, no dead-letter queue.
  • No structured logging framework is present — no audit trail is written beyond the AlertHistory SQL inserts.

Authorization

No authentication or authorization is enforced. Any caller with network access to port 8080 can invoke any service operation, including creating and modifying alert rules.

Version & Compatibility Risks

All declared references are built-in .NET Framework 4.8 assemblies — there are no third-party NuGet packages. The critical risk is the framework itself: .NET Framework 4.8 is in long-term maintenance mode and will not receive new features. WCF (System.ServiceModel) in its server-hosting form does not exist in modern .NET (net6.0+); migrating would require replacing the WCF service with a gRPC or REST/HTTP endpoint using CoreWCF (community port) or ASP.NET Core. System.Data.SqlClient has a migration path to Microsoft.Data.SqlClient on modern .NET. System.Web has no equivalent on modern .NET — web-hosted components must be rewritten using ASP.NET Core middleware.

Notable Observations

  • Zero third-party NuGet packages: The entire dependency surface is the .NET Framework BCL, which simplifies a modernization audit (no package CVEs or version conflicts to resolve) but means all modernization effort falls on framework-level API replacements.
  • RabbitMQ via HTTP API instead of AMQP client: The project avoids adding the RabbitMQ.Client NuGet package by using the RabbitMQ Management HTTP REST API directly through HttpWebRequest. This is unusual and introduces a dependency on the management plugin being enabled in RabbitMQ at runtime.
  • System.Web.Script.Serialization (JavaScriptSerializer) is obsolete: JavaScriptSerializer was superseded by Newtonsoft.Json and later System.Text.Json. Migrating to modern .NET would require replacing it.
  • No logging framework: There is no structured logging library (e.g., Serilog, NLog, Microsoft.Extensions.Logging). Errors are silently swallowed in RabbitMqPublisher.PublishAlert via a bare catch {} block, making production observability a concern.

Caching Strategy

No caching layer is present. Every request performs synchronous SQL queries against SQL Server without any in-memory or distributed cache. Rule sets are reloaded from the database on every EvaluateTransaction call, including the daily spend aggregation query which scans the Transactions table. This is a potential performance concern for high-volume evaluation scenarios.

Data Ownership Boundaries

ZavaAlertService operates against a shared SQL Server database (ZavaBankDB) that is not exclusively owned by this service. The Transactions and Accounts tables are read-only from this service's perspective — they are expected to be written by an upstream transaction-processing or core banking service. The AlertRules, AccountAlerts, and AlertHistory tables represent the alert domain and are owned exclusively by ZavaAlertService.

There is no database-per-service isolation. All tables reside in the same ZavaBankDB database, which means schema changes to shared tables (Transactions, Accounts) from another service could silently break ZavaAlertService query assumptions (column names, data types). No cross-service API calls are made for data access — the service reads directly from the shared database.
