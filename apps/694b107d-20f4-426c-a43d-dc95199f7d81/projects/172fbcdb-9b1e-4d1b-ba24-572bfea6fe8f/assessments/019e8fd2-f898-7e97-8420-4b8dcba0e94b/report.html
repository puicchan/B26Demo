Data Storage & External Services

The application uses a single SQL Server database (ZavaBankDB) accessed directly via ADO.NET SqlConnection / SqlCommand. It stores Customers, Accounts, AccountTypes, and Transactions. For balance and transaction details, the application calls an external Zava Ledger microservice (http://zava-ledger:8080) via HttpWebRequest, parsing the XML response. Authentication is delegated to an external ZavaAuthGateway, which hosts the actual login and logout pages; the application's Login.aspx acts as a redirect entry point that forwards users to the gateway with a ReturnUrl.

Key Architectural Decisions

  • Inline ADO.NET data access: All SQL is embedded directly in code-behind files using SqlConnection/SqlCommand with no abstraction layer or ORM.
  • Delegated authentication: The application relies on Forms Authentication cookies; actual credential verification is handled by an external ZavaAuthGateway service.
  • Fallback pattern for ledger data: Default.aspx attempts to fetch balance/transaction data from the Ledger REST service and falls back silently to the local SQL database if the external call fails.

DTOs & Contracts

ZavaAccountManager defines no formal DTO or contract classes. Data exchange patterns:

  • UI layer: Data flows between the database and GridView/FormView controls via System.Data.DataTable objects — there are no typed entity or DTO classes.
  • Ledger service responses: XML is parsed inline using XmlDocument.SelectSingleNode with XPath expressions. No typed deserialization classes exist; field values are extracted directly as strings.
  • No OpenAPI/Swagger specification: There is no OpenAPI, Swagger, protobuf, or GraphQL schema for this application.
  • Serialization: No JSON serializer is configured. The only serialization in use is ASP.NET Web Forms ViewState (binary, built-in) for page state, and raw XML string parsing for Ledger responses.

Communication Patterns

Synchronous — consumed services:

  • ZavaAuthGateway: Communication is via HTTP redirect (Response.Redirect). The application forwards the user's browser to the configured AuthGatewayLoginUrl / AuthGatewayLogoutUrl with a ReturnUrl query parameter. There is no direct server-to-server call; the gateway is responsible for returning a Forms Authentication ticket.
  • Zava Ledger Service: Called synchronously via System.Net.HttpWebRequest (blocking I/O) with no timeout configured. The URL is constructed from LedgerBaseUrl in appSettings. Responses are XML documents parsed in-process.

Resilience patterns:

  • No circuit breaker or retry policy: If the Ledger service is unavailable, a catch block silently swallows the exception and falls back to querying the local Transactions SQL table. This is the only resilience mechanism present. Balance requests (GetBalance) have no fallback — they return the string "Ledger balance unavailable." on error.
  • No timeout configured on HttpWebRequest calls; default system timeout applies (approximately 100 seconds).

Asynchronous messaging: None. No message broker, event bus, or background worker is used.

Service discovery: Services are located via hardcoded appSettings keys (LedgerBaseUrl, AuthGatewayLoginUrl, AuthGatewayLogoutUrl). No dynamic service registry (Consul, Eureka, Kubernetes DNS) is in use.

Security posture: The application uses ASP.NET Forms Authentication with a cookie named .ZAVAAUTH (30-minute sliding expiration, Protection=All). The machineKey for ticket encryption is hardcoded in web.config with static validationKey and decryptionKey values — this is a security risk. All pages except Login.aspx and Logout.aspx require authentication. Transport-level TLS (HTTPS) is not enforced in configuration. Calls to the Ledger service are made over plain HTTP with no authentication header.

Core Business Workflows

ZavaAccountManager is a banking account management portal that allows authenticated bank staff to manage customer accounts — opening and closing accounts, retrieving balances, and reviewing transaction history — backed by a SQL Server database and an external Ledger microservice.

Primary Workflows

Workflow 1: Staff Authentication

A bank staff member navigates to the portal and is redirected to the ZavaAuthGateway for credential verification. On successful authentication, the gateway issues a Forms Authentication cookie (.ZAVAAUTH) and redirects the user back to the portal's Default.aspx.

Steps:

  1. User requests any protected page (e.g., Default.aspx).
  2. Forms Authentication middleware detects no valid .ZAVAAUTH cookie and redirects to Login.aspx?ReturnUrl=....
  3. Login.aspx constructs the gateway URL and redirects the browser to AuthGatewayLoginUrl?ReturnUrl=....
  4. ZavaAuthGateway verifies credentials, issues .ZAVAAUTH cookie, and redirects to ReturnUrl.
  5. Portal validates the cookie and serves Default.aspx.

Workflow 2: Customer and Account Viewing

On page load, Default.aspx loads the top 20 customers. Staff selects a customer from the grid, which triggers a postback that loads all accounts for that customer.

Steps:

  1. Page loads → BindCustomers() fetches top 20 active customers from Customers table.
  2. Staff clicks a customer row → gvCustomers_SelectedIndexChanged captures CustomerID in ViewState and calls BindAccounts(customerId).
  3. Accounts grid renders with AccountNumber, Balance, AvailableBalance, Status, and OpenDate.

Workflow 3: Open a New Account

A staff member opens a new bank account for a customer by submitting the "Open Account" form on Default.aspx.

Steps:

  1. Staff enters Customer ID, selects Account Type from dropdown, and enters an opening balance.
  2. Validation: All three fields must parse to valid int/int/decimal values; invalid input cancels the operation with an error message.
  3. A unique AccountNumber is generated from the current UTC timestamp and customer ID.
  4. A new Accounts row is inserted with Status='Active', Balance and AvailableBalance set to the opening balance, and audit timestamps.
  5. The accounts grid is refreshed for the affected customer.

Workflow 4: Close an Account

Staff closes an existing account directly from the accounts grid.

Steps:

  1. Staff clicks the "Close" action button on an account row.
  2. The AccountID is resolved from DataKeys for the selected row index.
  3. An UPDATE sets Status='Closed', CloseDate=GETDATE(), and ModifiedDate=GETDATE() on the account.
  4. The accounts grid is refreshed.

No business rule check prevents closing an account with a non-zero balance — any open account can be closed immediately.

Workflow 5: Retrieve Account Balance

Staff requests the real-time balance for a specific account.

Steps:

  1. Staff clicks "Show Balance" for an account row.
  2. GetBalance(accountId) calls the Zava Ledger REST endpoint GET /api/accounts/{id}/balance.
  3. If the Ledger responds, the XML response is parsed and the balance and availableBalance values are displayed.
  4. If the Ledger is unreachable, the error is silently swallowed and "Ledger balance unavailable." is displayed. No SQL fallback exists for balance.

Workflow 6: Retrieve Transaction History

Staff views the transaction history for a specific account.

Steps:

  1. Staff clicks "Show Transactions" for an account row.
  2. BindTransactions(accountId) calls the Zava Ledger REST endpoint GET /api/accounts/{id}/transactions.
  3. If the Ledger responds, the XML transaction list is parsed and rendered in the transactions grid.
  4. If the Ledger is unreachable, a fallback query fetches the top 25 transactions from the local Transactions SQL table (which may be stale or incomplete compared to the Ledger's authoritative data).

Workflow 7: Staff Sign-out

Staff navigates to Logout.aspx, which calls FormsAuthentication.SignOut() to invalidate the .ZAVAAUTH cookie and redirects the browser to AuthGatewayLogoutUrl for gateway-side session termination.

Cross-Service Data Flows

Balance and transaction aggregation: ZavaAccountManager is the authoritative source for account ownership and status (stored in ZavaBankDB), while the Zava Ledger Service holds the authoritative real-time financial records. When staff requests a balance or transaction list, the portal calls the Ledger over HTTP and parses the XML response inline.

Fallback behavior (business impact): If the Ledger service is unavailable:

  • Balance requests return "Ledger balance unavailable." — staff cannot see the current balance.
  • Transaction requests fall back to the local Transactions SQL table, which may be incomplete or stale. Staff may see outdated transaction records without any indication that the data is from a fallback source, creating a risk of incorrect account management decisions.

Identity delegation: User identity is fully delegated to ZavaAuthGateway. The portal trusts the Forms Authentication ticket issued by the gateway and reads Context.User.Identity.Name for display purposes only; no local user store exists.

Business Rules & Decision Logic

Validation rules:

  • Open Account: CustomerID must be a valid integer, AccountTypeID must be a valid integer matching a selected value, and OpeningBalance must be a valid decimal. Any parse failure cancels the insert and displays "Invalid values." — no partial writes occur.
  • No minimum balance rule is enforced on account opening.
  • No rule prevents opening multiple accounts of the same type for the same customer.

State transitions — Account lifecycle:

  • Active → Closed: triggered by staff via the "Close Account" action. CloseDate and ModifiedDate are set to the database server time (GETDATE()). No reverse transition (re-opening) is implemented.
  • No Pending, Frozen, or Suspended states are handled.
  • No rule prevents closing an account with a non-zero balance.

Account number generation:

  • AccountNumber is derived deterministically as yyMMddHHmmss (UTC) concatenated with the customer ID zero-padded to 2 digits. This is not guaranteed unique under high concurrency; no database-level uniqueness constraint is visible in the code.

Authorization:

  • All pages except Login.aspx and Logout.aspx require an authenticated Forms Auth ticket (<deny users="?"/>). There is no role-based access control (RBAC) — any authenticated user can perform all operations (open accounts, close accounts, view all customers).
  • Customer data is not scoped to a specific staff member or branch; all authenticated users see all customers.

Transaction management:

  • No explicit TransactionScope or SQL transactions are used. Each SqlCommand executes as a single auto-committed statement. If fvOpenAccount_ItemInserting fails mid-execution, there is no rollback mechanism.

Error handling:

  • Ledger HTTP failures are silently caught with a bare catch {} block. No error is logged or surfaced to the user beyond the fallback message, making operational debugging difficult.
  • The customErrors mode="Off" setting means unhandled exceptions display full stack traces to users — a security and UX concern.

Audit trail:

  • CreatedDate and ModifiedDate are set on account creation; ModifiedDate and CloseDate are updated on closure. No audit log table, change tracking mechanism, or event sourcing is implemented.

Version & Compatibility Risks

All dependencies are .NET Framework 4.8 built-in assemblies with no separate versioning. The entire stack is tied to .NET Framework 4.8, which is in long-term maintenance mode (no new features) and is Windows-only. System.Web — the core assembly for ASP.NET Web Forms — does not exist in .NET 5+ / .NET Core, making it the single most significant migration blocker. System.Data.SqlClient has a cross-platform successor (Microsoft.Data.SqlClient), but upgrading requires a framework migration first. No third-party packages are present, so there are no NuGet-level CVE or version-drift concerns.

Notable Observations

  • Zero third-party NuGet packages: The application relies entirely on the .NET Framework 4.8 BCL. This simplifies dependency management but limits access to modern libraries (e.g., Serilog, Polly, Microsoft.Extensions.DependencyInjection).
  • System.Web is the critical migration blocker: ASP.NET Web Forms and System.Web have no direct equivalent in modern .NET. Migration would require rewriting the UI layer (e.g., Blazor Server, ASP.NET Core MVC/Razor Pages) or containerising the existing app on Windows Server.
  • No abstractions over data access: SQL queries are written directly in code-behind with inline SqlConnection/SqlCommand, making a future switch to Microsoft.Data.SqlClient or EF Core a code-level refactor rather than a package swap.
  • HttpWebRequest (System.Net): Used to call the external Ledger service; the modern replacement is HttpClient (available from .NET Framework 4.5+ and .NET Core).

Caching Strategy

No caching layer is configured. Every page load issues fresh SQL queries to ZavaBankDB. There is no in-process cache (MemoryCache, HttpRuntime.Cache), no distributed cache (Redis, Memcached), and no second-level cache. ASP.NET Web Forms ViewState is used to persist the selected CustomerID across postbacks, but this is UI state management, not a data cache.

Data Ownership Boundaries

ZavaAccountManager is a single-service application with a single shared SQL Server database. There are no service boundaries or data isolation concerns within the application itself. The Zava Ledger microservice maintains its own data store (not visible to this codebase); ZavaAccountManager calls the Ledger via REST for balance and transaction data, avoiding direct cross-database queries. The only cross-service data flow is read-only: ZavaAccountManager reads Ledger data via HTTP and falls back to its local Transactions table when the Ledger is unavailable — creating a potential data inconsistency risk if the two stores diverge.
