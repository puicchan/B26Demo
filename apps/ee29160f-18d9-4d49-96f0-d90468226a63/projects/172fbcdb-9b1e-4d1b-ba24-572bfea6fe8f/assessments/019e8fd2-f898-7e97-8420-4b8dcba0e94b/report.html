Core Business Workflows

The application supports banking statement lifecycle workflows: generation for a requested period, history retrieval, and statement detail access.

Primary Workflows

Workflow 1: Generate Monthly Statement

  1. Client invokes GenerateStatement(accountId, month, year).
  2. Service validates account and period rules.
  3. Service fetches account/customer and in-period transactions.
  4. Service computes credit/debit totals and renders HTML statement.
  5. Service writes request, generated statement, and archive metadata/checksum.
  6. Service returns generated statement payload.

Workflow 2: Retrieve Statement History

  1. Client invokes GetStatementHistory(accountId).
  2. Service validates account ID and queries latest generated statements.
  3. Service returns ordered history list for the account.

Workflow 3: Retrieve Statement Detail

  1. Client invokes GetStatement(statementId).
  2. Service validates ID and fetches single statement row.
  3. Service returns detail or faults when not found.

Cross-Service Data Flows

No cross-service composition flow is implemented. All business data reads and writes occur inside a single service boundary with direct SQL interactions.

Business Workflow Sequence

sequenceDiagram participant Client participant Service as "StatementService" participant DB as "SQL Server" Client->>Service: Request monthly statement Service->>Service: Validate account and period Service->>DB: Read account and transaction data DB-->>Service: Account and transaction records Service->>Service: Apply totals and build statement document Service->>DB: Persist request and generated statement DB-->>Service: Generated IDs saved Service-->>Client: Return generated statement response

Business Rules & Decision Logic

  • accountId must be greater than 0.
  • month must be within 1..12.
  • year must be in 2000..2100.
  • Credit/debit totals are computed from transaction type/amount signs.
  • Missing account or statement IDs result in explicit service faults.
