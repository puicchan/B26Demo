Data Storage & External Services

The application relies exclusively on a single Microsoft SQL Server database (ZavaBankDB) accessed via JDBC. It reads from and writes to two core tables: Accounts (joined with AccountTypes for interest rates) and InterestAccruals (created at startup if absent). No external APIs, message brokers, caches, or third-party services are used. Connection parameters (host, port, database name, credentials) are resolved from environment variables or the interest.properties configuration file.

Key Architectural Decisions

  • No framework: The application uses raw Java EE Servlets without Spring, Jakarta EE CDI, or any DI container — all wiring is manual through static factory calls.
  • Direct JDBC: Data access is performed via DriverManager.getConnection() and hand-written SQL, with manual transaction management (commit/rollback) rather than a persistence framework like JPA or Hibernate.
  • Configuration via environment variables with property file fallback: InterestConfig reads environment variables first and falls back to interest.properties, enabling container-friendly deployment.

DTOs & Contracts

The application has no formal DTO or contract classes. All API contracts are expressed inline within each Servlet using org.json.JSONObject for both request parsing and response building:

  • Request deserialization: Each Servlet reads the raw HTTP body as a string, parses it with new JSONObject(body), and extracts fields via optInt, optString, optDouble, and optBoolean accessors. No request model class exists.
  • Response serialization: Responses are constructed programmatically with JSONObject.put(...) and written directly to response.getWriter(). No response model class exists.
  • Internal snapshot: InterestCalculateServlet uses a private inner class AccountSnapshot (fields: balance, accountTypeRate) as a transient in-memory holder populated from the SQL ResultSet. It is not exposed over the wire.
  • No OpenAPI/Swagger specification exists.
  • No protobuf schemas or GraphQL schemas exist.
  • No serialization configuration (Jackson, custom serializers, content negotiation) — the application manually writes JSON strings.

Communication Patterns

Synchronous only. All interactions are synchronous HTTP request/response with no asynchronous or event-driven patterns:

  • Client → Application: Plain HTTP (no HTTPS/TLS configured at the application level). Clients POST JSON payloads directly to Servlet endpoints.
  • Application → Database: Synchronous JDBC via DriverManager.getConnection(). Each request opens a new connection, executes SQL, and closes the connection in a finally block. No connection pool (e.g., HikariCP, DBCP) is configured.
  • Transactions: InterestCalculateServlet and InterestAccrueAllServlet use manual JDBC transaction management (setAutoCommit(false) / commit() / rollback()).
  • No resilience patterns: No circuit breaker, retry logic, timeout configuration, or bulkhead patterns are implemented.
  • No service discovery: The application is a single-service WAR with no inter-service calls; database connection details are resolved from environment variables or interest.properties.
  • No API gateway: Requests reach the Servlet container directly.
  • No client-side load balancing.
  • Startup dependency chain: InterestBootstrapServlet is declared with load-on-startup=1, so it runs DDL (creates the InterestAccruals table if absent) before the other Servlets handle traffic. If the database is unavailable at startup the bootstrap will throw a ServletException, preventing application deployment.
  • Security posture: No authentication or TLS is configured at the application level. All endpoints — including /api/interest/accrue-all which modifies all active account balances — are publicly accessible with no authorization checks. There is no Spring Security, filter-based auth, or API key validation present.

Core Business Workflows

ZavaInterestCalculator automates bank interest calculations and accruals for ZavaBank, enabling per-account interest computation (simple, compound, or amortized) and bulk accrual processing across all active customer accounts.

ZavaInterestCalculator operates within the Interest Management bounded context. It reads accounts and account type rates from a shared database owned by an external core banking system, and writes accrual records back to that same database. There is no API or event-based integration with the owning system — data is accessed via direct SQL against the shared ZavaBankDB database.

Primary Workflows

Workflow 1: Calculate Interest for a Single Account

Triggered by a client POST to /api/interest/calculate with an account ID and calculation parameters.

Steps:

  1. Parse and validate request — Extract accountId (required, must be > 0), interestType (default: simple), days (default: 30), annualRate (optional override), compoundsPerYear (default: from config), termMonths (default: 12), applyToBalance (default: false).
  2. Load account — Query the database for the account's current balance and account-type interest rate. Return 404 if not found.
  3. Resolve annual rate — Use the rate from the request if provided; fall back to the account type's rate; if that is also zero, use the system default annual rate from configuration.
  4. Calculate interest — Apply the selected formula (simple, compound, or amortized) to the principal balance for the given number of days.
  5. Optionally persist accrual — If applyToBalance is true, update the account's balance and record a new InterestAccrual row in a single database transaction; if false, return the calculated amount without writing any changes.
  6. Return result — Return a JSON response with the interest amount, balance before/after, and all calculation parameters.

Business rules involved: Interest type selection, rate resolution hierarchy, minimum field validation, optional accrual persistence.

Workflow 2: Batch Accrue Interest for All Active Accounts

Triggered by a client POST to /api/interest/accrue-all, typically as a scheduled batch operation initiated by an external scheduler.

Steps:

  1. Parse parameters — Extract interestType (default: simple), days, compoundsPerYear, termMonths from the optional JSON body; all default to configured values if absent.
  2. Load all active accounts — Query the database for all accounts with Status = 'Active', along with their account-type interest rates.
  3. Begin database transaction — All updates are applied atomically.
  4. Process each account — For each active account: resolve rate (use account type rate or fall back to system default if zero), calculate interest using the selected formula, update the account's balance, and insert an InterestAccrual record.
  5. Commit or rollback — On success, commit the transaction and return totals; on any SQL error, roll back the entire batch.
  6. Return batch summary — Return a JSON response with the count of accrued accounts and total interest applied.

Business rules involved: Only Active accounts are eligible for batch accrual; system default rate applies when no account-type rate is defined; all-or-nothing transaction semantics.

Workflow 3: Application Startup — Schema Bootstrap

Triggered automatically by the Servlet container when the application starts (load-on-startup).

Steps:

  1. Open a JDBC connection to the database.
  2. Execute conditional DDL: create the InterestAccruals table only if it does not already exist.
  3. Close the connection; if the database is unavailable, throw a ServletException to abort deployment.

Workflow 4: Health Check

Triggered by a GET to /health. Returns a static HTML response confirming the application process is running. No database liveness check is performed.

Cross-Service Data Flows

ZavaInterestCalculator is a single-service application with no inter-service REST or messaging calls. The only data flows are between the application and SQL Server:

  • Inbound: The application reads Accounts and AccountTypes data from the shared database managed by an external core banking system. Account ownership is assumed to be managed externally.
  • Outbound: The application writes updated Balance, AvailableBalance, InterestAccrued, LastActivityDate, and ModifiedDate fields back to the Accounts table, and inserts new rows into InterestAccruals.
  • No fallback behavior: There is no circuit breaker, retry, or degraded mode. If the database is unavailable during a request, the operation returns HTTP 500.

Interest Calculation Formulas

TypeFormula
simpleprincipal × annualRate × days / 365
compoundprincipal × ((1 + annualRate/n)^(n × days/365) − 1) where n = compoundsPerYear
amortizedDaily interest derived from standard amortization payment formula over termMonths

Transaction Boundaries

  • Single account accrual (applyToBalance=true): A manual JDBC transaction wraps the UPDATE Accounts and INSERT InterestAccruals pair. On failure, the transaction is rolled back.
  • Batch accrual (/accrue-all): A single JDBC transaction wraps all account updates and accrual inserts for the entire batch. Any single failure rolls back all changes for all accounts processed so far — all-or-nothing semantics.

Error Handling

  • Invalid JSON payload or missing required fields → HTTP 400 with {"status":"ERROR","message":"..."}.
  • Account not found → HTTP 404.
  • SQL / database errors → HTTP 500; no compensating action or retry.
  • Unsupported interestType value → IllegalArgumentException caught as HTTP 400.

Authorization

No authorization or authentication is enforced. All endpoints are publicly accessible without any role check or API key. The /api/interest/accrue-all endpoint in particular can mutate all active account balances without any access control.

Audit / Logging

The InterestAccruals table serves as an implicit audit trail: every applied accrual is recorded with account ID, interest type, principal, rate, days, amount, and timestamp. However, there is no application-level structured logging, correlation IDs, or change-data-capture mechanism.

Version & Compatibility Risks

The javax.servlet-api 3.1.0 dependency targets the legacy javax. namespace (Java EE 7 era). Modern Servlet containers such as Tomcat 10+ and WildFly 27+ have migrated to the jakarta. namespace (Jakarta EE 9+), meaning the application cannot run on those containers without code changes. The org.json artifact is pinned to a 2014 release (20140107) that predates many bug fixes and improvements; while it is functional for basic use, it is significantly outdated and the old date-based versioning scheme makes vulnerability tracking harder. The mssql-jdbc 12.6.3.jre8 driver is a recent, actively maintained release and poses no immediate compatibility risk, though the JRE 8 variant should be updated to a JRE 11+ variant when the JDK is upgraded.

Notable Observations

  • No logging framework declared: The application has no SLF4J, Log4j 2, or Logback dependency. Any logging relies on System.out/System.err or the Servlet container's built-in logger, which limits observability in cloud environments.
  • No dependency injection or web framework: The entire stack is built on raw Java EE Servlets without Spring Boot, Quarkus, or Micronaut — migrating to a modern framework would require significant restructuring.
  • org.json 20140107 is over 10 years old: This ancient version has no release metadata (no MANIFEST.MF version, no OSGi bundle version) making it difficult to track in software composition analysis (SCA) and security scanning tools. Upgrading to a recent release (e.g., 20240303) or switching to Jackson is recommended.
  • Minimal dependency footprint: With only 3 declared dependencies the application has a very small transitive closure, which simplifies containerization and migration but also means many cross-cutting concerns (security, observability, resilience) are entirely absent.

Transaction management is manual: connection.setAutoCommit(false) / connection.commit() / connection.rollback(). There are no @Transactional annotations or transaction management frameworks.

Caching Strategy

No caching layer is implemented. Every request triggers a live JDBC query against SQL Server. Account data, type/rate lookups, and accrual records are fetched and written directly on each call with no in-memory cache, query result cache, or second-level cache.

There is no Redis, EhCache, Caffeine, Spring Cache, or JCache (JSR-107) integration.

Data Ownership Boundaries

ZavaInterestCalculator operates against a shared relational database (ZavaBankDB) that is assumed to be owned and populated by an external core banking system. The application has a narrow write scope:

  • Owns and manages: InterestAccruals table (created at startup, written by both calculate and accrue-all operations, not read externally by this service).
  • Reads but does not own: Accounts and AccountTypes tables — these are treated as pre-existing shared tables. The application reads balances and interest rates, and updates Balance, AvailableBalance, InterestAccrued, LastActivityDate, and ModifiedDate on Accounts rows, which represents cross-boundary write access to a table it does not own.
  • No database-per-service isolation: There is a single shared database with no logical schema separation between this service's domain and the broader banking schema.
  • No CQRS: All reads and writes use the same JDBC connection against the same tables.

The InterestAccrueAllServlet performs a full-table scan of all active accounts in a single transaction, making it a heavyweight batch operation that could contend with other writers on the Accounts table.
