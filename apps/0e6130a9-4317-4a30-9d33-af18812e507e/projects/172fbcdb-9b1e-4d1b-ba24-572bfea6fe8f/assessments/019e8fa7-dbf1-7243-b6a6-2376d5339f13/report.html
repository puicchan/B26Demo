Data Storage & External Services

The application stores and retrieves loan products and loan applications in SQL Server (ZavaBankDB) through direct SQL commands. It also integrates with an external authentication gateway for login/logout redirection and a loan-origination API for outbound submission.

Key Architectural Decisions

  • Uses classic WebForms code-behind event handlers instead of a separate controller/service layer.
  • Uses direct ADO.NET SQL statements for persistence without an ORM abstraction.
  • Uses forms authentication with unauthenticated users redirected to Login.aspx.

DTOs & Contracts

The codebase does not define dedicated DTO classes; contracts are implicit in page controls and an XML string payload used for outbound submission. The outbound contract to the loan-origination service is <loanApplication> XML containing customer, product, amount, and term values. No OpenAPI, protobuf, or GraphQL schema files were found.

Communication Patterns

Communication is synchronous throughout: browser-to-WebForms pages over HTTP, page logic to SQL Server via ADO.NET, and page logic to the external loan-origination API using HttpWebRequest. No asynchronous messaging, service discovery, circuit breaker, or retry policy is configured. Security posture is basic forms authentication and page authorization rules; there is no explicit API-level OAuth/JWT/TLS enforcement in this repository.

Core Business Workflows

The application supports customer-facing loan application intake: users authenticate, complete a multi-step wizard, submit applications, and review recent submission history.

Primary Workflows

Workflow 1: Submit Loan Application

  1. Authenticated user opens Default.aspx and loads available loan products and recent history.
  2. User progresses through wizard steps entering customer and loan information.
  3. System validates numeric loan amount and term before allowing final step.
  4. On finish, portal sends loan payload to downstream loan-origination API.
  5. Portal persists a submitted application record in SQL Server and refreshes history.

Workflow 2: Review Loan Application History

  1. Authenticated user opens the portal.
  2. System queries the latest loan applications and binds them to the history grid.
  3. User can choose a row action to mark selection for review in the UI.

Cross-Service Data Flows

The portal coordinates a simple cross-service flow during submission: user input is transformed into an XML payload and sent to the external loan-origination API, while a local database record is also created for portal-side history. Authentication is delegated to an external gateway via redirect links rather than direct credential handling in this service.

Business Rules & Decision Logic

  • Access control rule: unauthenticated users are redirected to login before using loan workflows.
  • Validation rule: submission step requires parseable numeric RequestedAmount and integer TermMonths.
  • Submission state rule: newly created records are persisted with Status='Submitted' and current timestamps.
  • Decision branch: if downstream API URL is empty, outbound submission is skipped; persistence still follows local workflow.
