Data Storage & External Services

The application relies on a single SQL Server database (ZavaBankDB) accessed directly via ADO.NET for reading loan products (LoanProducts table) and writing loan applications (LoanApplications table). Authentication state is maintained with ASP.NET Forms Authentication cookies (.ZAVAAUTH), but the actual credential verification is delegated to an external ZavaAuthGateway service whose URLs are configured in web.config. On final submission of a loan application, the portal also dispatches an HTTP POST carrying an XML payload to the Loan Origination API (zava-loan-origination-api:8080), providing a fire-and-forget integration with the downstream loan origination backend.

Key Architectural Decisions

  • Direct ADO.NET data access: No ORM or repository abstraction is used; all SQL is inline within page code-behind, tightly coupling data access to the UI layer.
  • Delegated authentication via external gateway: The portal does not validate credentials itself; it redirects unauthenticated users to a separate ZavaAuthGateway service, making Forms Authentication act purely as a session validator.
  • Fire-and-forget REST integration: Loan submission posts an XML document to the Loan Origination API synchronously but silently swallows all exceptions, meaning submission failures are invisible to the user.

DTOs & Contracts

ZavaLoanPortal defines no DTO classes. All data is passed between the web UI and the server via ASP.NET Web Forms ViewState and server-side control properties.

Outbound XML contract to Loan Origination API: The single outbound payload is an XML string constructed inline in Default.aspx.cs with no formal schema or strongly typed class:


<loanApplication>
  <customerId>{int}</customerId>
  <loanProductId>{int}</loanProductId>
  <requestedAmount>{decimal F2}</requestedAmount>
  <termMonths>{int}</termMonths>
</loanApplication>

There is no OpenAPI/Swagger specification, no protobuf schema, no GraphQL schema, and no shared contract library. Serialization is handled by manual string concatenation.

Communication Patterns

Synchronous — HTTP redirect (Auth Gateway)

Login and Logout flows are handled by issuing client-side HTTP 302 redirects to the ZavaAuthGateway service. The portal does not perform a server-to-server call; it instructs the browser to navigate to the configured AuthGatewayLoginUrl or AuthGatewayLogoutUrl with a ReturnUrl query parameter. There is no session token exchange or callback verification.

Synchronous — HTTP POST to Loan Origination API

On wizard completion, Default.aspx.cs constructs an XML payload and posts it synchronously to LoanOriginationApiUrl using the legacy HttpWebRequest API. The call is fire-and-forget: all exceptions are silently caught and discarded, and the response body is read but not parsed. There is no retry policy, circuit breaker, timeout configuration, or fallback behavior implemented.

Synchronous — Direct ADO.NET to SQL Server

All database reads (loan products, loan history) and writes (insert loan application) are performed via synchronous SqlConnection/SqlCommand calls. Connection strings are read from web.config. There is no connection pooling configuration beyond the ADO.NET default, and no retry policy for transient SQL errors.

Resilience patterns: None. No circuit breaker, no retry, no timeout is configured for any outbound call.

Service discovery: Hardcoded URLs in web.config (appSettings). No service registry (Consul, Eureka, Azure Service Discovery) is used.

Security posture: Forms Authentication is enabled with a cookie named .ZAVAAUTH (AES encryption, SHA1 validation, 30-minute sliding expiration). All pages except Login.aspx and Logout.aspx are protected by <deny users="?"/>. However:

  • The machineKey (validationKey and decryptionKey) is hardcoded in plaintext in web.config, which is a critical security risk.
  • Outbound HTTP calls to both the Auth Gateway and the Loan Origination API are plain HTTP (not HTTPS), meaning credentials and loan data are transmitted without transport encryption.
  • No CSRF protection beyond the built-in ASP.NET Web Forms ViewState MAC is present.
  • No authorization roles or claims-based access control are implemented; any authenticated user can access all pages.

Core Business Workflows

ZavaLoanPortal is a front-office loan origination portal that allows bank staff to submit loan applications on behalf of customers and review a history of recent applications, with all submitted applications forwarded to a downstream loan origination system.

Primary Workflows

Workflow 1: User Authentication

The portal does not perform credential validation itself. When an unauthenticated user accesses any protected page, ASP.NET Forms Authentication redirects them to Login.aspx, which immediately redirects the browser to the external ZavaAuthGateway login URL (configured in web.config). The Auth Gateway authenticates the user and redirects back to the portal with the .ZAVAAUTH session cookie set. On logout, Logout.aspx calls FormsAuthentication.SignOut() to clear the local session cookie and redirects the browser to the Auth Gateway logout URL. The portal has no control over or visibility into the credential verification process.

Steps:

  1. Unauthenticated user navigates to any page → Forms Authentication redirects to Login.aspx?ReturnUrl=...
  2. Login.aspx builds the Auth Gateway login URL (appending ReturnUrl) and issues a browser redirect
  3. ZavaAuthGateway authenticates the user and redirects back with .ZAVAAUTH cookie
  4. User accesses the requested page with an authenticated session

Workflow 2: Loan Application Submission (5-Step Wizard)

This is the primary business workflow. A logged-in staff member completes a multi-step wizard to submit a loan application on behalf of a customer.

Steps:

  1. Page Load — Portal fetches active loan products from LoanProducts table and the top 25 recent applications from LoanApplications table; both are displayed on the page.
  2. Step 1 — Personal Information — Staff enters Customer ID, first name, last name, and email address. These fields are held in ViewState; first name, last name, and email are NOT persisted to the database.
  3. Step 2 — Employment — Staff enters employer name, job title, years of employment, and annual income. These fields are held in ViewState only and are NOT persisted to the database.
  4. Step 3 — Loan Details — Staff selects a loan product from the dropdown, enters the requested amount (decimal), term in months (integer), and an optional purpose text.
  5. Validation on Next — When advancing from Step 3, the portal validates that RequestedAmount is a valid decimal and TermMonths is a valid integer. If invalid, navigation is cancelled and an error message is shown.
  6. Step 4 — Review — A summary of customer name, requested amount, and term is displayed for confirmation.
  7. Step 5 — Submit (Finish) — On wizard completion:

a. An XML payload is constructed from the form values and HTTP-POSTed to the Loan Origination API (fire-and-forget; all exceptions silently discarded).

b. A SQL INSERT writes the application to LoanApplications with Status='Submitted', AssignedOfficer=User.Identity.Name, and server-generated timestamps.

c. The loan history grid is refreshed and the wizard resets to Step 1.

Workflow 3: Loan History Review

A logged-in staff member can view the 25 most recent loan applications. The LoanApplications table is queried on every page load and after every submission, returning ApplicationID, CustomerID, RequestedAmount, TermMonths, Status, and ApplicationDate. A "Review" button is available on each row; clicking it sets a status label to "Application selected for review" but performs no further action in the current implementation (the review workflow is not implemented).

Cross-Service Data Flows

ZavaLoanPortal is not a microservices gateway and does not aggregate responses from multiple services. It interacts with two external services independently:

  1. ZavaAuthGateway → Portal: The authentication flow is entirely browser-mediated (redirects). No server-to-server call is made; the portal receives only the .ZAVAAUTH cookie and the User.Identity.Name from the authenticated session.
  2. Portal → Loan Origination API: On submission, the portal pushes a one-way XML payload to the Loan Origination API. There is no response validation, no correlation ID, and no fallback behavior — if the API call fails, the portal proceeds to write to the local database anyway, creating a potential inconsistency between the portal's record and the downstream system.

Inconsistency risk: The portal first calls the Loan Origination API and then inserts into SQL Server within the same HTTP request, but there is no transaction spanning both operations. A failure after the API call but before the SQL INSERT, or a failure after the SQL INSERT but before the page response, would leave the two systems in an inconsistent state.

Business Rules & Decision Logic

Validation Rules

  • Loan amount: Must be parseable as a decimal. Checked on wizard step advancement from Step 3. No minimum, maximum, or precision constraint is enforced in code.
  • Term months: Must be parseable as an int. Checked on wizard step advancement from Step 3. No minimum or maximum value constraint is enforced in code.
  • Customer ID: Parsed as int without validation on submission (int.Parse — will throw unhandled exception if non-integer).
  • Product ID: Parsed as int from dropdown selected value; no further validation.
  • Purpose: Optional; stored as-is with no length or content constraints enforced in application code.
  • Wizard inputs (name, email, employment): Not validated — collected by wizard controls but not checked for emptiness, format, or length before submission.

State Transitions

LoanApplications have a single initial status: Submitted. No state machine, workflow engine, or status transition logic is implemented in the portal. Further status transitions (e.g., Under Review, Approved, Rejected) are presumed to occur in the Loan Origination API or a separate back-office system.

Business Constraints

  • Authentication required: All pages except Login.aspx and Logout.aspx deny unauthenticated users (enforced by web.config <deny users="?"/>).
  • AssignedOfficer: The currently authenticated user's Windows identity (User.Identity.Name) is automatically set as the assigned officer on every submitted application.
  • Loan history: Always shows the 25 most recent applications across all officers and customers; there is no filtering by customer, officer, or date range.
  • Active products only: The product dropdown is populated only with loan products where IsActive=1.

Cross-Cutting Concerns

  • Transactions: No explicit transaction management (TransactionScope or SqlTransaction). The API call and the SQL INSERT are sequential but not atomic. An INSERT failure after a successful API call will leave the downstream system with a record the portal does not have.
  • Error handling: All exceptions from the Loan Origination API HTTP call are caught and silently discarded. SQL exceptions from ADO.NET are unhandled and will surface as unhandled exceptions to the user. There is no business exception type hierarchy.
  • Audit trail: No dedicated audit log. ApplicationDate, CreatedDate, ModifiedDate, and AssignedOfficer provide a minimal audit trail in LoanApplications. No change-tracking or event log is maintained.
  • Authorization: Role-based access control is not implemented. Any authenticated user can submit applications and view all loan history regardless of their role or the customer they are assigned to.

Version & Compatibility Risks

All dependencies are .NET Framework 4.8 in-box assemblies. .NET Framework 4.8 is in long-term support but is Windows-only and will not receive new features; Microsoft's recommended migration path is to .NET 8 or later (cross-platform). System.Web (Web Forms) has no direct equivalent in modern .NET — migrating the presentation layer requires a full rewrite to ASP.NET Core (Razor Pages, MVC, or Blazor). System.Data.SqlClient is superseded by the separately distributed Microsoft.Data.SqlClient NuGet package in .NET 8+, which includes Active Directory authentication and other improvements. The absence of any third-party NuGet packages simplifies dependency management but means the application carries no version pinning or reproducible restore for libraries beyond the runtime itself.

Notable Observations

  • No third-party NuGet packages: The packages.config is empty and no <PackageReference> entries exist; the entire dependency surface is the .NET Framework 4.8 BCL, which reduces supply-chain risk but also means no modern libraries (e.g., logging, HTTP client abstractions, DI container) are in use.
  • Direct ADO.NET without an ORM: System.Data is used with raw SqlConnection/SqlCommand; migrating to .NET 8 will require either adopting Microsoft.Data.SqlClient and retaining ADO.NET patterns or introducing an ORM such as Entity Framework Core.
  • System.Web dependency is a hard modernization blocker: The entire Web Forms programming model (System.Web.UI.Page, System.Web.UI.MasterPage, UpdatePanel, Wizard, GridView) is locked to .NET Framework and is not available in .NET 8+, making this the most significant migration effort.
  • Legacy project format: The .csproj uses the old MSBuild format (ToolsVersion="4.0" with explicit <Compile> entries) rather than the SDK-style format, which will need to be converted during any .NET modernization.

Caching Strategy

No caching is implemented at any layer. There is no in-memory cache (MemoryCache), distributed cache (IDistributedCache, Redis), second-level cache, or query result cache. Loan products are re-fetched from SQL Server on every page load. Loan history is re-fetched on every page load and after every submission. This means every request issues at minimum two SQL round-trips, with no opportunity to serve repeated reads from a cache.

Data Ownership Boundaries

ZavaLoanPortal uses a single shared SQL Server database (ZavaBankDB). There are no other services in the solution that access this database — the portal is the sole consumer of the two tables identified. However, the Loan Origination API (zava-loan-origination-api:8080) is a separate downstream service that may also write to or read from ZavaBankDB independently; this relationship is not visible from the portal's source code.

All database access is concentrated in a single page code-behind file (Default.aspx.cs), with no separation between the data-access concern and the UI concern. There is no CQRS pattern, no event sourcing, and no outbox table. Read and write operations share the same synchronous ADO.NET connection path.
