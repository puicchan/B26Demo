Data Storage & External Services

ZavaNotifyWorker does not maintain its own persistent storage. It relies on two external services: RabbitMQ acts as the message source, with the worker polling the q.notify.email queue over AMQP. Failed messages are negatively acknowledged (BasicNack) and routed to the zava.dlx dead-letter exchange for later inspection. The SMTP server (defaulting to MailHog for local development) is the delivery target — the worker opens a plain-text SMTP connection and sends a formatted alert email per message.

Key Architectural Decisions

  • Polling over push: The worker uses BasicGet (polling) rather than a push-based consumer (BasicConsume), with a configurable sleep interval between poll cycles, keeping the architecture simple at the cost of slight latency.
  • Multi-format message parsing: Incoming messages are auto-detected as either JSON or XML based on the first character, enabling backward compatibility with multiple upstream producers.
  • Env-var override pattern: All configuration keys have a dual-resolution strategy — environment variables take priority over app.config AppSettings, making the worker container-friendly without code changes.

Communication Patterns

Asynchronous inbound (RabbitMQ → Worker):

The worker uses polling via IModel.BasicGet rather than a push-based subscription. Each poll cycle opens a new AMQP connection and channel, calls BasicGet on q.notify.email, and closes the connection after processing. The poll interval is configurable via Worker.PollIntervalMs (default 5 000 ms). On successful processing the message is acknowledged with BasicAck; on any processing failure it is rejected with BasicNack(requeue: false), routing it to the zava.dlx dead-letter exchange.

Synchronous outbound (Worker → SMTP):

Email delivery is synchronous — SmtpClient.Send blocks the poll cycle until the SMTP server accepts the message or throws. No retry policy, circuit breaker, or timeout is configured on the SMTP call; a slow or unavailable SMTP server will block the polling loop indefinitely.

Resilience: No resilience library (Polly, Spring Retry, etc.) is used. The outer while (_keepRunning) loop catches all exceptions from PollQueue and logs them, then resumes polling after the configured interval — providing minimal fault isolation at the loop level but no per-operation retry or circuit breaking.

Service discovery: Services are located by hardcoded hostname/port resolved from configuration (env vars → app.config → defaults). No service registry (Consul, Eureka, Kubernetes DNS) is used.

Security posture: No TLS/SSL is configured for the AMQP connection (AutomaticRecoveryEnabled = false; no SSL options set on ConnectionFactory). SMTP uses UseDefaultCredentials = true with no explicit TLS configuration. The RabbitMQ credentials (zava_app / zava_pass) are stored in plaintext in app.config. There is no authentication or authorisation layer within the worker itself — all access control is delegated to the RabbitMQ broker's vhost permissions.

Primary Workflows

Workflow 1: Transaction Alert Email Delivery

Trigger: A message arrives on the q.notify.email RabbitMQ queue (published by an upstream transaction-processing or account-management service).

Steps:

  1. Poll: The worker's main loop wakes after PollIntervalMs (default 5 s) and calls BasicGet on q.notify.email.
  2. Detect format: The raw message body is inspected — if it begins with < it is treated as XML; otherwise as JSON.
  3. Deserialize: The body is parsed into a NotificationMessage DTO.
  4. Validate customer email: SendEmail checks that CustomerEmail is non-empty; if absent, an exception is raised and the message is dead-lettered.
  5. Compose email: Subject and body are formatted using alert type, transaction ID, amount, currency, customer name, and processing timestamp.
  6. Deliver: SmtpClient.Send transmits the email to the configured SMTP relay.
  7. Acknowledge: On success, BasicAck removes the message from the queue; on any failure, BasicNack (no requeue) routes it to zava.dlx.

Business outcome: The bank customer receives an email alert describing the transaction event within one polling interval of the message being published.

Workflow 2: Dead-Letter Handling

Trigger: A NotificationMessage cannot be processed (deserialization failure, missing customer email, SMTP error, etc.).

Steps:

  1. The exception is caught inside PollQueue.
  2. BasicNack(requeue: false) is issued — the message is not returned to q.notify.email.
  3. The broker routes the message to the zava.dlx dead-letter exchange.
  4. The failed message is available for manual inspection or replay by operations staff.

Business outcome: No notification is lost silently; failed alerts are quarantined for investigation and potential manual re-send.

Workflow 3: Worker Startup and Graceful Shutdown

Trigger: Process start (mono ZavaNotifyWorker.exe) or OS signal (SIGINT / Ctrl+C / process exit).

Startup steps:

  1. LoadConfig() reads all settings from environment variables then app.config with hardcoded defaults as fallback.
  2. Signal handlers are registered (Console.CancelKeyPress, AppDomain.ProcessExit).
  3. The polling loop begins immediately.

Shutdown steps:

  1. _keepRunning is set to false when the shutdown signal is received.
  2. The currently-executing PollQueue cycle completes (or its exception is caught) before the loop exits — no in-flight message is abandoned mid-processing.

Business outcome: The worker does not leave partially-processed messages in an unacknowledged state on clean shutdown.

Cross-Service Data Flows

ZavaNotifyWorker is a pure consumer; it does not aggregate data from multiple services nor does it call any downstream REST API. The complete data flow is linear:


Upstream Banking System
  │  publishes JSON or XML notification event
  ▼

Error Handling

Version & Compatibility Risks

.NET Framework 4.8 is in long-term maintenance mode and will not receive new features; it does not run cross-platform (Linux/macOS) without Mono, which is itself end-of-life. RabbitMQ.Client 5.2.0 is significantly outdated — the current stable line is 6.x/7.x, with v5.x no longer receiving security patches. System.Web.Extensions (used only for JavaScriptSerializer) is a legacy assembly tied to ASP.NET System.Web; JavaScriptSerializer itself has been deprecated in favour of System.Text.Json (available in .NET 6+) or Newtonsoft.Json. Migration to .NET 8+ (LTS) or .NET 10 would require replacing all five built-in BCL references with their modern equivalents and upgrading the RabbitMQ client to v6/v7.

Notable Observations

  • Single external dependency: Only one third-party NuGet package (RabbitMQ.Client) is declared in packages.config, making the migration surface unusually small — the main modernization effort is the framework upgrade itself.
  • Deprecated JavaScriptSerializer: System.Web.Extensions is included solely to use JavaScriptSerializer, which Microsoft deprecated and does not ship on .NET Core / .NET 5+. It must be replaced with System.Text.Json or Newtonsoft.Json before targeting any modern .NET version.
  • packages.config format: The project still uses the legacy packages.config NuGet format rather than PackageReference, which is a prerequisite for migrating to SDK-style project files.
  • No logging framework: The worker logs exclusively via Console.WriteLine, with no structured logging library (e.g., Serilog, NLog, Microsoft.Extensions.Logging). This limits observability and will require deliberate attention during cloud migration.
