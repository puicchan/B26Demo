Data Storage & External Services

The worker does not use a database or cache. It depends on RabbitMQ for message ingestion and dead-letter routing, and on an SMTP endpoint (default mailhog:1025) for outbound email delivery.

Key Architectural Decisions

  • Implements a pull-based worker loop using BasicGet instead of push consumer registration.
  • Supports mixed payload contract formats by parsing both JSON and XML message bodies.
  • Uses explicit ack and dead-letter nack behavior to avoid message loss and isolate failed notifications.

DTOs & Contracts

The service contract centers on the NotificationMessage model, which acts as an internal normalized payload object for both JSON and XML inputs. Input messages support flexible key aliases (for example customerEmail/CustomerEmail/email) and are transformed into this contract before processing. No OpenAPI, Swagger, protobuf, or GraphQL schemas were detected.

Communication Patterns

Communication is asynchronous from RabbitMQ to the worker via queue polling (BasicGet), followed by synchronous SMTP delivery to the configured mail host. Failure handling uses nack with requeue disabled and routes failed messages to a configured dead-letter exchange. No service discovery mechanism is used; hostnames are configured directly through environment variables or app settings. No API gateway, TLS enforcement, authentication, or authorization controls were detected in this service.

Core Business Workflows

This application executes notification delivery workflows for banking events by consuming queued messages and delivering customer-facing email alerts. Its business logic focuses on resilient message processing and failure isolation through dead-letter routing.

Primary Workflows

Workflow 1: Process Notification Message

  1. Worker polls the configured queue for one pending message.
  2. Message payload is normalized into NotificationMessage using JSON or XML parsing logic.
  3. Worker validates required communication attributes (customer email presence).
  4. Email subject and body are composed from normalized business data.
  5. SMTP delivery is attempted.
  6. On success, the queue message is acknowledged and removed.

Business rules involved:

  • Payload may use multiple field aliases and still map successfully.
  • Missing customer email is treated as a hard failure.
  • Acknowledgment occurs only after successful email send.

Workflow 2: Failure Isolation and Dead-Letter Handling

  1. Any parse or email send exception is captured during processing.
  2. Worker logs failure outcome and issues BasicNack with requeue=false.
  3. Broker routes failed message to configured dead-letter exchange.

Business rules involved:

  • Failed messages are not retried inline by this worker.
  • Delivery safety is prioritized by avoiding false-positive acknowledgments.

Cross-Service Data Flows

Upstream banking services publish raw notification events to RabbitMQ. ZavaNotifyWorker consumes those events, maps data into a communication-friendly contract, and sends emails through SMTP infrastructure. When email infrastructure or payload integrity fails, the worker degrades by routing to dead-letter exchange so that operational workflows can inspect and recover failures without blocking subsequent messages.

Business Rules & Decision Logic

  • Notification processing supports two payload formats (JSON/XML) and multiple key aliases to tolerate upstream variance.
  • Missing CustomerEmail stops processing and marks the message as failed.
  • Queue acknowledgment is conditional: success leads to Ack, any exception leads to dead-letter routing.
  • Worker loop continues after each poll cycle, enabling continuous processing even when individual messages fail.
