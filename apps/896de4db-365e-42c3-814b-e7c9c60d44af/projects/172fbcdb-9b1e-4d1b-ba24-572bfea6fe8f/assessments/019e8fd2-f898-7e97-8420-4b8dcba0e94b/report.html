Data Storage & External Services

The application uses a single Microsoft SQL Server instance (ZavaBankDB) as its only data store. It accesses tables including FraudAlerts, Transactions, Accounts, FraudRules, SessionTokens, and Users via raw JDBC (no ORM). Connection parameters (host, port, database name, credentials) are read from frauddetector.properties with environment-variable overrides. An external HTTP service — the ZavaBank Auth Gateway (WhoAmI.ashx) — is called as a fallback to resolve .ZAVAAUTH FormsAuthentication cookies that the Java application cannot decrypt natively.

Key Architectural Decisions

  • Struts 2 action-per-feature pattern: Each feature (login, queue, rule CRUD, alert decision) has its own ActionSupport subclass mapped in struts.xml, keeping request handling focused and explicit.
  • Custom SSO interceptor: AuthTokenInterceptor runs before every action (except login and health) to validate session tokens from cookies, headers, or query params, delegating to SsoSessionService which queries SessionTokens in SQL Server or calls the external auth gateway.
  • Direct JDBC with no ORM: All SQL is written inline in action classes via java.sql.*, with no Spring, Hibernate, or other abstraction layer, making the data-access coupling tight to the presentation layer.

Communication Patterns

Synchronous (inbound): All client interactions are synchronous HTTP request/response cycles handled by the Struts 2 StrutsPrepareAndExecuteFilter. The browser submits form POSTs or GET requests; the action executes, writes to the database, and either renders a JSP or redirects.

Synchronous (outbound — JDBC): Every action that needs data opens a raw JDBC connection via FraudConnectionFactory, executes a PreparedStatement, and closes the connection in a try-with-resources block. There is no connection pooling (no HikariCP, DBCP, or c3p0 configured), meaning a new TCP connection to SQL Server is opened and torn down on every HTTP request.

Synchronous (outbound — HTTP): SsoSessionService calls the Auth Gateway's WhoAmI.ashx endpoint via java.net.HttpURLConnection when a .ZAVAAUTH cookie is present and no session token can be resolved from the DB. A hard-coded 3-second connect and read timeout is applied. No retry logic or circuit breaker is implemented; failures are silently swallowed (the catch (Exception ignored) block returns an empty string), causing the user to be redirected to the login page.

Resilience: No circuit breaker (Resilience4j, Hystrix), retry policy, or bulkhead is configured. Timeout handling exists only for the Auth Gateway HTTP call (3 s connect / 3 s read). JDBC calls have no query timeout set.

Service Discovery: The Auth Gateway URL (http://zava-auth-gateway:8080/WhoAmI.ashx) and SQL Server host are resolved via properties file or environment variables. No service registry (Eureka, Consul, Kubernetes DNS) is used; services are addressed by hardcoded hostnames.

Security posture: There is no HTTPS/TLS configured at the application layer (transport security is expected from the servlet container or a reverse proxy in front). Authentication is enforced by AuthTokenInterceptor on all actions except login and health. Authorization is coarse-grained — any authenticated user can perform any action; no role-based access control (RBAC) or fine-grained authorization is implemented. Session tokens are stored in plain text in the SQL Server SessionTokens table and transmitted via cookies or HTTP headers with no signing or encryption at the application layer.

Primary Workflows

Workflow 1: Analyst Authentication

An analyst accessing any protected page is intercepted by AuthTokenInterceptor. The interceptor checks for a valid session in the following order:

  1. An existing SessionUser already in the HTTP session — if found, access is granted immediately.
  2. A session token in the request (query param sessionToken, header X-Session-Token, or Authorization: Bearer header) — validated against the SessionTokens table.
  3. A .ZAVAAUTH FormsAuthentication cookie — forwarded to the Auth Gateway (WhoAmI.ashx), which returns a session token, which is then validated against the database.
  4. If none resolves, the analyst is redirected to the login page where they can enter a session token manually.

Workflow 2: Review Fraud Alert Queue

The fraud analyst's primary workflow — a dashboard view of the 100 most recent unresolved fraud alerts:

  1. Analyst navigates to /flaggedQueue (authenticated).
  2. The application queries FraudAlerts joined with Accounts and Transactions for all alerts with status New or InReview, ordered by creation date descending, limited to 100 rows.
  3. Results are displayed in a table showing alert type, severity, status, description, account number, amount, and creation date.
  4. From this view, the analyst can navigate to a transaction's detail page or take no action on individual items.

Workflow 3: Transaction Detail Review

The analyst drills into a specific alert to review the full context before making a decision:

  1. Analyst clicks a row in the queue, triggering a GET to /transactionDetail?alertId={id}.
  2. The application fetches the alert with the same FraudAlerts ⟶ Accounts ⟶ Transactions join.
  3. If the alert is not found (or alertId is not provided), the analyst is redirected back to the queue.
  4. The detail page displays full alert and transaction context and offers the approve/reject decision form.

Workflow 4: Alert Decision (Approve / Reject)

The analyst records a decision on a fraud alert:

  1. Analyst submits the decision form on the detail page (POST to /alertDecision).
  2. AlertDecisionAction validates that alertId and decision are non-null and non-empty; missing values result in a no-op (returns SUCCESS without writing).
  3. The decision string is normalized: "Approved" (case-insensitive) maps to Approved; anything else maps to Rejected.
  4. The authenticated analyst's username is retrieved from the HTTP session; if unavailable, "system" is used as the actor.
  5. A resolution string is constructed: any analyst notes are prepended, followed by "Decision by <username>".
  6. The FraudAlerts record is updated: Status, ResolutionNotes, ResolvedDate, and ModifiedDate are written in a single UPDATE statement.
  7. The analyst is redirected to the fraud queue.

Workflow 5: Fraud Rule Management

Analysts can view, create, edit, and delete the fraud detection rules:

  • List (/rules): All rules are fetched from FraudRules ordered by RuleID and displayed.
  • Create (/ruleSave with no ruleId): A new rule is inserted with CreatedDate and ModifiedDate set to the current database time.
  • Edit (/ruleEdit?ruleId=N then POST /ruleSave): The rule is loaded, the form pre-populated, and on save the record is updated.
  • Delete (/ruleDelete?ruleId=N): The rule record is deleted and the analyst is redirected to the rule list.

Cross-Service Data Flows

ZavaFraudDetector is a single deployable unit with no inter-service REST or messaging calls for business data. All fraud alert and rule data is retrieved through direct SQL joins in the shared ZavaBankDB SQL Server database. The only outbound service call is to the Auth Gateway for SSO cookie resolution, which is part of the authentication handshake rather than a business data flow.

When the Auth Gateway is unavailable (connection timeout after 3 seconds), the fallback is silent failure: the unresolvable cookie is ignored and the analyst is redirected to the login page. There is no partial-data fallback for business queries — if SQL Server is unreachable, the action sets an action error message ("Unable to load flagged transaction queue.") and renders an empty or error page.

Business Rules & Decision Logic

Validation Rules

  • Alert Decision: alertId and decision must both be non-null and non-empty; if either is missing, AlertDecisionAction.execute() returns SUCCESS without writing, silently discarding the submission.
  • Rule Name required: RuleEditAction.save() rejects a save if ruleName is null or blank, setting an action error.
  • Severity required: RuleEditAction.save() rejects a save if severity is null or blank.
  • Numeric validation: thresholdAmount must parse as a valid BigDecimal; timeWindowMinutes must parse as a valid integer. Invalid values produce an action error and return INPUT (re-display form).

Decision Logic

  • Decision normalization: The analyst's decision string is compared case-insensitively to "Approved". Any other value (including "Rejected" or any typo) maps to "Rejected" status. This is a strict binary outcome.
  • Actor identification: If the HTTP session contains a valid SessionUser, their username is appended to the resolution notes. If the session is somehow empty at decision time, "system" is used as the actor, providing minimal audit trail.
  • Rule creation vs update: RuleEditAction.save() determines whether to INSERT or UPDATE based solely on whether ruleId is null. No uniqueness check on ruleName is enforced at the application layer.

State Transitions

FraudAlert lifecycle (inferred from status values used in queries and updates):

  • New → alert just created by an external detection system
  • InReview → alert is being examined (no explicit transition to this state is implemented in this service)
  • Approved / Rejected → terminal states, written by AlertDecisionAction

Only the terminal state transition (to Approved or Rejected) is implemented in this service. The New → InReview transition is not visible in the codebase and is presumably handled by an upstream system.

Transactions & Error Handling

No @Transactional annotation or explicit JDBC transaction management is used. Each JDBC operation is a single auto-committed statement. If a multi-step operation were introduced, there would be no rollback guarantee. Error handling is minimal: SQLException is caught and converted into a Struts action error message displayed to the user; there is no retry, compensating action, or dead-letter mechanism.

Authorization

Authorization is coarse-grained: any authenticated analyst can perform any operation — view all alerts, approve or reject any alert, and create, edit, or delete any fraud rule. No role-based access control (analyst vs admin), resource ownership checks, or @PreAuthorize-style guards are implemented.

Audit Trail

The only audit trail mechanism is the ResolutionNotes free-text field, which is appended with "Decision by <username>" at decision time. There is no structured audit log, change-history table, or immutable event record. Rule changes (create, edit, delete) produce no audit record.

Version & Compatibility Risks

Apache Struts 2.5.30 is the last release in the 2.5.x line, which reached end-of-life after the 2.5 series. Struts 2 is a legacy framework with a well-documented history of critical remote-code-execution vulnerabilities (e.g., CVE-2017-5638, CVE-2023-50164). The javax.servlet-api 4.0.1 uses the legacy javax. namespace, meaning the application targets Java EE 8 / Servlet 4.0 and is incompatible with Jakarta EE 9+ containers (jakarta. namespace) without code and dependency changes. The JDBC driver (mssql-jdbc 12.8.1) is current and poses no immediate compatibility risk.

Notable Observations

  • Struts 2 is end-of-life and a high-security-risk dependency: The Struts 2.5.x branch is no longer receiving security updates; migration to a modern framework (Spring MVC, Quarkus, Spring Boot) is strongly recommended.
  • No logging framework declared: The application relies entirely on Struts 2's bundled logging (which includes SLF4J and potentially Log4j2 transitively via struts2-core), with no explicit logging dependency — this makes log configuration opaque and potentially subject to transitive-dependency vulnerabilities.
  • No dependency management BOM: Versions are pinned directly in build.gradle with no BOM or version catalog, making future upgrades more error-prone.
  • Extremely minimal dependency footprint: The absence of security, caching, observability, or utility libraries means all cross-cutting concerns (session management, input validation, monitoring) are either absent or hand-rolled, increasing maintenance burden.

Caching Strategy

No caching layer is configured. The application performs a live SQL Server query on every HTTP request with no result caching, query caching, or session-level object caching. There is no EhCache, Redis, Caffeine, Hibernate second-level cache, or Spring Cache (@Cacheable) configuration. Fraud rule data (which changes infrequently) and session token lookups (called on every authenticated request) are both executed as database round-trips on every page load.

Data Ownership Boundaries

All data resides in a single shared SQL Server instance (ZavaBankDB). There is no database-per-service isolation; the ZavaFraudDetector WAR reads from tables that are conceptually owned by other subsystems (Users, SessionTokens, Accounts, Transactions). The application makes no REST or messaging calls to retrieve data — all cross-domain data access is done through direct SQL joins within the same database.

Read/Write split (observed):

  • FraudAlerts and FraudRules are both read and written by this service (owned tables).
  • Users, SessionTokens, Accounts, and Transactions are read-only from this service's perspective; write ownership lies with external subsystems (Auth service, Core Banking).

No CQRS or event sourcing patterns are in use; all reads and writes use the same JDBC connection to the same database.
