Data Storage & External Services

The service uses a single SQL Server database for fraud alerts, transactions, accounts, rules, and session-token lookups. It also calls an external authentication gateway (WhoAmI) over HTTP to translate the SSO cookie into a usable session token.

Key Architectural Decisions

  • Uses server-rendered MVC with Struts2 actions and JSP views instead of REST controllers.
  • Uses direct JDBC with inline SQL rather than ORM/repository abstractions.
  • Centralizes authentication gating in a custom interceptor applied through struts.xml interceptor stack.

DTOs & Contracts

The API contract uses in-repo POJOs rather than explicit versioned REST DTO schemas. FraudAlertRecord and FraudRuleRecord are mutable response models used by UI-bound actions, while SessionUser represents authenticated identity context in session state. Request contracts are mostly primitive form/query parameters bound directly to action properties (for example alertId, decision, ruleId). No OpenAPI, protobuf, or GraphQL schema files were detected.

Communication Patterns

Communication is synchronous throughout. Incoming browser calls reach Struts actions, which run direct JDBC queries against SQL Server. Authentication flow attempts local token resolution first, then makes a synchronous HTTP call to WhoAmI.ashx when only the SSO cookie is available. No asynchronous message queue/event bus patterns were detected. No circuit-breaker or retry libraries are configured; timeout handling exists only as fixed connect/read timeouts on the auth-gateway call. Service discovery is not present; external URL and database host are configured as explicit values. Security posture: no transport TLS or API-level authorization framework is configured in this service; authentication relies on custom interceptor/session checks and an external auth gateway.

Core Business Workflows

ZavaFraudDetector supports fraud analysts by surfacing suspicious transactions, enabling investigation, and allowing rule maintenance and decision capture. Its business flow centers on authenticated review queues and rule governance.

Primary Workflows

Workflow 1: Analyst opens flagged queue

  1. Analyst requests queue page.
  2. Authentication interceptor validates or resolves user session.
  3. Fraud queue action reads top pending alerts (New/InReview) with account and amount context.
  4. Queue page renders prioritized review list for analyst triage.

Workflow 2: Analyst reviews alert and records decision

  1. Analyst opens an alert detail by alertId.
  2. Detail action loads full alert context from persistence.
  3. Analyst submits decision (Approved/Rejected) with optional notes.
  4. Decision action updates alert status, resolution notes, and resolution timestamp.

Workflow 3: Analyst maintains fraud detection rules

  1. Analyst views current rule catalog.
  2. Analyst opens create/edit form and updates severity, expression, and thresholds.
  3. Save action validates required fields and number formats.
  4. Rule is inserted or updated; deletion flow removes obsolete rules.

Cross-Service Data Flows

The primary cross-service flow is session establishment: when a local session token is unavailable, the fraud service sends the SSO cookie to the auth gateway, receives a session token, then re-checks local token storage to bind an analyst user. Data composition for queue/detail flows is local to the SQL database through joins across fraud, account, and transaction tables. If the auth gateway is unavailable, protected workflows degrade to login redirect and analysts cannot access operational pages.

Business Rules & Decision Logic

  • Access rule: all workflows except login/health require a valid resolved SessionUser.
  • Alert decision rule: only explicit Approved maps to approved status; any other decision input maps to rejected status.
  • Rule maintenance validations: rule name and severity are mandatory; threshold/time-window values must parse as numbers.
  • State transition: fraud alerts move from operational review states into resolved states through analyst decisions captured with audit-style notes.
