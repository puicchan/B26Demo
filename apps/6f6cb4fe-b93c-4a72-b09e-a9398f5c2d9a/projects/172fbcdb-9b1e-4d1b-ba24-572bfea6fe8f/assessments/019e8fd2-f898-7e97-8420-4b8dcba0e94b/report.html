The service persists and reads risk-related data from a single SQL Server database (ZavaBankDb connection). There are no message brokers or third-party APIs; external integration is limited to runtime environment variables used to override connection settings.

Key Architectural Decisions

  • Uses a single service implementation (ZavaRiskEngineService) for all risk operations instead of splitting by bounded context.
  • Uses direct SQL statements via ADO.NET instead of ORM abstractions.
  • Keeps operational endpoint surface small: SOAP service operations plus one plaintext health endpoint.

Security posture is minimal at contract level: no explicit API authentication/authorization controls or TLS enforcement are configured in repository settings, and endpoints appear publicly reachable within the hosting boundary.

Core Business Workflows

ZavaRiskEngine supports loan-risk decisioning by scoring loan applications and exposing factor-based risk insights for customers. It also allows controlled updates to risk model parameters used in future scoring decisions.

Primary Workflows

Workflow 1: Evaluate Loan Application Risk

  1. A caller submits ScoreLoanApplication with an application identifier.
  2. The service validates input and loads loan, account, and latest credit score data.
  3. Business rules normalize credit score, balance ratio, loan term, and requested amount.
  4. Weighted model parameters are applied to compute a final numeric score and letter rating.
  5. Recommendation logic assigns Deny for the worst rating and Review otherwise.
  6. The service persists a RiskAssessments record and returns the response contract.

Workflow 2: Retrieve Customer Risk Factors

  1. A caller submits GetRiskFactors with a customer identifier.
  2. Service aggregates customer exposure, average balance, account activity, and latest credit score.
  3. Factor metadata is loaded and combined with computed values.
  4. Weighted impacts are summarized into an overall score returned to the caller.

Workflow 3: Update Risk Model Parameters

  1. A caller submits UpdateRiskModel with one or more parameter values.
  2. Service validates request payload and ensures parameter table exists.
  3. For each parameter, service performs update-or-insert within one database transaction.
  4. Service returns update count and operation completion timestamp.

Cross-Service Data Flows

No multi-service orchestration is implemented in this repository; all business data flows are internal to one service and one SQL Server datastore. Business degradation behavior is primarily input-validation faults and missing-record faults returned as WCF fault responses.

Business Rules & Decision Logic

  • Input identifiers must be greater than zero for scoring and factor retrieval operations.
  • Score computation applies weighted contributions from credit score, balance-to-request ratio, term length, and requested amount.
  • Rating bands map numeric scores to A through F, and approval multipliers are determined by rating.
  • Recommendation decision currently branches to Deny only for F, otherwise Review.
  • Model parameter updates are transactional and use upsert behavior for each submitted parameter.
