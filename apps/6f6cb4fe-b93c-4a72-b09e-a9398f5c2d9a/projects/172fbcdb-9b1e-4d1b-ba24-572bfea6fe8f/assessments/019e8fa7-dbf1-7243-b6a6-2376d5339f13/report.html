Data Storage & External Services

The application uses a single SQL Server database (ZavaBankDB) for both read and write operations, including risk model metadata and risk assessment outputs. No cache or message broker integration is present; database connectivity can come from web.config or environment variables.

Key Architectural Decisions

  • Uses a single-service WCF endpoint with direct SQL access rather than a repository abstraction.
  • Computes risk scoring in-process using configurable model parameters persisted in RiskModelParameters.
  • Publishes operational health through a lightweight custom health handler endpoint.

DTOs & Contracts

The service contract is defined by IRiskEngineService and implemented by ZavaRiskEngineService. Request and response contract types include RiskModelUpdateRequest, RiskModelParameter, RiskScoreResponse, RiskFactorsResponse, RiskFactorItem, and RiskModelUpdateResponse. These are WCF data contracts serialized via System.Runtime.Serialization and used as synchronous request-response payloads.

Communication Patterns

All service communication is synchronous SOAP over HTTP using basicHttpBinding. There is no asynchronous messaging, circuit breaker, retry policy, or service discovery mechanism in code or configuration. The service directly accesses SQL Server in each operation and returns faults using WCF FaultException when validation or lookup checks fail. No API-level authentication, authorization, or TLS enforcement is configured in the provided service configuration.

Core Business Workflows

ZavaRiskEngine supports risk evaluation workflows for loan decisioning by producing score outputs, factor breakdowns, and configurable model updates.

Primary Workflows

Workflow 1: Score Loan Application

  1. Consumer invokes ScoreLoanApplication on the service contract.
  2. Service validates the application identifier and loads loan/customer context from SQL.
  3. Service retrieves model parameter coefficients and normalizes business signals (credit, balance, term, amount).
  4. Decision logic derives final score, rating band, and max approved amount.
  5. Service persists a new RiskAssessment record and returns a RiskScoreResponse.

Workflow 2: Retrieve Risk Factors

  1. Consumer invokes GetRiskFactors for a customer.
  2. Service validates input and reads latest credit score, exposure, average balance, and active account count.
  3. Service maps values into weighted factor outputs with metadata from active RiskFactors.
  4. Service computes overall weighted score and returns RiskFactorsResponse.

Workflow 3: Update Risk Model Parameters

  1. Consumer sends UpdateRiskModel with one or more parameter updates.
  2. Service validates request envelope and starts a SQL transaction.
  3. For each named parameter, service upserts value and audit metadata (ModifiedBy, ModifiedDate).
  4. Service commits transaction and returns update summary.

Cross-Service Data Flows

The workflow composes risk outcomes from multiple data domains stored in a shared SQL database. ScoreLoanApplication and GetRiskFactors combine lending application data, account balances, and credit score snapshots into a single risk view. If upstream data rows are missing, the workflow degrades with service faults rather than partial responses.

Business Workflow Sequence

sequenceDiagram participant Consumer participant RiskSvc as "Risk Engine Service" participant Policy as "Risk Policy Parameters" participant Data as "Shared Banking Data" participant Assess as "Risk Assessment Store" Consumer->>RiskSvc: Request loan risk score RiskSvc->>RiskSvc: Validate request identity alt Valid request RiskSvc->>Data: Read application account and credit context Data-->>RiskSvc: Current customer risk inputs RiskSvc->>Policy: Load active model parameters Policy-->>RiskSvc: Parameter weights and base score RiskSvc->>RiskSvc: Compute score rating and approval limit RiskSvc->>Assess: Persist assessment decision Assess-->>RiskSvc: Assessment recorded RiskSvc-->>Consumer: Return scored decision else Invalid or missing record RiskSvc-->>Consumer: Return business fault end

Business Rules & Decision Logic

  • Requests must include positive numeric identifiers; invalid inputs are rejected immediately.
  • Final score is computed from normalized credit score, balance-to-loan ratio, loan term, and requested amount, weighted by configurable model parameters.
  • Risk ratings use ordered thresholds (A through F) and map to approval multipliers.
  • Model parameter updates are transactional to avoid partial policy changes.
  • Factor computation uses active factor metadata and applies default weights when metadata is missing.
