Core Business Workflows

ZavaLedger supports core ledger operations for account balance visibility and transfer posting between accounts. The business logic focuses on validating transfer requests, maintaining balance consistency, and recording transaction history.

Primary Workflows

Workflow 1: Retrieve Account Balance and History

A client requests account data using account endpoints. The workflow validates account ID format, queries persisted account and transaction records, and returns XML responses with either data or not-found/error status.

Workflow 2: Post Transfer Transaction

A client submits debit and credit account IDs, amount, description, and reference number. The workflow validates basic business rules (distinct accounts, positive amount), checks account existence, determines debit/credit transaction type mappings, updates both account balances, writes paired transaction entries, and commits atomically. On validation or database failure, it rolls back.

Cross-Service Data Flows

No multi-service aggregation flow was detected in this repository. Data composition occurs within the single service by reading account and transaction records from the same SQL Server database. Fallback behavior is local error handling (400 for invalid request data, 500 for processing/database failures).

Business Workflow Sequence

sequenceDiagram participant Client participant TxnServlet as "TransactionServlet" participant AccountStore as "Accounts Data" participant TypeStore as "Transaction Types Data" participant TxnStore as "Transactions Data" Client->>TxnServlet: Submit transfer request TxnServlet->>TxnServlet: Validate account IDs and amount TxnServlet->>AccountStore: Load debit and credit balances AccountStore-->>TxnServlet: Account balances TxnServlet->>TypeStore: Resolve debit and credit posting types TypeStore-->>TxnServlet: Posting type identifiers TxnServlet->>AccountStore: Update debit and credit balances TxnServlet->>TxnStore: Insert debit and credit transaction records alt All writes succeed TxnServlet-->>Client: Transfer posted with updated balances else Validation or DB failure Note over TxnServlet: Roll back all pending changes TxnServlet-->>Client: Error response end

Business Rules & Decision Logic

  • Transfer requests must reference two different accounts.
  • Transfer amount must be greater than zero.
  • Both accounts must exist before posting can proceed.
  • Debit and credit transaction types must be configured and active.
  • Account balance updates and transaction inserts execute in one transaction to preserve consistency.
