Data Storage & External Services

The application has a single external dependency: a Microsoft SQL Server instance hosting the ZavaBankDB database. Connection parameters (host, port, database name, username, password) are sourced first from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) and fall back to values in ledger.properties. All data access is performed through direct JDBC using DriverManager.getConnection, with prepared statements for account and transaction queries. There are no caches, message brokers, or additional external service integrations.

Key Architectural Decisions

  • Direct JDBC without an ORM: All data access uses raw JDBC prepared statements, avoiding ORM framework overhead but requiring manual resource management.
  • XML as the API contract: All API endpoints produce and consume application/xml, with a dedicated LedgerXml utility for safe XML escaping.
  • Configuration via environment variables with property file fallback: LedgerConfig reads database credentials from environment variables first, falling back to ledger.properties, supporting both local development and containerised deployments.

DTOs & Contracts

All request and response models are implicit XML documents — there are no formal DTO classes. Two private static inner classes exist within TransactionServlet:

  • TransactionRequest (private inner class): Carries parsed transaction inputs (debit account ID, credit account ID, amount, description, reference number). Used internally; not a public DTO class.
  • TransactionResult (private inner class): Carries the result of a committed transaction (generated transaction IDs, updated balances). Used internally to build the XML response.

No OpenAPI/Swagger specification, .proto files, or GraphQL schema are present. There is no formal serialization framework (Jackson, JAXB, etc.); XML is built and parsed manually using DocumentBuilder (request parsing) and StringBuilder (response generation). LedgerXml.escape() provides character escaping for XML output.

For the data model fields corresponding to the XML elements, refer to data-architecture.md.

Communication Patterns

Synchronous only: All communication is synchronous HTTP. There are no message brokers, event queues, or async patterns.

No inter-service communication: ZavaLedger is a single-service application. All business logic executes within the servlet request thread and communicates directly with the SQL Server database via JDBC.

No resilience patterns: There are no circuit breakers, retry policies, timeouts, or bulkhead configurations. Database failures propagate directly as HTTP 500 responses.

No service discovery: The application uses hardcoded database connection parameters resolved from environment variables or ledger.properties. There is no Eureka, Consul, or Kubernetes DNS-based service discovery.

No API gateway: Requests reach the servlets directly through Tomcat's servlet mapping defined in web.xml.

Security posture: No authentication, authorization, or TLS is configured at any level. All four endpoints (/health, /api/accounts/*, /api/transactions) are publicly accessible with no credential checks, session management, or role enforcement. Transport security (HTTPS) is not configured in the application — it would need to be handled externally (e.g., by a reverse proxy or load balancer).

ZavaLedger is a central banking ledger service that enables clients to query account balances, retrieve transaction histories, and post double-entry fund transfers between accounts.

ZavaLedger is a single bounded context with no inter-service communication. All domain logic resides within servlet request threads.

Primary Workflows

Workflow 1: Query Account Balance

A client queries the current and available balance for a specific account.

Steps:

  1. Client sends GET /api/accounts/{id}/balance
  2. AccountServlet parses the account ID from the URL path
  3. Validates: account ID must be numeric
  4. Executes a SQL SELECT against the Accounts table for Balance and AvailableBalance
  5. If the account is found: returns an XML <balanceResponse> with OK status and both balance values
  6. If the account does not exist: returns HTTP 404 with NOT_FOUND status

Workflow 2: Retrieve Transaction History

A client retrieves the complete transaction history for a specific account in reverse chronological order.

Steps:

  1. Client sends GET /api/accounts/{id}/transactions
  2. AccountServlet parses the account ID from the URL path
  3. Validates: account ID must be numeric
  4. Executes a SQL SELECT against Transactions ordered by TransactionDate DESC, TransactionID DESC
  5. Returns an XML <transactionsResponse> listing all transaction entries; returns an empty <transactions/> element if none exist
  6. All string fields in the response are XML-escaped via LedgerXml.escape()

Workflow 3: Post a Fund Transfer (Double-Entry Transaction)

A client posts a transfer of funds from one account to another, creating two balanced ledger entries.

Steps:

  1. Client sends POST /api/transactions with an XML body
  2. TransactionServlet parses the XML request using DocumentBuilder (with XXE protections enabled)
  3. Validates input:
    • Debit and credit account IDs must be different
    • Transfer amount must be greater than zero
  1. Opens a JDBC connection and begins a database transaction (setAutoCommit(false))
  2. Reads current balances for both the debit and credit accounts
  3. Verifies both accounts exist
  4. Looks up the active debit and credit TransactionType IDs from TransactionTypes
  5. Calculates new balances: debit account = current − amount; credit account = current + amount
  6. Updates both Accounts rows (balance, available balance, last activity date, modified date)
  7. Inserts two Transactions rows: one debit entry (negative amount) and one credit entry (positive amount), each referencing the counterpart account
  8. Commits the transaction; generated transaction IDs are returned
  9. On any failure: rolls back the transaction and returns an appropriate HTTP 400 or 500 response

Cross-Service Data Flows

ZavaLedger is a single-service application. All data flows are intra-service between the servlet layer and SQL Server. There are no cross-service aggregation patterns, circuit breakers, or fallback composition logic to document.

Business Rules & Decision Logic

Validation Rules

  • Account ID format: Account ID path parameter must be parseable as a positive integer; non-numeric values return HTTP 400.
  • Different accounts required: Debit and credit account IDs must differ; a self-transfer is rejected with HTTP 400.
  • Positive amount: Transfer amount must be strictly greater than zero; zero or negative amounts are rejected with HTTP 400.
  • Both accounts must exist: If either account ID is not found in the Accounts table, the transaction is rejected and rolled back.
  • Active transaction type required: At least one active debit type and one active credit type must exist in TransactionTypes; missing configuration causes an IllegalArgumentException which aborts the transaction.

State Transitions

  • Account balance: Updated atomically during a posted transaction — both debit and credit account balances change together or not at all (ACID transaction).
  • Transaction status: All inserted transaction records are given Status = 'Posted' immediately; there is no pending, reserved, or failed transaction state in the ledger.

Business Constraints

  • Double-entry integrity: Every fund transfer produces exactly two Transactions rows (one debit, one credit) so the ledger remains balanced.
  • No overdraft protection: The application does not check whether the debit account has sufficient funds before subtracting the amount — a debit account balance can go negative.
  • Channel always 'API': All transactions are recorded with Channel = 'API'; there is no distinction between originating channels (e.g., mobile, branch, batch).

Transactions & Error Handling

  • All write operations for a fund transfer are wrapped in a single JDBC transaction with explicit commit() / rollback().
  • SQLException and IllegalArgumentException both trigger a rollback; the HTTP response code differs (500 vs 400 respectively).
  • XML parsing errors in the request body surface as unchecked exceptions and are caught by the top-level try/catch in doPost, returning HTTP 500.

Audit & Logging

No audit trail, application-level logging, or event publishing is implemented. Database writes to the Transactions table serve as the implicit audit log, but there is no application event log, tracing, or structured logging output.

Authorization

No authentication or authorization is enforced. All endpoints are publicly accessible with no credential checks, role enforcement, or session management.

Version & Compatibility Risks

The javax.servlet-api 4.0.1 dependency uses the legacy javax. package namespace, which was superseded by jakarta. in Jakarta EE 9 (Servlet 5.0). The application targets Tomcat 9 (which still uses javax.*), so this is consistent, but upgrading to Tomcat 10+ or any modern Jakarta EE 10 runtime would require a namespace migration from javax.servlet to jakarta.servlet across all servlet classes. The mssql-jdbc 12.8.1.jre11 driver is recent and carries no immediate version risk, though the .jre11 classifier is used despite the project targeting Java 17 — the JRE 11 build is fully compatible with Java 17 but using the .jre17 or later artifact (if available) is preferable.

Notable Observations

  • No dependency injection or web framework: The application uses raw Jakarta EE Servlets with no Spring, CDI, or Quarkus — all wiring is manual, which increases migration effort when moving to a modern cloud-native framework.
  • No logging framework declared: There is no SLF4J, Log4j2, or Logback dependency; the application relies implicitly on java.util.logging or Tomcat's internal logging, making observability configuration harder.
  • No connection pooling: LedgerConnectionFactory uses DriverManager.getConnection directly with no connection pool (e.g., HikariCP or DBCP2), which is a scalability and performance concern for production workloads.
  • No test dependencies declared: The build file contains zero test-scoped dependencies, indicating the project has no automated test suite.
