Data Storage & External Services

The application depends on a single external data store: Microsoft SQL Server. Two tables are used — KYCWatchlist, which holds flagged names from compliance sources, and KYCVerification, which records all verification events. Connection details (host, port, database name, credentials) are read from kyc.properties at classpath, with environment variable overrides (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD). No caching layer, message broker, or third-party API integrations are present.

Key Architectural Decisions

  • Plain JDBC over an ORM: All database interactions use raw PreparedStatement calls through a static KycConnectionFactory, avoiding any ORM dependency. Every servlet opens and closes its own connection per request.
  • Servlet-based architecture without a web framework: The application uses Jakarta EE Servlets directly on Tomcat 9, with no Spring or JAX-RS layer, keeping the dependency footprint minimal.
  • Bootstrap-on-startup schema management: KycBootstrapServlet (load-on-startup order 1) issues idempotent IF OBJECT_ID ... IS NULL CREATE TABLE DDL statements to initialize the schema at container start, replacing a formal migration tool.

There are no formal DTO or model classes in the codebase. All request and response payloads are constructed inline using org.json.JSONObject directly inside each servlet's handler method:

  • Verify request: anonymous JSON body parsed with new JSONObject(body) — fields customerId, fullName, idNumber read via optInt/optString.
  • Verify response: JSONObject assembled inline — fields customerId, status, reason.
  • Status response: JSONObject assembled inline — fields customerId, status, reason, fullName, idNumber, verifiedDate.

No OpenAPI/Swagger specification, no protobuf schemas, no GraphQL schemas, and no immutable record/value-object types exist. There is no serialization framework configuration; all JSON is produced by manual JSONObject construction.

Core Business Workflows

ZavaKYCService implements the Know Your Customer compliance workflow for the Zava Bank platform, determining whether a customer's identity should pass or be flagged for review based on screening against a name watchlist.

ZavaKYCService is the sole source of truth for watchlist data and verification records. The CustomerID is a foreign key provided by the caller — the service trusts the caller to supply a valid customer identifier and does not validate it against any external customer registry.

Primary Workflows

Workflow 1: Application Bootstrap and Schema Initialization

Entry point: KycBootstrapServlet.init() — invoked automatically by Tomcat at startup (load-on-startup priority 1).

Steps:

  1. Container starts; Tomcat initialises KycBootstrapServlet first among all servlets.
  2. Bootstrap servlet opens a database connection and executes idempotent DDL to create KYCWatchlist and KYCVerification tables if they do not yet exist.
  3. If the watchlist table is empty, a single seed entry (John Alias, source OFAC-SAMPLE) is inserted to validate the schema and provide a test record.
  4. If the database is unreachable, the servlet throws ServletException and the application fails to start. No retry logic is present.

Workflow 2: KYC Identity Verification

Entry point: POST /api/kyc/verify — initiated by a downstream Zava Bank service or integration layer when a new customer must be screened.

Steps:

  1. Caller submits a JSON payload containing customerId (integer), fullName (string), and optionally idNumber (identity document number).
  2. Input validation: customerId must be greater than 0; fullName must not be blank. Any violation returns a 400 error immediately.
  3. Watchlist screening: The submitted fullName is compared case-insensitively against all entries in KYCWatchlist (UPPER(FullName) = UPPER(?)).
  4. Outcome determination:
    • If a match is found → status = REVIEW, reason = "Name matched watchlist."
    • If no match → status = PASS, reason = "No watchlist match."
  1. Record creation: A KYCVerification record is inserted with the submitted data, computed status, and reason. The record is never updated after creation.
  2. The caller receives the customerId, status, and reason as a JSON response.

Workflow 3: KYC Status Retrieval

Entry point: GET /api/kyc/status/{customerId} — queried by downstream services or operators to retrieve the current KYC standing of a customer.

Steps:

  1. Caller provides customerId as a URL path parameter.
  2. Input validation: customerId must be numeric and present. Non-numeric values return 400.
  3. The most recent KYCVerification record for that customer is retrieved, ordered by VerifiedDate DESC and VerificationID DESC.
  4. If no record exists for the customer, a 404 response is returned with status=NOT_FOUND.
  5. If a record is found, the full verification detail (status, reason, fullName, idNumber, verifiedDate) is returned.

Cross-Service Data Flows

ZavaKYCService is a leaf service in the Zava Bank platform with no outbound calls to other services. Cross-service data flow is unidirectional:

  • Inbound: The caller (typically another Zava Bank service such as an onboarding or account-opening service) provides the customerId and customer identity data when calling /api/kyc/verify. ZavaKYCService does not reach back to a customer service to validate or enrich the data.
  • Outbound: None. Results are returned synchronously to the caller.

There is no event publication, no message queue, and no gateway aggregation in this service. The customerId cross-reference to an external customer bounded context is resolved entirely by the calling service before invoking this API.

Degradation behavior: If the SQL Server is unavailable at verification time, the servlet catches SQLException and returns a 500 response with status=ERROR. No compensating action, retry, or dead-letter queue exists — the verification is simply lost from the caller's perspective.

Business Rules & Decision Logic

Validation Rules:

  • customerId must be a positive integer (> 0); zero or negative values are rejected with 400.
  • fullName must be present and non-blank after trimming; absent or blank values are rejected with 400.
  • idNumber is optional; it is stored as-is with no format validation.
  • customerId in the status endpoint must be numeric; non-numeric path segments return 400.

Decision Logic:

  • Watchlist matching: Case-insensitive exact full-name comparison (UPPER(FullName) = UPPER(?)). Partial matches, fuzzy matching, phonetic matching, and alias resolution are not implemented. A name must appear verbatim (case-aside) in the watchlist to trigger a REVIEW outcome.
  • Outcome determination: Binary — either REVIEW (name matched) or PASS (no match). No intermediate states, risk scores, or tiered review levels exist.
  • Status resolution: The most recent verification record (by VerifiedDate DESC, with VerificationID DESC as a tiebreaker) determines the current status returned for a customer. Earlier records are retained but not surfaced.

State Transitions:

  • KYCVerification records are append-only. Once written, a record is never updated or deleted. A customer can be re-verified by submitting a new request, which creates an additional record; the new record supersedes the old one for status queries.

Transactions:

  • No explicit transaction management. Each JDBC operation (watchlist lookup, verification insert) is performed on a single connection with auto-commit behaviour (SQL Server default). The lookup and insert are not wrapped in a single transaction, creating a potential race condition under concurrent requests for the same customer.

Error Handling:

  • SQLException during database access is caught in each servlet and mapped to a 500 response with a generic error message. No structured error types, compensating actions, or retry logic exist.
  • JSON parse failures on the request body return 400 with a fixed error message.

Audit / Logging:

  • No structured logging framework is present. There is no audit trail beyond the KYCVerification table itself and Tomcat's access log. VerifiedDate in KYCVerification provides a basic timestamp record of each verification event.

Authorization:

  • No authorization rules are enforced. Any caller that can reach the service network endpoint can submit verifications or retrieve status for any customerId.

Version & Compatibility Risks

org.json:json:20140107 is over a decade old and predates several API improvements and bug fixes in the library. While it still functions, it is far behind the current release line and has known issues (e.g. missing JSONObject.isEmpty() and other convenience methods). The javax.servlet:javax.servlet-api:4.0.1 dependency targets Servlet 4.0, which is the last version under the javax. namespace; the Jakarta EE namespace shift (Servlet 5.0+, package jakarta.servlet.) is a breaking change and requires updating all servlet imports as part of any Java 11+ / Tomcat 10+ migration. The MSSQL JDBC driver 12.6.3.jre8 is a recent release targeting Java 8 and is currently supported.

Notable Observations

  • No ORM or connection-pool library declared: the application drives SQL Server entirely through raw DriverManager calls with no connection pooling (no HikariCP, DBCP, or c3p0). Every servlet request opens and closes its own connection, which is a significant scalability and resource concern.
  • Extremely thin dependency tree: with only 3 declared dependencies the WAR is lightweight, but this also means all cross-cutting concerns (logging, validation, serialization) are implemented ad hoc or absent.
  • org.json version is critically outdated (2014): this version predates several security-related fixes and API improvements; upgrading to the current release (20240303 or later) is low-risk and strongly recommended.
  • No logging framework declared: the application has no SLF4J, Logback, Log4j2, or JUL binding. All observable runtime output goes through Tomcat's default container logging, making structured or configurable log output impossible without adding a logging dependency.

Caching Strategy

No caching layer is present. Every servlet request issues one or more direct SQL queries against SQL Server with no result caching, query caching, or second-level cache. There is no EhCache, Redis, Caffeine, or Spring Cache configuration.

Data Ownership Boundaries

ZavaKYCService is a self-contained service with its own isolated SQL Server database (ZavaBankDB). No other services are known to share this database directly. Cross-service data references are by convention only: KYCVerification.CustomerID is an integer that must match a customer record owned by another Zava Bank service, but no FK constraint or API callback enforces referential integrity between services.

There are no CQRS patterns, event sourcing, or read replicas. All reads and writes go to the same SQL Server instance. The application is read-write for both KYCWatchlist (read-only after initial seed in practice) and KYCVerification (append-only in practice; no update or delete operations are implemented).

Data Classification & Sensitivity

EntitySensitive FieldsClassificationControls in Place
KYCVerificationFullName, IdNumberPII (full name, government-issued ID number)None — no encryption-at-rest, no field masking, no column-level access control
KYCWatchlistFullNamePII (names of compliance-flagged individuals)None — no encryption-at-rest, no masking

KYCVerification stores customer full names and identity document numbers (IdNumber), which constitute PII under GDPR and comparable regulations. KYCWatchlist stores names of individuals flagged from compliance sources (e.g. OFAC). Neither table has encryption-at-rest, data masking, auditing, or field-level access controls configured within the application. The database connection is also configured with encrypt=false, meaning data is transmitted over the wire without TLS encryption.
