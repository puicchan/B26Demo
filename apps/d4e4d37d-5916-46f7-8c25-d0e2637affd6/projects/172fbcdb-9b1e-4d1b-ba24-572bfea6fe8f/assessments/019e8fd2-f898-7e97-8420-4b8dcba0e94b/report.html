Data Storage & External Services

ZavaQueueBridge does not use any database. Its sole external dependency is RabbitMQ, which is configured via environment variables or app.config. The application publishes parsed file records to a topic exchange (filedrop.events) with routing keys derived from the file extension (e.g., filedrop.csv). Unprocessable files are forwarded to a fanout dead-letter exchange (zava.dlx) before being moved to the error directory. Processed files are moved to a local processed/ subdirectory, and failed files are moved to an error/ subdirectory — both under the configured drop path (/shared/filedrop by default).

Key Architectural Decisions

  • Polling + FileSystemWatcher hybrid: The application combines a FileSystemWatcher for near-real-time event detection with a periodic poll loop (default 5 s) to catch files that may have been missed by OS events.
  • Dead-letter pattern for failures: Files that cannot be parsed or published result in a JSON error message being sent to a dedicated dead-letter exchange (zava.dlx) and the source file being moved to an error directory, enabling downstream error recovery.
  • Configuration via environment variables with app.config fallback: All runtime parameters (RabbitMQ connection, paths, poll interval) are read first from environment variables and then from app.config, making the service suitable for both containerised and traditional deployments.

DTOs & Contracts

ZavaQueueBridge does not use formal DTO classes. All message payloads are built as Dictionary<string, object> at the point of serialization and converted to JSON via System.Web.Script.Serialization.JavaScriptSerializer. There are no OpenAPI / Swagger specifications, protobuf schemas, or GraphQL schemas.

Published message shapes (informal contracts):

  • Success payload — published to filedrop.events exchange with routing key filedrop.<ext>:

```json

{

"sourceFile": "<filename>",

"processedAt": "yyyy-MM-ddTHH:mm:ss",

"record": { "<field>": "<value>", ... }

}

```

The record object structure varies by file type:

    • CSV: keys are the header row column names; values are the corresponding cell values.
    • Fixed-width (.txt/.dat): keys are RecordType, CustomerId, AccountId, Amount, Details (fixed positions within an 80-char line).
  • Dead-letter payload — published to zava.dlx exchange:

```json

{

"sourceFile": "<filename>",

"error": "<exception message>",

"failedAt": "yyyy-MM-ddTHH:mm:ss"

}

```

All payloads are serialized to UTF-8 JSON with content-type: application/json and delivery-mode: 2 (persistent).

Communication Patterns

Asynchronous (primary): The application uses RabbitMQ AMQP 0-9-1 for all outbound communication. Messages are published to a topic exchange (filedrop.events) with routing keys derived from the file extension. Downstream consumers can subscribe via queue bindings using patterns such as filedrop.* or filedrop.csv. A separate fanout dead-letter exchange (zava.dlx) receives error messages for unprocessable files.

Synchronous (internal): There is no synchronous inter-service communication. The only synchronous I/O is local file system reads (via File.ReadAllLines and File.Open) and the RabbitMQ connection establishment itself (blocking CreateConnection()).

Resilience patterns: No circuit-breaker, retry, or timeout policies are configured for the RabbitMQ connection or publish operations. The ConnectionFactory has AutomaticRecoveryEnabled = false, meaning any connection drop will cause the current poll cycle to throw and be caught by the top-level try/catch in the main loop. File-read retries use a simple spin-wait loop (up to 5 attempts × 250 ms) to handle files still being written. There are no Polly or similar resilience library integrations.

Service discovery: No service discovery mechanism is used. The RabbitMQ host is resolved from a static environment variable (RABBITMQ_HOST) or app.config value. Connection is made directly to the configured hostname and port (RABBITMQ_PORT, default 5672).

Security posture: No authentication or TLS is configured for any interface. The file-drop path is accessed via local file system with no ACL enforcement at the application level. The RabbitMQ connection uses plain-text AMQP with username/password credentials (RABBITMQ_USER / RABBITMQ_PASSWORD) stored in environment variables or app.config. No AMQP TLS (amqps://) configuration is present and AutomaticRecoveryEnabled = false means there is no built-in reconnection on credential/TLS errors.

Startup dependency chain: RabbitMQ must be available before the first poll cycle completes (approximately 5 seconds after startup). There is no startup probe, wait mechanism, or retry on initial connection — if RabbitMQ is unavailable, the first PollFiles() call will throw and be logged to stdout, and the service will retry on the next poll interval.

Primary Workflows

Workflow 1: Successful File Ingestion

Trigger: A .csv, .txt, or .dat file appears in the drop directory (via FileSystemWatcher event or periodic poll).

Steps:

  1. Detection — the file path is added to the PendingFiles work set (thread-safe, deduplicated).
  2. Poll cycle — at each poll interval (default 5 s), the pending set is drained and the broker connection is opened.
  3. File readiness check — the service waits for the file to be fully written (up to 5 × 250 ms retry attempts for exclusive read access).
  4. Parsing — the file is read in full; dispatched to the CSV parser (header row + data rows) or the fixed-width parser (80-char line layout) based on file extension.
  5. Record publishing — for each parsed record, a JSON Filedrop Message is built (record fields + sourceFile + processedAt) and published to the filedrop.events topic exchange with routing key filedrop.<ext> (e.g., filedrop.csv). Each message is marked persistent.
  6. File archival — on success, the original file is moved to the processed/ subdirectory (with a timestamp suffix on name collision).

Outcome: All records from the file are durably queued for downstream consumption; the original file is preserved in the processed archive.

Workflow 2: Failed File Handling (Dead-Letter)

Trigger: An exception is thrown during file readiness check, parsing, or broker publish for a given file.

Steps:

  1. The exception is caught at the per-file processing level in PollFiles().
  2. A Dead-Letter Message is built (sourceFile + error message + failedAt timestamp) and published to the zava.dlx fanout exchange (no routing key; all bound queues receive it).
  3. The original file is moved to the error/ subdirectory.
  4. Processing continues with the next file in the current batch.

Outcome: The failure is recorded as a message in the dead-letter exchange for downstream error handling; the failed file is quarantined in the error directory and does not block subsequent files.

Workflow 3: Process Startup & Watcher Initialization

Trigger: Application process starts (Main() entry point).

Steps:

  1. Configuration load — all settings resolved from environment variables → app.config → hardcoded defaults.
  2. Directory setup — drop path, processed path, and error path are created if absent.
  3. Watcher activation — FileSystemWatcher is attached to the drop path for Created, Changed, and Renamed events; only .csv, .txt, and .dat files are enqueued.
  4. Poll loop — the main thread enters the continuous poll loop (sleep → PollFiles() → repeat).
  5. Graceful shutdown — Ctrl+C or SIGTERM sets _keepRunning = false; the current poll cycle is allowed to complete before the watcher is disposed and the process exits.

Cross-Service Data Flows

ZavaQueueBridge is a single-service application; there are no cross-service API calls or shared databases. Data flows are strictly:

  • Inbound: Upstream producers (external systems or human operators) write files to the shared file system volume. The service detects them via OS file events and periodic polling.
  • Outbound (success path): Records are forwarded to downstream consumers via the filedrop.events RabbitMQ topic exchange. Consumers subscribe by binding queues to routing-key patterns (e.g., filedrop.*, filedrop.csv). The bridge has no knowledge of downstream consumer identity or processing outcome.
  • Outbound (error path): Failed files produce dead-letter messages on zava.dlx. Downstream error-handling consumers (alert systems, retry workers) bind queues to this exchange.

Fallback behaviour: If the RabbitMQ broker is unavailable, the publish call throws an exception, the current file is dead-lettered (to the exchange — which will also fail), and the file is moved to the error directory. There is no local message store or in-process retry buffer — messages cannot be delivered until the broker is restored and the file is manually re-queued.

Business Rules & Decision Logic

File Candidate Rules

  • Extension filter: Only files with extensions .csv, .txt, or .dat are processed. All other files are silently ignored.
  • Path exclusion: Files residing inside the processed/ or error/ subdirectories are skipped even if their paths appear in the pending set (prevents re-processing of archived files).
  • Directory exclusion: Files named processed or error (without extension) are excluded by name check.

File Readiness Rule

  • A file is considered ready for reading when it can be opened with shared read access. Up to 5 attempts are made at 250 ms intervals. If all fail, the exception propagates and the file is dead-lettered.

Parsing Rules

  • CSV: The first line is treated as a header row; subsequent non-blank lines are data rows. Quoted fields (delimited by ") are handled; commas inside quotes are not treated as delimiters. Each column value is trimmed of leading/trailing whitespace.
  • Fixed-width (.txt and .dat): Each line is padded to 80 characters and sliced into five fixed fields — RecordType (chars 0–9), CustomerId (10–29), AccountId (30–49), Amount (50–64), Details (65–79). No schema validation is performed on field values.

Routing Rule

  • The AMQP routing key is filedrop.<ext> where <ext> is the lowercase file extension without the dot (e.g., filedrop.csv, filedrop.txt). Files without an extension get routing key filedrop.unknown.

File Archival Rules

  • On success: file is moved to the processed/ directory. If a file with the same name already exists there, the file is renamed with a yyyyMMddHHmmss timestamp prefix to prevent overwrite.
  • On failure: file is moved to the error/ directory (same rename logic for collisions).

Error Handling & Dead-Letter Rules

  • Any exception thrown during file processing (read, parse, or publish) for a single file is caught at the batch level; a Dead-Letter Message is published and the file is quarantined. Processing continues for remaining files in the batch — one bad file does not block the entire batch.
  • Dead-letter messages contain the file name, the exception message, and a failure timestamp. No stack trace or raw file content is included.

Ordering & Deduplication Rules

  • Files are processed in the order they are returned by Directory.GetFiles() (OS-defined, typically alphabetical). No guaranteed FIFO order with respect to write time.
  • Duplicate detection relies on the processed/ move: once moved, the file path is no longer in the drop directory and will not be re-detected. However, during a crash between last publish and file move, records may be published again on restart (at-least-once delivery semantics with no consumer-side deduplication enforced by this service).

Cross-Cutting Concerns

  • Transactions: There are no distributed transactions. File move and message publish are not atomic — a crash between BasicPublish and File.Move will result in a re-published record on the next startup scan (at-least-once semantics).
  • Concurrency: The PendingFiles set is protected by a lock during enqueue and drain operations. Watcher events and the poll thread share this set safely.
  • Shutdown safety: The _keepRunning flag is volatile; the per-file loop checks it before each file to allow a clean mid-batch shutdown.
  • Audit / logging: All significant events are logged to stdout (Console.WriteLine) — no structured logging, no log levels, no log aggregation.
  • Authorization: No authentication or authorization is enforced. Any process or user with write access to the drop directory can inject files for processing.

Version & Compatibility Risks

RabbitMQ.Client 5.2.0 was released in 2018 and is several major versions behind the current 7.x release line. The 5.x branch has reached end-of-active-support, and the API changed significantly in 6.x (asynchronous consumer model, IChannel replacing IModel). Migrating to the latest client requires non-trivial code changes. Additionally, the application targets .NET Framework 4.8, which is in long-term maintenance mode with no new feature development. The System.Web.Extensions assembly (used for JavaScriptSerializer) is a legacy .NET Framework-only API deprecated in favour of System.Text.Json or Newtonsoft.Json. Migration to .NET 8+ would require replacing all three BCL dependencies with their modern cross-platform equivalents.

Notable Observations

  • Single third-party dependency: The project intentionally minimises its dependency surface — RabbitMQ.Client is the only NuGet package. This simplifies migration but also means all serialisation and configuration logic relies on legacy .NET Framework BCL APIs.
  • JavaScriptSerializer is deprecated: System.Web.Script.Serialization.JavaScriptSerializer is a legacy class removed from .NET Core / .NET 5+. Any modernisation effort must replace it with System.Text.Json.JsonSerializer or Newtonsoft.Json.
  • RabbitMQ.Client API breaking changes: Upgrading from 5.x to 6.x or 7.x requires replacing IModel with IChannel, adopting async dispatch, and updating connection lifecycle management throughout Program.cs.
  • No lock file present: The project uses packages.config (NuGet v2 format) rather than PackageReference, meaning there is no lock file and transitive dependency resolution is implicit via the .nupkg contents of RabbitMQ.Client 5.2.0.

Caching Strategy

No caching layer is implemented. ZavaQueueBridge processes each file exactly once (move-on-completion prevents re-processing) and maintains a single in-process HashSet<string> (PendingFiles) purely as a transient work queue within a single poll cycle. This set is cleared after each processing batch and is not persisted or shared across restarts.

There is no Redis, EhCache, Caffeine, IDistributedCache, MemoryCache, or any cache provider configured.

Data Ownership Boundaries

ZavaQueueBridge is a single-service application with no shared database and no cross-service data access. The data topology is:

  • Inbound: The drop-path directory (/shared/filedrop) is a shared-volume mount in the Docker deployment. Files are produced externally (by an upstream banking system or human operator) and consumed destructively — moved to processed/ or error/ after handling. The application treats the file system as a reliable, ordered inbox.
  • Outbound: RabbitMQ is the single durable outbound data store. Each published message is marked persistent (delivery-mode: 2) and the exchanges are declared durable, so messages survive broker restarts. Downstream consumers own their own queue bindings and persistence.
  • No shared database: There is no database shared with other services. The service does not read from or write to any relational or document store.
  • Restart behaviour: The in-process PendingFiles set is volatile — any files detected between the last successful move and a crash are silently dropped. On restart, the poll loop re-scans the drop directory, so files present at startup are recovered; files already in transit during a crash may be reprocessed if not yet moved.
